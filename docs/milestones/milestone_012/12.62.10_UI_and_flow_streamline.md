1. Top-level information text is too prominent.

2. Create Source Profile button should be visually separate from status/list controls.

3. Prepopulated fields in modal are confusing.

   - Better placeholder text:
     “Example: Chuck PC”
     “Example: External Drive 1”
     “Example: Chuck iCloud”
   - Avoid making example text look like an already selected value.

4. Profile Status dropdown is unnecessary during creation.
   
   - A newly created profile should default to active.
   - Status management can happen after creation if needed.
   - Remove from create modal or hide under advanced/admin.

5. Close and Cancel are redundant.
   
   - Keep one dismiss action.
   - Preferred:
     Create Profile
     Cancel

6. Source Type terminology needs cleanup.
   
   - “Cloud Export / iCloud” should become “Cloud Export” or “Cloud Source.”
   - iCloud should be determined by Cloud Provider, not embedded in Source Type.
   - Source Type should describe source family.
   - Cloud Provider should describe provider.

7. “Scan Batch” is unclear.
   
   - User does not understand what it means.
   - Likely internal/legacy/technical.
   - Should be hidden from normal users unless there is a real v1 workflow for it.
   - If retained, it needs a clearer name and help text.

8. “Other” is too vague.
   
   - Creates uncertainty.
   - Should probably be hidden from normal source creation.
   - Could remain as admin/developer-only or be removed until there is a defined use case.

9. Source Root Path vs Managed Staging Path needs more elegant handling.
   
   - Local/external sources need editable source root path.
   - Cloud sources should not ask for source root path.
   - Cloud profile should show managed staging path as an automatic preview.
   - The UI should explain: “Photo Organizer will create and manage this staging folder.”

10. Acquisition Method is currently redundant/confusing.
    
    - For now acceptable.
    - Future simplification:
      infer acquisition method from source type + cloud provider.
    - Example:
      Cloud Export + iCloud => icloudpd
      Local Folder => folder_scan
      External Drive => folder_scan
    - OneDrive/Google can be refined later because their workflows differ from iCloud.

11. Managed Staging Path preview appears useful but should be cleaner.
    
    - Show full path or expandable/copyable path.
    - Current field truncates path, which may reduce confidence.
    - Keep preview path and resolved path details, but make them less technical.

### User observations

1. **Readiness should be binary.**
   
   User-facing readiness should be either:
   
   ```text
   Ready to acquire
   Blocked
   ```
   
   A user-facing “Warning” state is confusing because it does not clearly answer whether the user should proceed.
   
   If a warning exists, it should either be:
   
   - fixed automatically by the workflow,
   
   - converted into a clear blocker,
   
   - or moved into technical/advanced details.

2. **Authentication should be part of readiness if readiness claims the source is ready.**
   
   A source is not truly ready to acquire from iCloud unless the iCloud session is usable.
   
   Future flow should likely include authentication/session health before acquisition:
   
   ```text
   Create Profile
   → Verify staging/source setup
   → Check iCloud authentication/session health
   → Ready to acquire
   → Acquire from iCloud
   ```
   
   If authentication remains part of the Acquire from iCloud step, it should not also appear earlier as a readiness warning. It should live in one place.

3. **Photo Organizer should not store Apple credentials, but the UI may help launch authentication.**
   
   User challenged the assumption that authentication must be terminal-only.
   
   Revised design direction:
   
   ```text
   Photo Organizer should not collect, store, log, or own Apple passwords or 2FA codes.
   However, Photo Organizer may provide a UI-controlled, isolated icloudpd authentication helper.
   ```
   
   Possible future behavior:
   
   ```text
   Check iCloud Session
   → Authentication required
   → Open iCloud authentication helper
   → icloudpd handles authentication/session storage externally
   → Photo Organizer records only session health/status, not secrets
   ```
   
   Important boundaries:
   
   - Do not store Apple password in Photo Organizer database.
   
   - Do not log Apple password or 2FA code.
   
   - Do not expose credentials in command history.
   
   - Keep icloudpd session storage separate.
   
   - Persist only authentication/session status, not credentials.

4. **Readiness, warnings, blockers, conflicts, and authentication state should be consolidated.**
   
   Current UI has separate tiles for:
   
   ```text
   Readiness
   Operation Conflicts
   Blocking Reasons
   Warnings
   iCloud Authentication
   Recommended Next Action
   ```
   
   These should not be separate main workflow tiles.
   
   Future design should roll them into one readiness summary:
   


### Acquisition mode clarification

**Standard mode:**

```text
Downloads the requested number of recent iCloud assets into the selected Source Profile’s managed staging folder.
It does not intentionally avoid assets previously acquired for that profile.
It is profile-targeted by destination path, but not repeat-aware by Photo Organizer history.
```

**List-first / non-repeat mode:**

```text
Lists candidate iCloud assets first, compares them against prior acquisition records for the selected Source Profile, and downloads only assets that have not already been acquired for that profile.
```

Preferred future user-facing naming:

```text
Standard mode → Standard recent download
List-first / non-repeat → Download new items only
```

---

### Key observation

The separation between local ingestion and iCloud ingestion appears functionally correct.

However, as cloud ingestion is refined, the app should unify as much of the user-facing workflow as possible between local and cloud sources.

The underlying processes differ:

```text
Local / external:
Source Profile → folder path → Source Intake

iCloud:
Source Profile → managed staging → iCloud acquisition → Source Intake → cleanup dry run
```

But the user-facing experience should have a shared structure and look/feel wherever possible.

---

### Future design direction

Use a common ingestion workflow shell for all source types:

```text
1. Source Profile
2. Readiness
3. Intake / Acquisition
4. Import / Source Intake
5. Review Results
6. Cleanup / Follow-up, only where applicable
```

For local sources, some steps can be hidden or simplified:

```text
Source Profile
→ Readiness
→ Run Source Intake
→ Review Results
```

For iCloud sources, the same shell can expose the extra cloud-specific steps:

```text
Source Profile
→ Readiness / Authentication
→ Acquire from iCloud
→ Run Source Intake
→ Cleanup Dry Run
→ Review Results
```

The goal is not to force identical backend operations. The goal is to make the workflows feel consistent to the user.

---

### UX principle

```text
Different source types may require different backend steps, but the UI should use shared language, shared status patterns, and shared summary tiles wherever possible.
```

Suggested shared concepts:

```text
Source
Readiness
Run / Intake
Files found
Files imported
Issues
Next safe action
Advanced details
Report path
```

Avoid source-specific technical tiles unless they are truly needed for that source.
