# Milestone 12.62.28.1 - Historical Backfill Acquisition from Inventory Pre-Code Plan Closeout

## 1. Executive Summary

This is a planning gate only. No product code implementation was started for acquisition from inventory.

The recommended 12.62.28.2 direction is a backend-only, inventory-driven acquisition service that selects eligible inventory rows, re-resolves those rows against fresh helper listing metadata, then reuses the existing exact-selection acquisition, durable batch, and batch Source Intake path. Source Intake must remain the only authority that writes into Vault.

The key design constraint is that the 12.62.27 inventory is intentionally metadata-only and does not persist a complete per-resource download manifest for every logical item. The existing exact-selection download path requires helper item IDs plus per-resource IDs, relative paths, expected sizes, and expected checksums. Therefore 12.62.28.2 should not try to construct download requests from inventory rows alone. It should use inventory as the selection ledger, then build the exact selected download request from a fresh helper listing snapshot.

## 3. Proposed Backend Design

Add a dedicated inventory acquisition service, tentatively:

```text
backend/app/services/icloud_backfill_acquisition_service.py
```

Recommended service responsibilities:

1. Validate the source profile using the existing iCloud exact-selection profile validation path.
2. Ensure acquisition, backfill, and Source Intake schema helpers are available.
3. Preflight the staging area using existing exact-selection/internal-loop safety checks.
4. Query eligible inventory rows for the source.
5. Optionally run a metadata-only known-state check before selection.
6. Re-list helper candidates metadata-only up to an approved listing cap.
7. Match selected inventory remote identities to current helper listing items.
8. Build an exact selected-items download request from fresh listing metadata.
9. Execute the existing durable exact-selection acquisition path or a narrowly factored equivalent.
10. Hand the durable batch to `run_batch_source_intake()`.
11. Update backfill inventory/state rows after Source Intake result reconciliation.

Recommended new durable/acquisition adapter boundary:

- Add a small reusable preparation path that can accept explicitly selected `ExactSelectionLogicalItem` objects from a listing.
- Avoid duplicating helper protocol request construction.
- Avoid bypassing `IcloudAcquisitionRun`, `IcloudAcquisitionBatch`, `IcloudAcquisitionItem`, and `IcloudAcquisitionResource` tracking.

Do not introduce a new direct-to-Vault path.

## 4. Inventory Selection Rules

Inventory rows should be selected only when all are true:

- `source_profile_id` matches the requested source.
- `eligibility_state = "eligible_metadata_only"`.
- `known_state` is not an exact known/complete state.
- The row is not already completed for backfill acquisition.
- `remote_identity_basis` and `remote_identity` are populated.
- The row has enough metadata to safely re-resolve against a fresh helper listing.
- The row is not marked ambiguous or unsupported.

Recommended selection ordering:

```text
created_remote_at ASC NULLS LAST
added_remote_at ASC NULLS LAST
first_observed_at ASC
id ASC
```

Important constraints:

- Do not use `observed_remote_position` as a durable cursor.
- Do not assume helper listing order is stable across runs.
- Treat one Live Photo logical item as one inventory/acquisition item, even if it has multiple resources.
- Use bounded acquisition limits no larger than the existing exact-selection selected-item cap.

Rows missing from the fresh helper listing should not block the whole run. They should be recorded as stale or currently unavailable and skipped for that run.

## 5. Known-State Strategy

Known-state must continue to mean whether the candidate is already known to the local Photo Organizer library, not whether the media type is recognized.

The existing `known_state_service.evaluate_known_state()` is path/provenance based. It can be useful when a candidate has a reliable source-relative path, but it is not a remote-identity proof by itself. For 12.62.28.2:

- Use exact provenance plus verified Vault asset evidence for strong known-state updates.
- Do not use fuzzy filename matching.
- Do not mark a row known solely because the helper media type is supported.
- Leave uncertain rows as `pending_known_state_check` or equivalent.
- Let Source Intake remain the final dedupe/intake authority after files are staged.

Recommended new/clarified inventory fields:

- `acquired` or `backfill_completed`
- `acquired_at` or `backfill_completed_at`
- `acquisition_run_id`
- `acquisition_batch_id`
- `source_intake_run_id`
- `last_acquisition_attempt_at`
- `acquisition_attempt_count`
- `acquisition_state`
- `skip_reason`
- `last_error_code`
- `last_error_message`

Product-owner decision recommended before coding:

```text
Should "acquired = true" mean "downloaded as a new asset" only,
or "this inventory row has been fully resolved by the backfill flow,"
including exact already-known/duplicate outcomes?
```

The safer naming is `backfill_completed` plus `backfill_resolution_state`, because it avoids implying that every completed row created a new Vault asset.

## 6. Acquisition/Reuse Strategy

Reuse the existing exact-selection acquisition components as much as possible:

- `validate_exact_selection_profile()`
- `build_exact_download_request()`
- helper request validation from `exact_selection_protocol.py`
- durable acquisition run/batch/item/resource tables
- retry/reconcile behavior in `durable_exact_service.py`
- `run_batch_source_intake()`
- existing cleanup review flow after Source Intake

Inventory should not replace the durable batch manifest. The durable batch remains the acquisition evidence ledger.

Recommended flow:

1. Inventory service selects a bounded list of candidate inventory rows.
2. Helper is asked for a fresh metadata listing.
3. Service matches inventory `remote_identity` to helper `item_id`.
4. Service builds selected logical items/resources from the fresh listing.
5. Existing exact-selection request builder creates the helper download payload.
6. Existing durable acquisition persists the selected manifest and downloads to staging.
7. Existing batch Source Intake processes staged resources.
8. Backfill inventory rows are updated from durable batch and Source Intake results.

Avoid these designs:

- constructing selected download payloads only from 12.62.27 inventory rows,
- downloading directly from inventory without a fresh helper manifest,
- marking rows complete before Source Intake has succeeded or exact known-state evidence exists,
- writing assets or provenance outside Source Intake.

## 7. State Machine Changes

Current 12.62.27 backfill status is inventory-scan oriented. 12.62.28.2 should extend the state model rather than overloading scan status.

Recommended source-level states:

- `inventory_ready`
- `acquiring`
- `paused_for_source_intake`
- `paused_for_cleanup`
- `complete`
- `blocked`
- `failed`

Recommended row-level acquisition states:

- `pending`
- `selected`
- `staged`
- `source_intake_completed`
- `known_vault_verified`
- `skipped_stale_listing`
- `skipped_unsupported`
- `failed_retryable`
- `failed_terminal`

Recommended stop reasons:

- `acquire_limit_reached`
- `no_eligible_inventory_rows`
- `source_intake_required`
- `cleanup_review_required`
- `preflight_blocked`
- `staging_not_clean`
- `helper_listing_missing_selected_identity`
- `helper_manifest_changed`
- `source_intake_failed`
- `cleanup_failed`

State transitions must be idempotent. Retrying a run should not duplicate completed inventory rows or create duplicate durable acquisition batches for the same already-completed rows.

## 10. Safety/Preflight Plan

Before any download:

- validate active iCloud source profile,
- validate managed staging path and approved root alignment,
- reject symlinked staging/source roots,
- block if partial workspace files exist,
- block if staged unknown resources are pending Source Intake,
- block if another iCloud acquisition lease is active,
- block if Source Intake guardrail reports an active operation,
- use bounded `acquire_limit`,
- use bounded helper listing candidate limit,
- write only to managed staging through the existing helper path,
- pass all staged files through batch Source Intake.

After acquisition:

- do not write Vault directly,
- do not delete remote iCloud assets,
- do not run cleanup without existing cleanup dry-run/review controls,
- keep report output redacted and count-focused,
- store durable batch evidence for retry/reconcile.

## 12. Risks and Open Questions

Open product question:

- Should completion be represented as `acquired`, `backfill_completed`, or both? Recommendation: use `backfill_completed` and a separate resolution state to avoid implying every resolved row was newly acquired.

Technical risks:

- The helper may not support direct fetch-by-item-id, so acquisition from inventory may require a broad fresh listing scan to re-resolve selected identities.
- Current inventory rows do not persist complete per-resource manifests, so request construction must use fresh helper metadata.
- Live Photos require complete multi-resource manifests; primary-path-only known-state checks are not sufficient for full logical-item certainty.
- Large helper list responses may still hit response-size or runtime limits even with the 100000 metadata cap.
- Helper item IDs are observed stable identities, not documented permanent CloudKit guarantees.
- Existing `IcloudBackfillState.status` uses inventory-scan terminology; acquisition states should be added carefully to avoid ambiguous status meanings.
- Cleanup behavior should stay behind existing review/dry-run controls unless separately approved.

Decision needed before implementation:

- Whether 12.62.28.2 should include automatic Source Intake after staging, or stop after durable acquisition with `next_safe_action = Run batch Source Intake`.
- Whether acquisition endpoint should default to `dry_run = true` for first backend release.
- Whether skipped stale rows should remain pending for future retries or move to a separate skipped state requiring re-inventory.
