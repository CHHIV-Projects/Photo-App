# Milestone 12.62.15 — iCloud Helper Runtime and Exact-Selection Adapter Prototype

## Recommended Reasoning Level

Extra High.

This milestone crosses iCloud authentication boundaries, helper/runtime design, logical item identity, exact selected downloads, Live Photos, partial staging, and future durable ingestion. A subtle mistake could leak credentials, redownload known resources, produce incomplete Live Photo intake, or create an unsafe foundation for later batch orchestration.

## Goal

Prototype or precisely block a secure internal iCloud helper/adapter capable of true exact-selection acquisition.

The target capability is:

```text
list logical iCloud items newest-to-oldest
→ expose logical item and resource relationships
→ evaluate selected Source Profile known-state
→ select up to N new logical iCloud items
→ download exactly the selected unknown resources
→ stage through .partial
→ publish only complete verified items
→ avoid credential leakage
```

This milestone is the first step in the durable cloud-ingestion track. It should not implement UI simplification or the full acquire/intake/cleanup loop.

## Background

Recent milestones established the following:

### 12.62.12

Validated that current fixed-window list-first/non-repeat behavior can detect all-known candidate windows and avoid reacquisition after guarded cleanup.

But it did not prove true “download N new items.”

### 12.62.13

Defined desired new-count semantics and implemented an internal planner.

Approved semantics:

```text
N = logical iCloud items, not raw resource files.
```

A Live Photo should count as one logical item while still reporting both the still image and motion resource as physical resources/files.

The planner is internal and not exposed through UI/API.

### 12.62.14

Determined that the currently installed `.tools/icloudpd` runtime exposes only the bundled CLI behavior and cannot execute exact selected downloads.

The upstream/source internals appear promising, but they are not available to the installed runtime. Therefore, true new-count acquisition remains blocked unless Photo Organizer introduces a separate, isolated helper/runtime or obtains a supported upstream interface.

## Product Direction / Lock-ins

The user has chosen the secure and durable path, not the pragmatic fixed-window v1 path.

Durable iCloud ingestion means:

```text
true non-repeat acquisition
batch-based execution
total run limit
resumable progress
clear stop/failure reasons
no duplicate redownload where avoidable
no partial/ambiguous intake batches
safe recovery after Apple timeout, helper hang, cloud failure, auth/session failure, network failure, or local compute interruption
```

Future user-facing model:

```text
Batch: 50
Limit: 1000
Mode: Download only new

Acquire 50 new logical iCloud items
→ Source Intake those 50
→ Cleanup those 50
→ checkpoint
→ repeat until 1000 new logical items, no more new items, user stop, or failure
```

This milestone does **not** implement that full loop. It proves or blocks the helper/adapter capability required before that loop can be built.

## Helper / Backend Protocol Requirements

If a helper is implemented, use narrow JSON communication.

The backend may send only non-secret request data such as:

```json
{
  "operation": "list" | "download_selected",
  "source_profile_id": 66,
  "staging_root": "...",
  "target_new_item_count": 50,
  "candidate_scan_limit": 500,
  "selected_items": []
}
```

The helper may return secret-free JSON such as:

```json
{
  "status": "completed" | "partial" | "blocked" | "failed",
  "logical_item_count": 50,
  "resource_file_count": 63,
  "items": [],
  "failures": [],
  "stop_reason": "target_new_count_reached"
}
```

Rules:

- Do not include Apple credentials.

- Do not include cookies.

- Do not include session tokens.

- Do not include raw download URLs in persisted reports unless explicitly approved.

- Use bounded samples for large result sets.

- Keep remote iCloud IDs transient in memory for this milestone.

- Do not persist remote IDs to DB/provenance.

## Logical Item / Resource Semantics

Approved semantics:

```text
N counts logical iCloud items.
```

A logical item may have one or more downloadable resources.

Examples:

```text
Normal photo:
  1 logical item
  1 resource file

Live Photo:
  1 logical item
  2 resource files, usually still + motion

Unsupported or ambiguous sidecar:
  block or mark unsupported; do not infer
```

Reports must distinguish:

```text
logical_item_count
resource_file_count
selected_new_item_count
selected_new_resource_count
downloaded_item_count
downloaded_resource_count
failed_item_count
failed_resource_count
```

Generated XMP should not count as a cloud resource.

Unsupported sidecars, RAW/alternate relationships, adjusted versions, or ambiguous resource roles should block rather than be guessed.

## Exact Selected Download Requirements

If feasible, the helper/prototype must prove:

1. Known resources are not redownloaded in a mixed known/new candidate set.

2. Selected unknown resources are downloaded exactly.

3. Live Photo selected items download all required selected resources.

4. Known sibling resources are not redownloaded.

5. Unknown sibling resources are not suppressed by known resources.

6. A logical item is not published to normal staging until all selected resources for that item succeed.

7. Partial item failures remain in `.partial` or are cleaned/marked failed.

8. Source Intake should not see incomplete logical items as normal staged files.

9. Downloaded resources are verified by size/checksum where available.

10. Per-item and per-resource failures are reported.

## `.partial` Staging Requirements

Downloads must occur under a temporary area inside the selected Source Profile staging root, for example:

```text
<managed_staging_root>/.partial/<run_id>/<logical_item_id>/
```

Publication rules:

```text
download selected resources to .partial
→ verify size/checksum where available
→ if all selected resources for the logical item pass, move/publish into normal staging
→ if any required resource fails, do not publish partial item
```

`.partial` must be:

- scoped to the selected Source Profile

- safe to retry

- identifiable by run/batch

- excluded from normal Source Intake scans unless explicitly intended

- cleaned or reported safely after failure

## Staged Unknown Rule

If unknown resources are already present in normal managed staging but not ingested, block new exact-selection acquisition with:

```text
Run Source Intake first.
```

Rationale:

```text
Acquisition defines the batch.

Do not create oversized or ambiguous mixed staging batches.
```

## 2. Authentication isolation

Approved with clarification.

For this milestone, use a dedicated external cookie/session directory for the helper while reusing the OS keyring if that is the safest path.

Photo Organizer must not receive, store, log, or persist:

* Apple passwords
* 2FA codes
* cookies
* session tokens
* keyring values
* raw authentication secrets

Authentication or 2FA may be user-operated outside Photo Organizer for this prototype.

Long-term product direction:

Photo Organizer should eventually provide UI-integrated authentication guidance, such as:

* Check iCloud Session
* Authenticate iCloud
* Re-authenticate
* Open iCloud Authentication Helper

But “UI-integrated” does not mean Photo Organizer owns credentials. The actual credential/session handling should remain in an isolated helper, browser flow, or external process.

Please design the helper/runtime boundary so future UI-integrated authentication is possible. The helper should return secret-free auth states such as:

* authenticated
* authentication_required
* session_expired
* reauthentication_required
* authentication_failed
* helper_unavailable

Do not implement full authentication UI in this milestone unless it is strictly necessary for feasibility.

## 3. Publication guarantee

Approved for this prototype.

For now, it is sufficient to:

* download into a `.partial` area
* ensure incomplete files retain a `.partial` suffix
* verify every selected resource
* publish only after all selected resources for the logical item pass
* roll back ordinary publish failures
* document the narrow process-crash window

True crash-safe manifest-gated Source Intake can be deferred to the durable-run milestone.

Important requirement:

Because current Source Intake recursively scans `.partial` directories, directory name alone is not enough protection. Every incomplete file must have a temporary `.partial` suffix or equivalent protection so Source Intake cannot mistake it for a complete staged file.
