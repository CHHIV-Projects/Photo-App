# Milestone 12.62.28.3 — Historical Backfill Acquisition Execution + Source Intake Handoff

## Purpose

Implement the backend execution path for historical iCloud backfill acquisition from inventory.

12.62.28.2 added the dry-run acquisition preview: select eligible inventory rows, re-resolve them against a fresh helper listing, count logical/resource previews, and expose the preview through Admin API without downloads, staging, Source Intake, or Vault writes.

12.62.28.3 should now add the **real execution path**, guarded by safe defaults:

```text
POST /api/admin/icloud-backfill/acquire
dry_run = true by default
real acquisition only when dry_run = false is explicitly requested
```

When `dry_run=false`, the service may perform real acquisition/download/staging, but only through the existing exact-selection/durable acquisition path, and then hand off to existing Source Intake. Source Intake remains the only path into Vault.

```text
Historical backfill inventory
    ↓ selection ledger

    ↓ exact-selection selected item/resource manifest
Durable iCloud acquisition
    ↓ staged files
```

```text

```

## Core design decision

12.62.27 inventory is a **selection ledger**, not a complete download manifest.

Therefore, 12.62.28.3 must:

```text
- select inventory rows
- re-resolve selected rows against a fresh helper listing
- build exact-selection selected item/resource data from fresh helper metadata
- use existing exact-selection/durable acquisition machinery for download/staging
- hand staged results to existing batch Source Intake
- update inventory/backfill state only after Source Intake or exact-known resolution
```

Do not construct download requests from inventory rows alone.

## Fresh helper re-resolution

Use existing helper/list-candidates path.

Execution service should:

```text
1. Select bounded eligible inventory rows.
2. Run fresh helper metadata listing.
3. Match inventory.remote_identity to fresh helper listing item.item_id.
4. Skip stale/missing rows as retryable.
5. Build exact selected logical-item/resource data from fresh helper listing metadata.
6. Reject or skip unsafe/partial manifests.
7. Pass only safe matched selections into existing exact-selection/durable acquisition path.
```

Rows missing from fresh listing should remain retryable:

```text
backfill_completed = false
backfill_resolution_state = "skipped_stale_retryable" or equivalent
```

Safe unrelated rows must not be blocked by stale/unsafe rows.

## Source Intake handoff

For `dry_run=false` and `auto_run_source_intake=true`:

```text
1. execute durable acquisition/download/staging through existing path
2. confirm durable batch/staging is ready
3. call existing batch Source Intake handoff
4. update backfill inventory rows only after Source Intake reports success or exact-known resolution
```

If Source Intake fails or is blocked:

```text
backfill_completed must remain false
row state should be failed_retryable or source_intake_failed
response stop_reason should be source_intake_failed or source_intake_blocked
next_safe_action should indicate retry/review
```

Do not mark rows completed merely because download/staging succeeded.

## Inventory/backfill state updates

Rules:

```text
dry_run=true:
  do not update inventory completion state

dry_run=false before download:
  may increment attempt count and set acquisition_state = "selected" or "acquiring"

download/staging succeeds but Source Intake not yet successful:
  do not set backfill_completed=true

Source Intake succeeds:
  set backfill_completed=true
  set backfill_completed_at
  set backfill_resolution_state appropriately
  record acquisition_run_id/acquisition_batch_id/source_intake_run_id where available

Source Intake exact duplicate/provenance-known result:
  set backfill_completed=true only if Source Intake result proves the row is resolved
  use resolution state such as exact_duplicate_resolved or already_known

stale/missing helper listing:
  do not set backfill_completed=true
  mark retryable skip/state if additive fields exist
```

State updates must be idempotent. Retrying should not duplicate completed inventory rows.

### 2. Durable acquisition entrypoint

Approved, but keep it narrow.

If `run_durable_exact_selection_batch()` cannot be reused directly because it performs its own listing/planning, add the smallest reusable durable entrypoint that accepts an already-selected backfill manifest prepared from fresh helper metadata.

The entrypoint should still reuse existing exact-selection/durable acquisition structures and staging behavior. Do not create a parallel downloader or parallel staging path.

Stop and ask if this requires broad refactoring.

### 3. Listing cap versus acquisition cap

Correct.

Keep these separate:

```text
max_listing_candidates: up to 100000 for fresh metadata listing / re-resolution
acquire_limit / selected logical items: max 1000
download/acquisition selected-item cap: max 1000
```

Do not weaken the existing 1000 acquisition safety cap.

### 4. Source Intake result mapping

Correct that this is the delicate part.

For 12.62.28.3, inventory completion must only be recorded after Source Intake or exact known-state evidence proves the logical item is fully resolved.

Download/staging success alone must not mark completion.

### 5. Mixed Source Intake outcomes / Live Photos

Yes: for a logical item with multiple selected resources, including Live Photos, mark:

```text
backfill_completed = true
```

only when **every required selected resource for that logical item** is resolved by Source Intake.

Resolved means one of:

```text
imported / processed successfully
exact duplicate linked or otherwise resolved by Source Intake
already-known / skipped-known with strong Source Intake evidence
```

If any required selected resource fails, is missing, is unprocessed, has an unsafe manifest, or cannot be mapped back to Source Intake evidence, then:

```text
backfill_completed = false
backfill_resolution_state = "failed_retryable" or "source_intake_partial_failed"
acquisition_state = "source_intake_partial_failed" or equivalent
```

Record the acquisition/source-intake run IDs and error details for evidence, but do not call the inventory row complete.

For Live Photos specifically:

```text
HEIC/JPEG still + MOV motion resource must both resolve
```

unless the existing Source Intake result explicitly proves the logical Live Photo item was resolved as a whole.

Partial Live Photo completion is not acceptable.

### 6. If Source Intake result granularity is insufficient

If Source Intake does not expose enough per-resource or per-logical-item result detail to safely prove completion, do not over-claim success.

In that case, either:

```text
- add the smallest necessary result mapping hook/test around existing Source Intake results, or
- stop and report that completion reconciliation cannot be safely implemented without a Source Intake result contract change
```

Do not infer completion from file names or download success.

### 7. Retry/idempotency

Completed rows must not be reselected.

Rows with stale helper listing, partial Source Intake failure, blocked Source Intake, or unsafe manifest should remain retryable unless there is a clear terminal reason.

Retrying should not duplicate completed inventory rows or bypass existing durable acquisition/Source Intake idempotency.

Proceed with 12.62.28.3 under these rules.
