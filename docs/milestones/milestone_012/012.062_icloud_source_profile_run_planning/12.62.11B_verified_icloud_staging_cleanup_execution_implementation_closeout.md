## 1. Scope Completed

Implemented the guarded preview-to-execution workflow for verified local iCloud staging cleanup.

Delivered:

- side-effect-free cleanup candidate planning shared by dry run and execution
- mandatory staged-file and Vault-file SHA-256 verification
- deterministic manifest fingerprinting across all observed staging files

- 10-minute dry-run authorization window

- fail-closed manifest recomputation before deletion

- structured protected, verification-failed, missing-file, and delete-failed outcomes

No real cleanup was run against the user's iCloud staging folders. Automated deletion validation used temporary staging and Vault fixtures only.

## 2. Operational Behavior

### Cleanup readiness

The new endpoint:

```text
GET /api/admin/icloud-staging-cleanup/readiness?source_id=<id>
```

validates cleanup independently from iCloud authentication. Cleanup requires:

- an existing active Source Profile
- `source_type = cloud_export`
- `cloud_provider = icloud`
- `acquisition_method = icloudpd`
- existing source-root and managed-staging directories
- both paths to resolve to the canonical path derived for the selected Source Profile
- the canonical path to remain below `storage/exports/icloud`
- no active acquisition, Source Intake, or cleanup conflict

### Dry-run planning

The existing run endpoint remains the dry-run entry point:

```text
POST /api/admin/icloud-staging-cleanup/run
{
  "source_id": 123,
  "dry_run": true
}
```

`dry_run=false` is now rejected with `GUARDED_EXECUTION_REQUIRED`.

The planner scans the selected canonical staging root and classifies files without deleting. An eligible file requires:

- safe resolved regular-file containment under the selected staging root
- exact selected-source provenance and normalized relative path
- one unambiguous provenance/Asset SHA-256
- staged content hashing to that SHA-256
- linked Asset row
- regular Vault file under the configured Vault root
- Vault content hashing to the Asset SHA-256
- no unresolved failure/deferred conflict evidence

Vault hashes are cached within each planning pass when multiple staging paths map to the same Vault object.

After successful dry run, the run stores a deterministic manifest fingerprint and expires 10 minutes after completion.

### Dry-run binding

DB-backed authorization state is stored on `IcloudStagingCleanupRun`:

- `manifest_fingerprint`
- `planner_version`
- `preview_expires_at`
- `authorized_dry_run_id`
- `authorization_consumed_at`
- progress/stage fields
- protected/verification/missing/delete-failure counts

The fingerprint covers the complete observed plan so additions and other non-eligible changes also invalidate execution. For eligible files it includes:

- normalized relative path
- staged SHA-256
- provenance/Asset SHA-256
- file size and stable modification time
- resolved Vault path
- Vault SHA-256

For non-eligible files it includes classification, reason, path, size, and stable modification time where available. Rows are sorted before canonical JSON hashing, making the fingerprint independent of scan order.

### Guarded execution

Execution uses:

```text
POST /api/admin/icloud-staging-cleanup/execute
{
  "source_id": 123,
  "dry_run_run_id": 456,
  "explicit_confirmation": "DELETE LOCAL STAGING COPIES"
}
```

The server requires:

- exact confirmation phrase `DELETE LOCAL STAGING COPIES`
- an existing completed dry run for the same source/root
- current planner version and a stored fingerprint
- at least one eligible candidate
- unused authorization
- authorization younger than 10 minutes
- no active conflicting operation

Authorization is consumed atomically when the execution row is created. It cannot be reused.

The background executor recomputes the entire plan and fingerprint. Any difference causes a failed execution with zero deletions and guidance to run another dry run. It does not delete the unchanged intersection.

Immediately before every unlink, the executor again validates source identity, resolved containment, regular-file status, relative path, size, staged SHA-256, Vault containment, regular-file status, and Vault SHA-256.

### Reporting and partial failures

Before the first deletion, execution creates its JSON report and an adjacent append-only `.events.jsonl` audit journal. Every per-file outcome is flushed and synchronized to the journal.

Final results distinguish:

- `deleted`
- `protected`
- `verification_failed`
- `file_missing`
- `delete_failed`

Known protected files do not make an otherwise successful execution fail. A revalidation or deletion failure produces `completed_with_errors`. If final JSON report replacement fails after deletion, the run records the error and the append-only audit journal remains available.

Empty descendant directories are removed best effort. The selected source root is retained.

### Operation start locking

Acquisition, Source Intake, cleanup dry run, and cleanup execution API check-and-start sections now use one shared lock boundary.

On PostgreSQL, a dedicated connection holds a transaction-scoped advisory lock across the entire check-and-start section. It is intentionally separate from the request session so commits inside existing start services cannot release it early. An in-process reentrant lock additionally protects the local process and serves as the test/non-PostgreSQL fallback.

After acquiring the lock, each endpoint rechecks operation conflicts before creating its run.

### Guided UI

The Ingestion Source Profile flow now:

- treats `pending` as active and polls it
- shows progress, stage, eligibility, protected, verification, missing, deletion, and report summaries
- exposes execution only for a selected iCloud profile with a fresh completed unconsumed dry run
- shows profile, canonical local staging path, dry-run run/time/expiry, eligible count/bytes, protected count, and report
- requires acknowledgement plus the typed phrase `DELETE LOCAL STAGING COPIES`
- states that iCloud, Vault, DB, provenance, Source Profile, and source-registry records are untouched
- refreshes cleanup readiness/status and workflow summary after start

The historical Admin `Execute Cleanup` button was removed. Admin retains preview/status visibility and directs operators to the guided Ingestion Source Profile flow.

```text

```

remains valid for `dry_run=true`; `dry_run=false` is rejected.

### Cleanup-run schema additions

Added through the existing idempotent schema-ensure pattern:

```text
total_files
processed_files
current_stage
protected_count
verification_failed_count
file_missing_count
delete_failed_count
manifest_fingerprint
planner_version
preview_expires_at
authorized_dry_run_id
authorization_consumed_at
```

Existing rows receive safe count defaults. Old dry runs lack planner/fingerprint authorization and cannot be used for execution.

## 5. Safety Boundaries Preserved

- No iCloud/cloud deletion API is called.
- Only local files under the selected canonical managed staging directory can reach unlink.
- Vault files are hashed and verified but never modified or deleted.
- Drop Zone files are not scanned or deleted.
- Asset, provenance, Source Profile, source registry, and other DB records are not deleted.
- Source Intake remains the only ingestion authority.
- Cleanup remains manual and operator-triggered.
- Cleanup does not start acquisition, Source Intake, reacquisition, or scheduled work.
- Candidate changes fail closed and delete nothing.
- Frontend confirmation is backed by independent server enforcement.
- No Apple credentials, sessions, tokens, passwords, or 2FA values are read, stored, or logged.
- Local/external profiles do not receive cleanup execution controls.

## 7. Deviations from Prompt

- No dedicated frontend test framework exists in the current package, so guided UI behavior was validated through TypeScript checking and the production Next.js build rather than component tests.
- No real user staging cleanup was run. This is intentional; real cleanup/reacquire validation remains 12.62.12.
- Cross-operation advisory locking is enforced at all current API start surfaces. Direct internal service calls retain their service-specific checks; callers outside the API should not be introduced without using the shared protected start boundary.

## 8. Known Limitations

- Initial large-library cleanup will be I/O-heavy by design. Dry run hashes staging and Vault content; execution recomputes the plan and then rehashes each candidate immediately before unlink. This prioritizes archival safety.
- Planner data is retained in memory and full candidate detail is serialized to JSON reports. Normal UI responses remain bounded summaries, but very large libraries should be performance-tested before production use.
- The schema additions were validated through model/tests and existing schema-ensure conventions, but the real development PostgreSQL schema was not mutated during automated testing.
- PostgreSQL advisory locking was unit-tested through the lock-selection path; multi-process contention should be exercised during production-runtime validation.
- Source Intake negative evidence is historically sample-limited. Positive provenance + staged hash + Asset + Vault hash is therefore the authoritative cleanup proof, while known failure/deferred conflicts remain conservatively protected.
- Empty-directory removal remains best effort and may leave empty folders on Windows.
- Cleanup stop/cancel is intentionally not implemented because interruption during destructive work needs separate partial-operation design.
