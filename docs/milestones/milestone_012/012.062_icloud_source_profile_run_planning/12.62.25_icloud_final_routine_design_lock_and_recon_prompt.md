# Milestone 12.62.25 — iCloud Final Routine Design Lock + Recon

## Objective

Perform a **reconnaissance-only design lock** for the final iCloud ingestion routine.

Do **not** implement the final routine yet.

The purpose of this milestone is to inspect the current codebase and produce a clear implementation plan for separating:

1. **Recent iCloud Sync**

2. **Historical iCloud Backfill**

3. **Future optional automation**

This milestone should answer what state, cursor, endpoint, service, and UI changes are needed before coding begins.

---

## Current project intent

The Photo Organizer project treats iCloud as a source of files, not as a separate library pipeline.

The architectural boundary is:

```text
iCloud Acquisition
    ↓ staged files + manifest
Source Intake / Ingestion
    ↓ assets + provenance + metadata + duplicate/canonicalization logic
Photo Library
```

### Acquisition owns

```text
- talking to iCloud
- scanning/listing candidates
- classifying remote candidates for eligibility
- downloading selected files into controlled local staging
- preserving logical asset pairing, especially Live Photo HEIC + MOV pairs
- reporting unsupported / skipped / ambiguous candidates
- handing staged files/manifests to Source Intake
- cleaning only local staged/temp files
```

### Source Intake / Ingestion owns

```text
- hashing
- exact duplicate detection
- Asset creation/reuse
- provenance recording
- metadata extraction
- canonicalization
- Vault placement
- existing duplicate/canonicalization behavior
```

Do **not** redesign duplicate grouping, canonical scoring, metadata conflict rules, demote/promote behavior, or duplicate review workflow during this milestone.

Small ingestion-boundary observations are allowed, for example if iCloud provenance recording appears missing or if the current acquisition handoff bypasses Source Intake, but do not fix those yet unless explicitly approved later.

## Target final behavior, for planning purposes only

Do not implement this yet. Use it to guide reconnaissance.

### Recent Sync

User-facing idea:

```text
Get New iCloud Assets
```

Expected behavior:

```text
- start from newest iCloud candidates
- scan up to 1000 recent logical candidates
- acquire up to 500 logical assets by default
- allow future/admin override up to 1000 logical assets
- preserve current all_supported_media behavior
- preserve Live Photo logical asset handling
- hand off to Source Intake
```

Vacation scenario:

```text
If the user takes 500+ photos/videos on a trip, recent sync should be able to see them within the newest 1000 candidates.
```

### Historical Backfill

User-facing idea:

```text
Continue Historical iCloud Backfill
```

Expected behavior:

```text
- separate onboarding/backfill flow
- manual launch for now
- resume from durable historical cursor/state
- do not restart from newest every time
- acquire up to 500 logical assets by default
- allow future/admin override up to 1000 logical assets
- inspect multiple older candidate pages/chunks as needed
- proposed max inspected candidates per run: 10000
- mark complete only when true historical end of iCloud listing is reached
```

Historical backfill is mainly for initial onboarding. Once complete, normal use should rely on recent sync only unless the user/admin explicitly requests a reconcile/audit.

### Future automation

The design should support future mini-server automation, but this milestone should assume manual operator-launched actions.

Future automation should reuse the same services/state, not require a separate ingestion architecture.

Answer carefully:


```text

- conservative page/offset with validation
```

### 4. Inventory table decision

Determine whether the historical backfill should require an inventory table before acquisition.

Evaluate two possible designs:

#### Option A — Direct cursor backfill

```text
Historical backfill directly pages through iCloud listing and acquires eligible assets.
```

#### Option B — Inventory-first backfill

```text
Metadata-only inventory scan records iCloud candidates first.
Acquisition later reads eligible unknown inventory rows.
```

For each option, report:

```text
pros
cons
risk
implementation complexity
whether it fits current code
whether it supports future mini-server automation
```

Make a recommendation.
