# Goal

Add **iCloud staging cleanup readiness** and **cleanup dry-run** support to the Ingestion tab.

This milestone should let the operator answer:

```text
Can staged iCloud acquisition files be safely cleaned up later?
Which files would be eligible?
Which files are protected?
Which files are skipped?
Why?
```

This milestone must **not delete any files**.

It is a dry-run-only milestone.

# Cleanup Readiness States

Use conservative states.

Suggested readiness labels:

```text
Not ready
Ready for dry run
Dry run running
Dry run complete
Attention needed
```

## Ready for Dry Run

Show when:

```text
- profile is iCloud
- profile is active
- source/profile path readiness has no hard blockers
- no acquisition/source-intake/cleanup operation conflict exists
- staging path is approved/aligned
- at least some acquisition/intake context exists OR staged path may contain files
```

## Not Ready / Attention Needed

Show when:

```text
- profile inactive
- readiness not_ready
- path mismatch
- source registration mismatch
- approved root blocked
- active operation conflict
- auth required/session expired, if existing readiness treats it as blocking
- missing source id / source label / staging path needed by cleanup API
```

## Warning-Allowed State

Allow dry run with warning when:

```text
- no recent acquisition exists
- no recent Source Intake exists
- staged inventory count is unavailable
```

provided hard path/source/readiness checks are valid.

Warning text:

```text
Recent acquisition or Source Intake evidence is incomplete. Cleanup dry run will inspect staged files and report what would be eligible, protected, or skipped. No files will be deleted.
```

# Answers to Coder Questions — Milestone 12.62.9

## Overall Direction

Proceed with implementation.

Use the existing cleanup run/status flow and wire it into the Ingestion details drawer for iCloud profiles only.

The Ingestion UI must call cleanup with:

```text
dry_run = true
```

There should be no Ingestion UI path that sends:

```text
dry_run = false
```

Actual cleanup execution remains out of scope.

---

## 1. Cleanup status block: active only or always visible?

Show the **iCloud Cleanup Readiness** section whenever the selected profile is an iCloud profile.

Inside that section:

```text
Idle state:
  Show readiness, dry-run availability, and latest dry-run result if available.

Active state:
  Show Cleanup Dry Run running/status block while dry run is active.

Terminal state:
  Show Cleanup Dry Run Result after dry run completes/fails/stops.
```

So yes, the section is visible when idle, but the detailed “running” status block should only appear while a dry run is active.

Preferred layout:

```text
iCloud Cleanup Readiness
  - readiness badge
  - short explanation
  - Run Cleanup Dry Run button

Cleanup Dry Run Result
  - show only after a dry run result exists
```

Do not create a large permanent status panel if there is no dry-run activity or result.

---

## 2. Canonical dry-run summary fields

Use whatever the backend currently exposes. Do not force new backend fields unless the execution summary already computes them and they are safe to expose through existing response schemas.

Preferred UI field priority:

```text
1. eligible count
2. protected count
3. skipped count
4. failed/error count
5. bytes eligible, if already available
6. report path/reference
7. dry_run flag / mode
8. final status
```

If the backend exposes only a subset, show the subset clearly.

Minimum acceptable dry-run result display:

```text
- final status
- dry_run = true
- eligible files, if available
- protected/skipped files, if available
- failed/error count, if available
- report path/reference, if available
```

Important wording:

```text
Eligible for future cleanup
```

not:

```text
Safe to delete now
```

Reason: actual cleanup execution is not available in this milestone.

---

## 3. Cleanup readiness derivation

Derive cleanup readiness primarily from existing readiness/guardrail snapshots.

Use recent acquisition/intake evidence as **context and warning**, not as a hard blocker unless the backend cleanup dry-run API itself requires it.

Hard blockers:

```text
- profile is not iCloud
- profile inactive
- path mismatch
- source registration mismatch
- approved root blocked
- active operation conflict
- missing source id / source label / required cleanup payload data
- readiness_status = not_ready for hard path/source reasons
```

Warning-allowed states:

```text
- no recent acquisition
- no recent Source Intake
- no live staged inventory count
- incomplete summary evidence
```

In warning-allowed states, the user may still run dry run if path/source/guardrail checks are valid.

Reason:

```text
Dry run is the safe discovery step. It should inspect staged files and report eligibility without deleting anything.
```

Overall readiness labels:

```text
Ready for dry run
Ready with warning
Not ready
Dry run running
Dry run complete
Attention needed
```
