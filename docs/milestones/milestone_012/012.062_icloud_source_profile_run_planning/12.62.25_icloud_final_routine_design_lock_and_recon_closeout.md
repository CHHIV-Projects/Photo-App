# Milestone 12.62.25 — iCloud Final Routine Design Lock Closeout

## 1. Executive Summary

This milestone performed a read-only design recon to answer the questions required before implementing
the final iCloud ingestion routine split into Recent Sync and Historical Backfill.

Key findings:

- The current internal single-flow run is already bounded, uses logical-asset counting, and passes
  all_supported_media correctly. It is the right starting chassis.
- `MAX_CANDIDATE_SCAN_LIMIT = 500` is hardcoded in the helper protocol and the API request schema.
  Both must be raised to support a recent_candidate_scan_limit of 1000.
- No stable listing cursor exists. icloudpd via the internal provider returns candidates in whatever
  order pyicloud_ipd hands back the library, which is observed to be newest-first but is NOT
  guaranteed. A persistent remote asset identity table is the safest supplement for historical backfill.
- Durable state exists for run/batch/orchestration and cleanup, but has no historical cursor/progress
  field. A new `IcloudBackfillState` model is required.
- The architecture boundary (acquisition → Source Intake → Vault) is intact and must not change.
- Safety boundaries from 12.62.24 are all still in place.

## 3. Current iCloud Flow Map

```
UI (IcloudInternalRunCard)
  → POST /api/admin/internal/icloud-runs
      body: { source_id, batch_size, total_limit, candidate_search_cap, media_scope, auto_cleanup_if_safe }
  → internal_icloud_run_service.start_internal_single_flow_run()
      maps media_scope → ordinary_still_only bool
  → internal_loop_orchestrator.start_internal_icloud_loop()
      phases a bounded multi-batch loop:
        [phase 0] phase0_validation — staging clean? partial workspace clean?
        [dry run] exact_selection_adapter.list_candidates()
                  → icloud_exact_selection_helper (IcloudpdInternalProvider via pyicloud_ipd)
                  new_count_planner.plan_explicit_new_count_selection()
                    classify each logical item as: already_known / safe_unknown / ambiguous / unsupported
                    select up to batch_size logical items
        [execution] durable_exact_service.execute_exact_selection()
                    downloads selected files to staging path
        [source intake] batch_source_intake_service → source_intake_execution_service
                        hashes, deduplicates, writes Vault, records provenance
        [cleanup dry run] icloud_staging_cleanup_execution_service (dry_run=True)
        [cleanup execute] icloud_staging_cleanup_execution_service (execute with confirmation)
        [final verification] cleanup dry-run confirms staging clean
      loop repeats until total_limit reached or no more eligible candidates
  → IcloudOrchestrationRun / IcloudOrchestrationBatch written to DB throughout
  → report JSON written to storage/logs/icloud_internal_loop_reports/
  → GET /api/admin/internal/icloud-runs/{run_id} polls status
```

## 4. Existing Durable State Findings

### What exists today

| Model | Table | Purpose |
|---|---|---|
| `IcloudAcquisitionRun` | `icloud_acquisition_runs` | Single icloudpd execution record (source, mode, counts, manifest, stop reason) |
| `IcloudAcquisitionBatch` | `icloud_acquisition_batches` | Exact-selection batch state (selected/downloaded/intake counts, cleanup readiness) |
| `IcloudOrchestrationRun` | `icloud_orchestration_runs` | Internal loop controller (batch_size, total_limit, scan_limit, ordinary_still_only, pause_before_cleanup, completed_logical_items) |
| `IcloudOrchestrationBatch` | `icloud_orchestration_batches` | Per-batch checkpoint (acquisition/intake/cleanup run IDs, selected/intaken counts, candidates_considered) |
| `IcloudStagingCleanupRun` | `icloud_staging_cleanup_runs` | Cleanup dry-run and execution records |

### Is this sufficient for recent sync?

Mostly yes. With a raised `MAX_CANDIDATE_SCAN_LIMIT` (500 → 1000) and updated `total_limit` default (500),
the existing loop machinery handles recent sync.

### Is this sufficient for historical backfill?

No. Missing fields:

- No historical cursor (no last-seen asset identity, no page offset, no last-scanned date boundary)
- No backfill-mode state (not_started / in_progress / paused / complete)
- No per-source backfill progress tracking
- No total candidates inspected across multiple runs
- No "reached historical end" marker

### What is needed

A new `IcloudBackfillState` model (or equivalent fields added to `IngestionSource`).
See Section 9 for proposed schema.

## 5. Listing Order / Cursor Findings

### How candidates are currently listed

The helper (`IcloudpdInternalProvider`) uses pyicloud_ipd's internal library API. The listing returns
logical items from the iCloud Photos Library in whatever order pyicloud_ipd iterates the remote library.

### Is listing order guaranteed?

No. Observed to be newest-first in practice (as iCloud Photos typically presents recents at the top),
but this is not a documented guarantee of pyicloud_ipd or the CloudKit API. It is an observed behavior.

### Does a stable cursor exist?

No. There is no page token, offset, date cursor, or asset ID checkpoint exposed by the current helper
protocol or the pyicloud_ipd library. The `candidate_scan_limit` only caps how many items are ingested
from the front of the listing, not where to resume from.

### Implications

- Recent sync can safely rely on observed newest-first order: scan the first 1000 candidates.
- Historical backfill cannot reliably page backward without additional anchoring.

### Recommended supplement for historical backfill

**Persistent remote asset identity table** (safest option):
Scan and record asset identities from iCloud into a local inventory table (asset ID, relative path,
observed date, known status) before or during each backfill run. Use the inventory to determine what
has not been acquired yet, independent of listing order.

This avoids re-scanning from page 0 every run and avoids depending on order stability.

## 6. Inventory-First vs Direct-Cursor Recommendation

### Option A — Direct cursor backfill

| | |
|---|---|
| Pros | No new table; simpler initial implementation |
| Cons | No stable cursor from pyicloud_ipd; order not guaranteed; must rescan from top every time unless rough page offset used; fragile for large libraries |
| Risk | Medium-high: rescan cost grows with library size; may re-download already-known assets |
| Complexity | Low initial, high long-term reliability cost |
| Fits current code | Partially — would need offset tracking added to protocol |
| Supports future automation | Poorly — no anchor for mini-server to resume cleanly |

### Option B — Inventory-first backfill (recommended)

| | |
|---|---|
| Pros | Stable inventory anchors backfill progress independent of listing order; supports pause/resume; supports future mini-server automation; aligns with architecture (acquisition classifies remote candidates → intake records assets) |
| Cons | Adds one model/table and a separate scan phase before acquisition |
| Risk | Low — non-destructive scan only; acquisition unchanged |
| Complexity | Moderate — new model, new scan endpoint, updated backfill state machine |
| Fits current code | Well — scan is a dry listing pass; acquisition unchanged |
| Supports future automation | Yes — mini-server reads unacquired inventory rows |

**Recommendation: Option B.**

The inventory-first approach is the only design that survives large libraries, can be paused/resumed
safely, and supports future automation without architectural re-work.

## 8. Historical Backfill Implementation Plan

### Phase 1 — Inventory scan (new)

A new background task scans the iCloud listing and writes logical item identities to
`icloud_remote_asset_inventory`. This is a dry listing pass — no downloads.

### Phase 2 — Acquisition from inventory

Existing acquisition flow reads from inventory rows where `acquired = False`,
sorted by `observed_remote_position ASC` (oldest-first for true historical coverage).

### Phase 3 — State tracking

`IcloudBackfillState` tracks per-source backfill progress across multiple runs.

### Statuses (proposed)

```
not_started → in_progress → paused → complete
                                   → failed
```

### Stop reasons (proposed)

```
acquire_limit_reached
candidate_inspection_cap_reached
historical_end_reached
operator_cancelled
safety_blocked
preflight_failed
no_eligible_candidates_found
error
```

### Admin UI controls needed

- "Start Historical Backfill" button (launches initial inventory scan + first acquisition run)
- "Continue Backfill" button (resumes from last cursor/inventory position)
- Status card showing: total_inventory_count, acquired_count, remaining_count, current_status, last_run_at
- "Pause" / "Reset" with confirmation guard

- `selected_resources` counts actual downloaded files (Live Photo pair = 2 resources)

Confirmed in `IcloudOrchestrationBatch` model:
- `selected_logical_items` (int)
- `selected_resources` (int)

Confirmed in `internal_loop_orchestrator.py`:
- `completed_logical_items` drives total_limit comparison
- `is_ordinary_still_logical_item()` helper imported for type classification

### No correction needed

The existing logical asset counting is correct for both recent sync and historical backfill targets.

## 15. Open Questions / Risks

1. **pyicloud_ipd listing order**: Observed as newest-first but not guaranteed. If order changes, recent
   sync correctness depends on the first N candidates still being the most recent.
   Mitigation: add a `listing_order_observed` metadata field to reports so regressions are detectable.

2. **Remote asset identity**: pyicloud_ipd may or may not expose a stable unique asset ID per logical item.
   If it does not, the inventory table must use relative_path + observed_date as composite identity.
   This needs verification before `IcloudRemoteAssetInventory` is finalized.

3. **iCloudpd library listing scale**: Scanning the full library for inventory (potentially thousands of
   items) via the internal helper may be slow or may hit rate limits.
   Mitigation: chunk inventory scans and persist progress as rows are written.

4. **MAX_SELECTED_ITEM_COUNT raise**: The protocol constant caps how many items can be passed in a
   download payload. Raising from 500 to 1000 should be safe, but the helper JSON payload size limit
   (`MAX_HELPER_JSON_BYTES = 5_000_000`) should be validated at the raised item count.

5. **`IcloudOrchestrationRun.candidate_scan_limit` DB field**: Currently stored as int. After raising
   the cap to 1000, existing rows with value 500 remain valid. No migration required.
