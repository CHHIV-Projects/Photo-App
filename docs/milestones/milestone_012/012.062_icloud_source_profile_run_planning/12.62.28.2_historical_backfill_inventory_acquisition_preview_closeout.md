# Milestone 12.62.28.2 - Historical Backfill Inventory Acquisition Preview

## 1. Summary

Implemented a backend-only dry-run acquisition preview for historical iCloud backfill inventory rows.

The preview selects eligible inventory rows, re-resolves those rows against a fresh helper metadata listing, and returns count-focused preview results through a new Admin API endpoint. It does not download, stage, invoke Source Intake, write Vault assets, clean up files, mark rows completed/acquired, add UI, or add scheduler behavior.

## 3. Implementation Details

Added `preview_icloud_backfill_acquisition()` in `backend/app/services/icloud_backfill_acquisition_preview_service.py`.

Behavior:

- validates source profile exists, is active, is iCloud-backed, uses `icloudpd`, and has an account username,
- validates `acquire_limit` between 1 and 1000,
- validates `max_listing_candidates` between 1 and 100000,
- selects only inventory rows with:
  - `eligibility_state = "eligible_metadata_only"`,
  - `known_state = "pending_known_state_check"` or `"unknown"`,
  - populated `remote_identity_basis`,
  - populated `remote_identity`,
- skips and counts unsupported, ambiguous, pending-classification, known, and missing-identity rows,
- orders selection by:
  - `created_remote_at ASC NULLS LAST`,
  - `added_remote_at ASC NULLS LAST`,
  - `first_observed_at ASC`,
  - `id ASC`,
- does not use `observed_remote_position` as a durable cursor,
- calls the helper listing path only through `list_candidates`,
- matches inventory `remote_identity` to fresh helper listing `item_id`,
- counts stale/missing helper listing rows as retryable preview skips,
- counts preview logical items and resources from fresh helper listing metadata,
- treats Live Photo rows as one logical item while counting all helper-exposed resources,
- marks partial/incomplete Live Photo manifests unsafe for execution preview without blocking safe unrelated rows,
- returns count-focused results by default.

Optional `include_items` returns only local inventory IDs, resource counts, Live Photo flag, and primary relative path. It does not return helper item IDs, credentials, tokens, or URLs.

## 5. Schema/Model Changes

No database schema or model changes were made.

Caution preserved from pre-coding review: current 12.62.27 inventory rows do not have `acquired` or `backfill_completed` fields. The preview therefore does not add or infer completion fields, and it does not mark rows acquired/completed. Future completion semantics should use safer naming such as `backfill_completed`, `backfill_completed_at`, and `backfill_resolution_state`.

## 9. Known Limitations

- Preview depends on a fresh helper listing to expose matching item IDs and complete resource metadata.
- Rows absent from the fresh listing are reported as stale/retryable but are not persisted with a skip reason in this milestone.
- The preview does not perform strong local known-state detection beyond filtering existing `known_state` values on inventory rows.
- Because no completion fields exist yet, the preview cannot exclude future completed/acquired rows until 12.62.28.3 or later adds those fields.
- Large real helper listings may still hit runtime or response-size constraints despite the 100000 metadata cap.
- Helper item IDs remain observed helper identities, not documented permanent CloudKit/iCloud guarantees.

## 10. Recommended Next Milestone

Milestone 12.62.28.3 should implement durable acquisition/download from the previewed selection and Source Intake handoff.

Recommended 12.62.28.3 scope:

- stage only into the managed iCloud staging path,
- hand off to existing batch Source Intake,
- update inventory rows only after Source Intake or exact known-state resolution,
- keep cleanup behind existing dry-run/review controls.
