## Product Direction

The durable iCloud model remains:

```text
Batch size: N
Total limit: M
Mode: Download only new exact-selection items

Acquire up to N new logical iCloud items
→ Source Intake those N
→ Cleanup those local staging copies
→ checkpoint
→ repeat until M total items, no more new items, user stop, or failure
```

This milestone should remain internal and bounded.

Do not jump to the final UI.

Do not run a large unattended ingestion.

## Critical Terminology Clarification

Do not treat `scan_limit` as a user-facing equivalent to batch size or total limit.

Going forward, use these semantics:

```text
batch_size
  How many new logical iCloud items to process per acquire → Source Intake → cleanup cycle.

total_limit
  How many new logical iCloud items to process across the whole orchestration run.

candidate_page_size
  How many iCloud candidate items to inspect per listing page/window.

candidate_search_cap
  Maximum total iCloud candidates to inspect while trying to fill one batch.
```

The user-facing future workflow should eventually expose mainly:

```text
Batch size
Total limit
Download only new items
```

Candidate page size and candidate search cap should usually be internal engine settings or Advanced options, not normal user decisions.

## Example Desired Behavior for Large iCloud Library

For a user with a large iCloud library, for example 20,000 photos:

```text
User request:
batch_size = 500
total_limit = 2000

Internal policy:
candidate_page_size = 100
candidate_search_cap = 2000 or another safe internal cap
```

The planner should behave conceptually like this:

```text
Batch 1:
scan page 1 of 100 candidates → find 50 new safe items
scan page 2 of 100 candidates → find 100 new safe items
scan page 3 of 100 candidates → find 50 new safe items
scan page 4 of 100 candidates → find 100 new safe items
scan page 5 of 100 candidates → find 75 new safe items
scan page 6 of 100 candidates → find 100 new safe items
scan page 7 of 100 candidates → find 100 new safe items, but only 25 more are needed

select exactly 500 for this batch
leave the remaining safe-but-not-selected candidates untouched
acquire 500
Source Intake 500
cleanup 500 local staging copies
checkpoint

Batch 2:
repeat search/selection for the next 500

Stop when:
completed logical items = 2000
or no more new safe items exist
or candidate search cap is reached
or user stops
or a failure/blocker occurs
```

Important behavior:

```text
The app should not download more than the batch target.
The app should not stage more than the batch target.
The app should not Source Intake more than the batch target.
The app should not cleanup unrelated files as part of a batch loop.
Safe-but-not-selected candidates remain untouched and are eligible for later batches.
```

## Required Semantics for Search Results

The planner/orchestrator must clearly distinguish these outcomes:

```text
batch_target_filled
  The requested batch size was filled.

partial_batch_selected
  Some new safe items were selected, but fewer than the batch target.

candidate_search_cap_reached
  The planner reached the maximum allowed candidate search depth before filling the batch.

no_more_candidates
  The helper/source can prove there are no more candidates to inspect.

no_more_new_items
  Candidates remain or were inspected, but no unknown safe items are available under the current policy.

unsupported_or_ambiguous_candidates_only
  The planner found candidates, but they were unsafe for the current ordinary-still-only policy.

operator_decision_required
  The system found fewer than batch size but more than zero items, and policy requires operator confirmation before running a partial batch.
```

For internal testing, it is acceptable to stop on partial selection unless the prompt or command explicitly allows partial batches.

For future product behavior, partial batch policy should be explicit:

```text
Option A:
Process partial batch if at least one safe item is found.

Option B:
Ask operator/user before processing partial batch.

Option C:
Continue searching deeper up to candidate_search_cap.
```

Do not silently pretend a partial batch is a full batch.

### 5. Staging verification

The internal loop must not require the separate manual UI “Verify Staging” step.

The orchestrator should automatically verify:

```text
before acquisition:
  staging has no unsafe unknown files
  no .partial files
  Drop Zone clean

after acquisition:
  selected resources exist
  local SHA-256 matches durable resource row
  no unrelated staged unknown files
  Drop Zone still clean

after cleanup:
  eligible_count = 0 or known-safe stop state
  .partial workspace clear
  staging clean
  Drop Zone clean
```

If staging is not clean, stop before the next batch.

### 6. Stop-condition clarity

Make sure the following states are clearly distinct:

```text
total_limit_reached
batch_target_filled
partial_batch_selected
candidate_search_cap_reached
no_more_candidates
no_more_new_items
unsupported_or_ambiguous_candidates_only
cleanup_review_required
cleanup_preview_expired
staged_unknown_pending_intake
partial_workspace_present
drop_zone_not_empty
source_intake_failed
cleanup_dry_run_ineligible
cleanup_execution_failed
auth_required
helper_timeout
user_stopped
unexpected_mismatch
```

Every stop condition should include a next safe action.

### 7. Retry/resume safety

Do not blindly restart.

Before resume:

```text
reconcile orchestration row
reconcile acquisition batch/resource rows
reconcile Source Intake evidence
reconcile cleanup evidence
verify staging
verify Drop Zone
verify cleanup preview still valid if continuing cleanup
```

Acceptable resume behavior:

```text
paused_for_cleanup → continue cleanup with confirmation
completed cleanup but not checkpointed → verify clean and checkpoint
batch already Source Intaked → do not re-run Source Intake blindly
batch already cleaned → do not delete again
```

Blocked resume behavior:

```text
missing files
SHA mismatch
partial files
dirty Drop Zone
ambiguous provenance
wrong Source Profile
expired cleanup preview
unclear processed/unprocessed mix
```

## Cleanup Policy

For this milestone, default remains:

```text
pause_before_cleanup: true
```

Cleanup execution requires the exact confirmation phrase:

```text
DELETE LOCAL STAGING COPIES
```

Do not add unattended cleanup as the default.

If the coder believes a guarded automation option is now safe, propose it in the closeout as a future option rather than enabling it silently.

Actual cleanup remains source-wide guarded cleanup, with the invariant:

```text
only one acquisition batch staged at a time
staging verified clean before each acquisition
staging verified clean after cleanup
```

Do not implement batch-specific cleanup execution in this milestone unless it is trivial, separately approved, and clearly safer than the existing guarded path.
