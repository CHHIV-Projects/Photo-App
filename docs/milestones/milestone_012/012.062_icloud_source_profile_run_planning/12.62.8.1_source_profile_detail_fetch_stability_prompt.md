# Preferred Fix Strategy

Prefer the smallest safe fix.

Potential fixes may include, depending on findings:

```text
- keep source profile detail fetch read-only
- avoid unnecessary row locks
- avoid writes during detail fetch
- split expensive or unrelated queries out of the critical request path
- avoid nested transactions where not needed
- avoid eager operations that cause conflicting database access
- ensure session rollback/cleanup after OperationalError
- reduce duplicate frontend detail fetches
- add frontend retry/refresh only if backend behavior is already safe
```

Do not add broad architectural changes unless necessary.

If the fix requires a migration, schema change, or semantic backend change, stop and ask before implementing.

## 1. Schema ensure / ALTER TABLE hypothesis

Treat the schema-ensure note as a hypothesis, not a confirmed fact.

Please inspect and confirm:

```text
- Does get_source_profile_detail() call any schema ensure / ALTER TABLE / migration-like operation?
- Does the source profile detail endpoint call any service that performs schema ensure?
- Are schema ensure operations happening during normal read requests?
```

If schema ensure or ALTER TABLE is in the detail fetch path, that is the first fix target.

Lock-in:

```text
Schema mutations must not run inside a concurrent-read endpoint like GET /api/admin/source-profiles/{source_id}.
```

Preferred fix if confirmed:

```text
- remove schema ensure from the detail fetch path
- move it to app startup, migration/init path, or an explicit admin maintenance path
- preserve existing endpoint response shape
```

If schema ensure is not in the path, document that in the closeout and continue investigating.

---

## 2. Frontend simultaneous fetch / drawer race

Yes, inspect whether the frontend can fire overlapping detail fetches when the user switches profiles quickly.

Preferred frontend behavior:

```text
- cancel or ignore stale in-flight detail requests when selected profile changes
- do not let an older response overwrite newer drawer state
- avoid duplicate detail fetches for the same selected profile unless explicitly refreshed
```

If feasible, add fetch de-racing even if the backend fix is also made.

Acceptable implementation:

```text
- AbortController if current API helper pattern supports it
- request sequence id / stale-response guard if AbortController would be invasive
```

Do not redesign the drawer.

## 3. Transaction/session scope

Confirm whether the endpoint uses:

```text
- implicit SQLAlchemy session scope
```


If the endpoint has a broad transaction scope, narrow it if safe.

If any write occurs in the detail path, stop and document it unless it is clearly a legacy schema ensure issue that can be safely removed from the read path.
