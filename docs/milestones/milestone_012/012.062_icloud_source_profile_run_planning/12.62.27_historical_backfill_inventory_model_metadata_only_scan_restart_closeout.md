# Milestone 12.62.27 - Historical Backfill Inventory Model and Metadata-Only Scan Closeout

## 1. Summary

Implemented the metadata-only historical backfill inventory foundation.

This milestone adds persistent remote iCloud inventory state, a per-source backfill state row, a metadata-only scan service, and Admin API endpoints to run an inventory scan and read status. The implementation does not acquire, download, stage, intake, or write Vault assets.

## 3. DB Models/Tables Added

Added `IcloudRemoteAssetInventory`:

- Table: `icloud_remote_asset_inventory`
- Purpose: one metadata-only row per observed remote logical iCloud asset identity and source profile.
- Unique identity: `(source_profile_id, remote_identity_basis, remote_identity)`.

Added `IcloudBackfillState`:

- Table: `icloud_backfill_state`
- Purpose: one per-source status/count row for historical backfill inventory scanning.
- Unique source state: `source_profile_id`.

The helper follows the existing repo convention:

- requires `ingestion_sources` to exist first,

- commits after schema sync,
- is called during FastAPI startup in `backend/app/main.py`,
- is also called by the scan/status service entrypoints.

No Alembic migration was introduced.

## 5. Inventory Identity Strategy

Inventory rows use:

```text
remote_identity = item.item_id
remote_identity_basis = "helper_item_id_observed_stable"
```

This is deliberately conservative. The code treats the helper/listing item ID as the best available observed helper identity. It does not claim CloudKit or iCloud guarantees long-term stability.

`observed_remote_position` is stored only as a scan-local ordinal from the current listing pass. It is not treated as a durable cursor.

No composite fallback was added in this milestone because the existing adapter requires a valid helper `item_id` before an item reaches the scanner.

## 6. Inventory Scan Service Behavior

Added `run_icloud_backfill_inventory_scan()` in `backend/app/services/icloud_backfill_inventory_service.py`.

Behavior:

- validates the source profile is active, iCloud, and `icloudpd` backed,
- uses helper listing metadata only via `list_candidates`,
- upserts inventory rows by source plus remote identity,
- records eligible rows as:
  - `eligibility_state = "eligible_metadata_only"`
  - `known_state = "pending_known_state_check"`
- records unsupported or ambiguous rows without blocking safe rows,
- classifies Live Photo metadata when helper listing resources expose a Live Photo grouping/resource,
- updates `IcloudBackfillState` scan counts and status,
- accepts `max_candidates` up to 100000.

The helper protocol now has a separate `MAX_LIST_CANDIDATE_SCAN_LIMIT = 100000` for list operations while preserving `MAX_CANDIDATE_SCAN_LIMIT = 1000` and `MAX_SELECTED_ITEM_COUNT = 1000` for recent-sync/acquisition limits.

## 11. Known Limitations

- The inventory scan records helper-observed identities; it does not prove long-term iCloud/CloudKit identity stability.
- Inventory scan does not check the local Photo Organizer library, so eligible rows use `known_state = "pending_known_state_check"`.
- The scan records metadata and counts only; it does not perform acquisition, pause/resume acquisition, or completion of historical backfill.
- The helper response remains bounded by the existing helper JSON response size limit. Very large real listings may require chunking or streaming in a future milestone.
- Existing tables created before this milestone will receive additive columns, but unique constraints are only guaranteed when the new tables are created fresh through SQLAlchemy metadata.
