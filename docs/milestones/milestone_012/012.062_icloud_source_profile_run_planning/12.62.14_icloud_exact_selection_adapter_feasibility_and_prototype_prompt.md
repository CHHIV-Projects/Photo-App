# Milestone 12.62.14 — iCloud Exact-Selection Adapter Feasibility and Prototype

## Goal

Determine whether Photo Organizer can safely implement true iCloud new-count acquisition by using an adapter capable of:

```text
list stable logical iCloud candidates newest-to-oldest
→ retain explicit logical-item/resource relationships
→ evaluate selected Source Profile known-state
→ select up to N new logical iCloud items
→ download exactly the selected unknown resources
→ avoid redownloading known resources in mixed known/new windows
→ report logical item counts and physical resource counts separately
```

If feasible, implement the smallest internal proof/prototype needed to validate exact selected download capability.

If not feasible without a broader adapter redesign, close with a precise technical blocker and recommended architecture path.

This milestone should use **Extra High reasoning**.

## Background

12.62.13 defined the desired iCloud new-count semantics and implemented an internal pure planner.

Approved product semantics:

```text
N = logical iCloud items, not raw resource files.
```

A Live Photo or related multi-resource iCloud item should count as one logical item while still reporting all physical resources/files.

Reports should show both:

```text
logical iCloud items selected / known / downloaded
physical resources/files selected / known / downloaded
```

12.62.13 also established that the current `icloudpd` CLI adapter cannot safely execute the planner because it lacks the required contract for stable logical IDs, continuation/pagination, explicit item/resource relationships, and exact selected-resource download.

Current production behavior remains fixed-window:

```text
inspect fixed recent window
→ if all known, skip download
→ if any unknown, download complete window
→ stop
```

That behavior is not true “Download up to N new items.”

## Product Lock-ins

The future guided iCloud flow should be:

```text
Step 1: Create Source Profile
  - not required every run

Step 2: Verify staging/readiness
  - eventually automatic or part of readiness

Step 3: Acquire from iCloud
  - acquisition owns batch, total limit, and download mode

Step 4: Run Source Intake
  - no independent user batch/limit
  - intake processes what acquisition staged

Step 5: Run Cleanup
  - cleanup acts on files verified by Source Intake
```

Future one-step model after profile setup:

```text
Run iCloud Intake
  internally performs:
    acquire batch
    → Source Intake batch
    → cleanup verified staging
    → repeat until total limit / no more new items / stop condition
```

For this milestone, do **not** implement the full acquire/intake/cleanup orchestration. First prove whether exact selected iCloud acquisition is executable.

Batch/limit semantics for the future workflow:

```text
batch_size = number of new logical iCloud items per acquire/intake/cleanup cycle
total_limit = maximum new logical iCloud items for the full coordinated run
candidate_scan_limit = backend-controlled cap for how far acquisition may search
```

Normal UI should eventually ask for target logical item counts. Candidate scan limits belong in Advanced settings or backend configuration.

## Safety-Critical Rules

Preserve these rules:

- Acquisition may only download to the selected Source Profile’s managed iCloud staging path.

- Acquisition must never write directly to Vault.

- Acquisition must never write directly to DB Asset rows or provenance.

- Source Intake remains the only path into Drop Zone, Vault, Asset rows, and provenance.

- Cleanup remains guarded by the 12.62.11B preview/execution contract.

- No iCloud cloud-library deletion.

- No Vault deletion or mutation.

- No DB/provenance/source deletion.

- No Source Profile deletion.

- No cloud credential storage.

- No secret logging.

- Ambiguous identity must block, not silently suppress or download.

- Wrong-profile known evidence must not suppress download for the selected profile.

- Unknown files already staged but not ingested should block further new-count acquisition with clear guidance: `Run Source Intake first`.

## Implementation Requirements if Feasible

If exact selected download is feasible, implement the smallest internal adapter path.

The adapter/prototype must:

1. Accept a selected iCloud Source Profile.

2. List logical iCloud candidates newest-to-oldest.

3. Preserve logical item IDs and resource relationships.

4. Evaluate known-state using selected Source Profile evidence.

5. Use the 12.62.13 planner or equivalent planner contract.

6. Select up to `target_new_item_count` logical items.

7. Download only selected unknown resources.

8. Download all required resources for each selected logical item where possible.

9. Block if unknown files are already staged but not ingested.

10. Respect a backend candidate scan limit.

11. Report stop reason:
    
    - `target_new_count_reached`
    
    - `no_more_candidates`
    
    - `scan_limit_reached`
    
    - `tooling_limit_reached`
    
    - `staged_unknown_pending_intake`
    
    - `identity_unavailable`
    
    - `logical_item_identity_ambiguous`
    
    - `auth_or_acquisition_error`

12. Report both logical item counts and physical resource/file counts.

13. Keep this internal unless sufficiently proven.

Do not expose this as a normal user-facing mode in the Ingestion UI during this milestone unless explicitly approved after prototype validation.

Answers to the three questions:

1. Yes, you may implement a source-controlled, non-UI helper executed inside `.tools/icloudpd`, using narrow JSON communication with the backend.

   Conditions:

   * The helper may use the helper environment’s existing keyring/cookie authentication.
   * Photo Organizer must not receive, store, log, or persist Apple credentials, 2FA codes, cookies, sessions, or tokens.
   * The backend/helper interface should be narrow and schema-like: request in, JSON result out.
   * The helper should remain internal and not be exposed in the UI during this milestone.
   * Any helper output must avoid secrets and should use bounded samples for large result sets.

2. Approved on sidecars/counting.

   N should count logical iCloud items, not raw resource files.

   Still/motion resources for a Live Photo should belong to one logical item where the adapter can identify that relationship safely.

   Reporting should show both:

   * logical item counts
   * total resource/file counts

   Count only remotely downloadable resources exposed by the iCloud/PhotoAsset adapter. Generated XMP should not count as a cloud resource. Unsupported, ambiguous, or unsafe sidecar relationships should block rather than being inferred.

3. Approved on item-level safety.

   Download selected unknown resources beneath a temporary `.partial` area inside the selected Source Profile’s staging root.

   Publish/move the logical item into normal staging only after every selected resource for that item succeeds and is verified.

   Known sibling resources should not be redownloaded.

   If a logical item partially fails, report it as partial/failed and do not make it appear as a complete staged item for Source Intake.

Additional confirmations:

* The planner should accept adapter-provided logical item/resource groups instead of reconstructing groups from filenames whenever the adapter can provide that information.
* Remote iCloud IDs should remain transient in this prototype. Do not persist remote IDs to DB/provenance without a separate schema/provenance approval milestone.
* Offset pagination is acceptable inside one bounded run, but it should not be described as a durable cross-run checkpoint yet.
* Since the Codex shell cannot launch the real helper environment, fixture tests are acceptable for automated validation. Any real-account validation will be user-operated by me.
