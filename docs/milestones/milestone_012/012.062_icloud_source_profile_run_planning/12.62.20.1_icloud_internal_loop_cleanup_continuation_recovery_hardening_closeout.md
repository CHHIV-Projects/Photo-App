# Milestone 12.62.20.1 — iCloud Internal Loop Cleanup-Continuation Recovery Hardening

## 1. Scope Completed

Implemented a narrow recovery hardening patch in internal cleanup continuation flow.

Completed scope:

- stale preview handling (`DRY_RUN_EXPIRED`) now blocks safely with recovery guidance
- consumed preview handling (`DRY_RUN_ALREADY_CONSUMED`) now:
  - reconciles idempotently when staging is already clean and evidence is complete
  - blocks safely with recovery guidance when staging is not clean
- blocked recovery state can accept a fresh cleanup dry-run ID and continue
- normalized operator-facing stop reasons while preserving original cleanup error codes in details
- recovery-required payload now includes next safe action and ready-to-copy command templates
- focused tests added for required recovery scenarios
- 12.62.20 closeout updated with addendum

## 2. Root Cause / Failure Mode

Continuation attempted cleanup execution using a stale/already-consumed cleanup dry run and treated authorization errors as terminal failures.

This caused orchestration status to become `failed` even for recoverable stale-preview conditions.

## 3. Behavior Changes

- Valid cleanup dry run:
  - unchanged: execute guarded cleanup, verify staging clean, complete batch, continue/finish.

- Expired cleanup dry run:
  - now blocked safely with `stop_reason=cleanup_preview_expired`
  - original code preserved in details: `original_cleanup_error_code=DRY_RUN_EXPIRED`
  - no failed-batch increment for this stale-preview condition

- Already-consumed cleanup dry run + staging clean:
  - now reconciles idempotently when Source Intake/provenance evidence is complete
  - no second deletion attempted
  - batch/run checkpointed and resumed or completed

- Already-consumed cleanup dry run + staging not clean:
  - now blocked safely with `stop_reason=cleanup_preview_already_consumed`
  - original code preserved in details: `original_cleanup_error_code=DRY_RUN_ALREADY_CONSUMED`

- Fresh dry-run recovery:
  - continuation now accepts a new cleanup dry-run ID while in recovery-blocked state
  - re-binds the paused batch cleanup dry-run ID and continues through guarded checks

- Recovery payload/output additions (for blocked recovery-required outcomes):
  - `next_safe_action`
  - `source_profile_id`
  - `current_batch_index`
  - `stale_cleanup_dry_run_id`
  - `original_cleanup_error_code`
  - `continue_cleanup_command_template`
  - `fresh_cleanup_preview_command`
