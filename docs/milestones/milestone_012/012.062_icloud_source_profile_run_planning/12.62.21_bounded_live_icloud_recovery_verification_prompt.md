# Milestone 12.62.21 — Bounded Live iCloud Recovery Verification

## Recommended Reasoning Level

High.

This milestone is primarily an operator validation milestone, not a feature-building milestone. Keep changes minimal and safety-focused.

## Goal

Perform a bounded live iCloud verification after the 12.62.20.1 cleanup-continuation recovery hardening patch.

The goal is to confirm that the internal iCloud loop can safely:

```text
Acquire a bounded batch of new ordinary stills
→ Source Intake them
→ pause for guarded local cleanup review
→ execute cleanup with explicit confirmation
→ recover safely if a cleanup preview is stale, expired, or already consumed
→ finish without falsely marking the orchestration failed
```

This milestone should answer:

```text
Does the 12.62.20.1 recovery hardening work in a real operator flow?
```

## Background

12.62.20 implemented and validated:

```text

--candidate-search-cap alias

MOV/non-still exclusion under ordinary_still_only

larger bounded live validation
```

However, the 12.62.20 live run exposed a cleanup-continuation issue:

```text
DRY_RUN_ALREADY_CONSUMED
```

12.62.20.1 then implemented recovery hardening:

```text
expired cleanup dry-run handling
already-consumed cleanup dry-run handling

```

Automated validation passed in 12.62.20.1, but no additional live iCloud retry was performed.

12.62.21 is the bounded live retry.

## Phase 2 — Normal Execution and Cleanup

If dry run is safe, execute the bounded loop.

Preferred command:

```powershell

& '.\.venv\Scripts\python.exe' '.\backend\scripts\run_icloud_internal_loop.py' `
  --execute `
  --source-id 66 `
  --batch-size 2 `
  --total-limit 2 `
  --candidate-search-cap 50 `
  --ordinary-still-only `
  --pause-before-cleanup
```

Expected behavior:

```text

create acquisition run/batch rows
run batch Source Intake
create Source Intake / ingestion run evidence
pause for cleanup review
emit cleanup dry-run ID
emit ready-to-copy continue cleanup command
```

Then continue cleanup only after reviewing the cleanup dry-run report:

```powershell
$env:PYTHONPATH='backend'
& '.\.venv\Scripts\python.exe' '.\backend\scripts\run_icloud_internal_loop.py' `
  --continue-cleanup `
  --orchestration-run-id <orchestration_run_id> `
  --cleanup-dry-run-id <cleanup_dry_run_id> `
```

Expected final behavior:

```text

staging file count is zero or known-safe

.partial workspace clear

no cloud deletion
```

## Phase 3 — Recovery Verification

Exercise one recovery path if it can be done safely without a large run.

Preferred recovery scenario:

```text

```

Acceptable alternatives:


Do not force an unsafe scenario.

Do not manually corrupt files.

```text

```


Answers to questions:

1. **Use Source Profile 66 as-is.**
   Do not modify the source profile for this exercise. If Source Profile 66 is missing, invalid, unauthenticated, or appears materially different than expected, stop and report rather than substituting another profile.

2. **Start with `batch_size=2,total_limit=2`.**
   Fall back to `batch_size=1,total_limit=1` only if the dry run shows there are not enough safe ordinary stills for a 2/2 test. Keep the scope intentionally small.

3. **Attempt one live recovery-path exercise.**
   Preferred path is an expired/stale cleanup preview recovery exercise. The goal is to confirm the recovery behavior safely, not to force a risky cleanup. If it becomes impractical, unsafe, authentication-blocked, or cannot be exercised without ambiguity, defer it and document the reason.

4. **Include backend/auth/helper-access checks as preconditions.**
   Do not assume the backend is already ready. First verify backend health, source profile access, and helper/authentication access. If already running and authenticated, simply record that verification. If not, include startup/auth check steps before proceeding.

Additional guardrail: this should remain a narrow validation exercise, not a broad iCloud ingestion run. Stop on unexpected profile state, unclear cleanup authorization state, helper authentication failure, or any condition where the recovery test could become destructive or ambiguous.
