# Milestone 12.62.17 — iCloud Acquisition Batch Source Intake Handoff

## Recommended Reasoning Level

Extra High.

This milestone connects the new internal durable iCloud exact-selection acquisition batch to Source Intake. It touches acquisition manifests, staged files, Source Intake, Vault ingestion, provenance, cleanup readiness, and batch-level reporting.

## Goal

Implement the internal handoff from a durable exact-selection iCloud acquisition batch to Source Intake.

The target behavior is:

```text
durable exact-selection acquisition batch
→ verified published resources in managed iCloud staging
→ Source Intake consumes only that batch inventory
→ Source Intake records batch/resource evidence
→ batch status reflects intake result
→ cleanup becomes safe only after Source Intake verifies the batch
```

This milestone should make **acquisition define the batch** and **Source Intake process that batch**.

Source Intake should not ask for an independent batch size or total limit in this internal iCloud exact-selection path.

## Product Lock-ins

The durable iCloud model is:

```text
Batch: 50
Limit: 1000
Mode: Download only new

Acquire 50 new logical iCloud items
→ Source Intake those 50
→ Cleanup those 50
→ checkpoint
→ repeat until total limit, no more new items, user stop, or failure
```

For the guided iCloud flow:

```text
Acquisition owns:
- batch size
- total limit
- download mode
- candidate scan limit / advanced scan policy

Source Intake owns:
- processing the acquisition batch inventory
- Vault ingestion
- Asset/provenance creation or duplicate linkage
- failures/deferred classification

Cleanup owns:
- verified local staging cleanup after Source Intake evidence exists
```

Source Intake should not ask the user for a separate unrelated limit/batch size in this exact-selection workflow.

## Scope

Scope includes:

1. Implement an internal Source Intake entry point that accepts a durable iCloud acquisition `batch_id` or equivalent batch reference.

2. Use the 12.62.16 handoff manifest as the authoritative intake inventory.

3. Source Intake should process only resources marked ready for that acquisition batch.

4. Exclude failed, deferred, incomplete, missing, conflicting, partial, or non-ready resources.

5. Verify each ready resource still exists in staging before Source Intake processes it.

6. Verify each ready resource still matches durable metadata, including local SHA-256 where available.

7. Preserve existing Source Intake behavior for local/external/current iCloud fixed-window paths.

8. Preserve existing Vault, Asset, provenance, exact duplicate, and failure semantics.

9. Link Source Intake results back to acquisition batch/item/resource rows.

10. Update acquisition batch/item/resource status based on Source Intake results.

11. Produce clear batch-level reporting.

12. Define or implement cleanup readiness after Source Intake verifies the batch.

13. Add tests for successful intake, skipped/known files, failures, missing files, SHA mismatch, partial files, and wrong-batch/wrong-profile isolation.

14. Keep the exact-selection path internal and non-UI.

## Out of Scope

Do not implement full one-click iCloud ingestion.

Do not implement full multi-batch orchestration.

Do not implement total-limit orchestration.

Do not automatically start Source Intake immediately after acquisition unless explicitly approved.

Do not automatically run cleanup after Source Intake.

Do not expose exact-selection acquisition or batch Source Intake in the normal UI yet.

Do not simplify the Ingestion UI.

Do not change existing `standard` or `list_first_non_repeat` behavior.

Do not bypass Source Intake.

Do not write acquisition results directly to Vault, Asset rows, or provenance.

Do not delete iCloud cloud-library data.

Do not weaken guarded cleanup.

Do not store Apple credentials, cookies, tokens, sessions, 2FA codes, or raw download URLs.

Do not persist raw remote iCloud IDs.

## Required Behavior

### 1. Batch handoff validation

Before Source Intake starts from a durable acquisition batch, validate:

```text
batch exists
batch belongs to selected/active iCloud Source Profile
batch is marked batch_ready_for_source_intake
run is internal exact-selection mode
run/batch are not already consumed unless retry/reconciliation allows it
normal staging root still matches Source Profile
files are inside the selected managed staging root
no .partial files are selected
no failed/incomplete resources are selected
no conflicting resources are selected
```

If validation fails, stop with a clear reason and next safe action.

### 2. Resource-level verification

For each ready resource, verify:

```text
file exists
path is contained inside selected staging root
path matches expected source-relative path
file size matches expected size
local SHA-256 matches durable resource row, when present
resource status is ready/published/verified
resource is not failed/deferred/incomplete
```

If a resource fails verification, do not process it blindly.

Classify the failure, update durable status, and report the next safe action.

### 3. Source Intake processing

Source Intake should process the batch inventory as its input set.

Expected behavior:

```text
ready acquisition batch resources
→ Source Intake stages/processes those files
→ existing exact duplicate/Vault duplicate logic applies
→ existing provenance logic applies
→ batch/resource rows are updated with intake results
```

Important:

- Source Intake remains the ingestion authority.

- Source Intake must create or update provenance as it does today.

- Exact duplicates should still create/confirm proper provenance for the selected Source Profile where appropriate.

- Source Intake should not infer known-state from another source profile without recording this source profile’s evidence.

- The durable acquisition batch should receive evidence of what happened during Source Intake.

### 4. Batch/item/resource status after Source Intake

Add or use statuses sufficient to distinguish:

```text
ready_for_source_intake
intake_running
intake_completed
intake_completed_with_errors
intake_failed
intake_blocked
resource_intake_processed
resource_intake_skipped_known
resource_intake_duplicate_linked
resource_intake_failed
resource_intake_deferred
resource_missing_before_intake
resource_sha_mismatch_before_intake
```

Names may differ if repo conventions suggest better names, but the result must be clear.

### 5. Reporting

Source Intake report or acquisition batch report should show:

```text
acquisition run id
acquisition batch id
source profile id
logical items ready for intake
resources ready for intake
resources processed by Source Intake
new unique assets created
exact duplicates / Vault duplicates linked
already-known files skipped
failures
deferred files
missing files
SHA mismatches
batch status after Source Intake
next safe action
```

Reports should use counts and bounded samples, not huge lists.

### 6. Cleanup readiness

After Source Intake verifies the batch, the system should be able to determine whether the batch is ready for guarded cleanup.

This milestone does not need to automatically run cleanup.

Preferred behavior:

```text
Source Intake batch complete
→ batch marked ready_for_cleanup_dry_run or equivalent
→ cleanup dry run can use existing guarded cleanup path
```

If batch-specific cleanup readiness cannot be safely implemented yet, document the exact gap and keep cleanup source-wide but guarded.

The closeout must explain whether cleanup is now batch-aware or still source-wide guarded cleanup.

### 7. Retry/reconciliation

If Source Intake fails or is interrupted, retry must not blindly reprocess unknown state.

Before retrying batch Source Intake:

```text
reconcile batch/resource statuses
verify files still exist
verify local SHA-256 where available
detect already processed resources
detect missing/conflicting files
resume only safe work
```

If safe retry is not implemented, provide a clear blocked state and next safe action.

### 8. Existing paths must remain unchanged

Current paths must keep working:

```text
standard iCloud acquisition
list_first_non_repeat fixed-window acquisition
current guided Source Intake
local Source Profile intake
external Source Profile intake
guarded cleanup
```

The new exact-selection batch handoff must not alter normal UI workflows.

## Safety-Critical Rules

Preserve these rules:

- Acquisition writes only to selected Source Profile managed staging.

- Acquisition must not write directly to Vault.

- Acquisition must not create Asset rows.

- Acquisition must not create provenance rows.

- Source Intake remains the only path into Drop Zone, Vault, Asset rows, and provenance.

- Cleanup remains guarded by the 12.62.11B contract.

- No iCloud cloud deletion.

- No Vault deletion.

- No DB/provenance/source deletion.

- No Source Profile deletion.

- No credential/session storage in Photo Organizer.

- No secret logging.

- No raw remote iCloud IDs in DB/provenance/reports/UI.

- Ambiguous identity blocks.

- Wrong-profile known evidence must not suppress selected-profile ingestion.

- Existing standard/list-first modes must remain unchanged.
