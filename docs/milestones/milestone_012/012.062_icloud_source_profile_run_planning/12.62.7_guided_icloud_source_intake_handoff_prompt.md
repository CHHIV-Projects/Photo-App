# Product Direction

The iCloud workflow should remain step-based:

```text
Step 1 — Acquire from iCloud
Step 2 — Run Source Intake for staged iCloud files
Step 3 — Review acquisition + intake results
Step 4 — Cleanup staging manually later
```

12.62.7 implements **Step 2 only**.

After this milestone, the user should be able to:

```text
1. Open Ingestion.
2. Open an iCloud Source Profile.
3. Acquire from iCloud.
4. After acquisition completes, click Run Source Intake.
5. Confirm Total Limit and Batch Size.
6. Run Source Intake against the iCloud staging folder.
7. View Source Intake status and terminal summary.
```

Do not automatically run Source Intake after acquisition.

```text

- backend readiness has no blocking path/source-registration issues
```

```text
- readiness_status = not_ready
- blocking_reasons exist

- approved root blocked
```

# Source Intake Options

Use the same Source Intake run options established for local/external profiles.

## Total Limit

Field:

```text
Total Limit
[        ]
```

Behavior:

```text
blank/null = no total limit for this Source Intake run
positive integer = maximum eligible unknown files selected for this run
```

Suggested default:

```text
If latest acquisition file_inventory_count or staged inventory count is available:
  prefill Total Limit with that value, capped if current Admin behavior caps it.

If unavailable:
  leave blank / unlimited.
```

Do not hard-code blindly.

Helper text:

```text
Total Limit controls the maximum number of eligible unknown staged files selected for this intake run. Leave blank for no limit.
```

## Batch Size

Field:

```text
Batch Size
[500]
```

Behavior:

```text
positive integer required
default = 500
```

Helper text:

```text
Batch Size controls how many files are staged/processed per ingestion batch.
```

These are per-run options only.

Do not store them on Source Profile.

# Answers to Coder Questions — Milestone 12.62.7

## 1. When should the Source Intake handoff be visible?

The handoff should be visible in the iCloud Details drawer whenever the profile is an iCloud Source Profile, but its enabled/disabled state should depend on readiness and staged-file context.

Preferred behavior:

```text
Show:
  Run Source Intake for Staged iCloud Files

Enable when:
  - iCloud profile is active
  - readiness has no blocking path/source-registration issues
  - no active operation conflicts exist
  - staging path is valid/aligned/safe
  - source id / ingestion_source_id is valid
  - acquisition status is completed OR there is reasonable staged-file evidence

Allow with warning when:
  - latest acquisition is stopped but staged files may exist
  - no recent acquisition is found but staging path is valid
  - staged inventory count is unavailable but Source Intake can scan the staging folder
```

Do not require acquisition to have completed in the same UI session.

Reason:

```text
A user may have acquired from Admin, a prior session, or may have stopped acquisition after some files were staged.
The Source Intake step should remain manual and informed, not artificially tied to one current-session acquisition event.
```

If acquisition is currently running, Source Intake should be disabled because guardrails should block overlapping operations.

---

## 2. Total Limit default: acquisition file_inventory_count vs staged inventory count

Use staged inventory count as the primary source if available.

Priority order:

```text
1. Staged inventory count / current staging folder inventory, if available
2. Latest acquisition file_inventory_count, if available
3. Latest acquisition recent_count, if useful and no better count exists
4. Blank / unlimited
```

Reason:

```text
Source Intake operates on the staging folder.
The best default is the current staged inventory, not merely the acquisition request size.
```

If the count comes from acquisition summary rather than live staging inventory, label it clearly:

```text
Suggested from latest acquisition inventory.
```

Do not hard-code a limit if there is no reliable count.

---

## 3. If staged inventory count is unavailable, can user proceed?

Yes, allow proceeding if path/readiness is otherwise valid, but show a warning.

Warning text:

```text
Staged inventory count is unavailable. Source Intake will scan the staging folder and skip files already known for this source.
```

Default Total Limit in that case:

```text
blank / unlimited
```

Reason:

```text
The Source Intake pipeline already scans the source path and skips known files.
A missing count should not block a valid manual handoff.
```

But still block if readiness has real blockers:

```text
path mismatch
source registration mismatch
approved root blocked
active operation conflict
missing source id
profile inactive
```

---

## 4. Where should the handoff appear?

Keep it inside the iCloud readiness/acquisition area in the Details drawer.

Do not add a higher-level drawer chip or table-level summary in this milestone.

Reason:

```text
This is still a guided workflow with important context.
The user should review acquisition/readiness/staging context before starting Source Intake.
```

A higher-level summary chip can be deferred until after end-to-end testing.
