## 12.62.10 Blocker Triage Addendum (2026-06-18)

Observed browser evidence during the blocker:

- Console showed repeated fetch failures for `http://127.0.0.1:8001/api/admin/source-profiles?status=active` and related source-profile calls.
- Browser surfaced these as CORS-style errors (`No 'Access-Control-Allow-Origin' header ...`) from origin `http://127.0.0.1:3000`.

Investigation outcome:

- Backend CORS configuration already allows localhost and 127.0.0.1 origins/ports.
- The failure pattern is consistent with intermittent backend read-path failure/stall rather than a static CORS policy misconfiguration.
- Source profile listing route used a runtime schema-ensure call on hot read path, which can intermittently degrade/fail under validation activity and appear in browser as fetch/CORS-style failures.

Mitigation applied:

- Removed runtime schema-ensure calls from hot read functions in source profile/source listing service paths.
- Read routes now rely on startup schema synchronization already performed in app startup lifecycle.

Files updated for mitigation:

- `backend/app/services/admin/source_intake_service.py`

Expected validation impact:

- Reduced chance of no-op Acquire behavior caused by pre-modal/source-profile fetch failure.
- Reduced false CORS symptom presentation during local 127.0.0.1 E2E validation.

Follow-up hardening (same-day):

- Added transient fetch retry for the Acquire pre-modal `include_username` source-profile detail call.
- Surfaced Acquire click errors directly in the Acquire card so operator-visible feedback appears at the point of action (instead of only lower in the detail view).
- Updated Acquire UX to always open the confirmation modal on click, then load username in-modal (with retry) so action no longer appears as a silent no-op.
- Added manual Apple ID username input in the modal for fallback when username prefetch is unavailable.
- Clarified operator expectation: password/2FA are handled externally by icloudpd and are not entered/stored in the Photo Organizer UI.

## iCloud Authentication Finding

During E2E validation, `AUTH_REQUIRED` persisted and manual 2FA did not trigger correctly with the existing project-local `icloudpd` installation.

Resolution:

- Updated project-local `icloudpd` to the latest available version.
- After update, 2FA completed successfully.

Operational implication:

- iCloud acquisition reliability depends on the bundled/project-local `icloudpd` version.
- Future production readiness should include an `icloudpd --version` display/check in diagnostics or documentation.

# Answers — 12.62.10 Quick Bug Fix Questions

## 1. Profile-matched acquisition matching priority

Yes. Prefer `source_id` when present.

Recommended matching precedence:

```text
1. source_id / ingestion_source_id match, if acquisition/status/report payload includes it
2. managed_staging_path / expected_acquisition_path match
3. canonical/sanitized iCloud source slug match
4. normalized display label match
5. global latest acquisition fallback
```

Reason:

```text
source_id is the least ambiguous identity.
Path is the next safest because acquisition operates against a concrete staging/export path.
Slug/label matching is useful but has already proven fragile with punctuation and sanitization differences.
```

If the payload does not currently include `source_id`, do not expand the backend unless needed for this quick fix. Use path/slug matching.

---

## 2. If status payload references selected profile’s staging path, should path match override label mismatch?

Yes.

If the acquisition status/report clearly points to the selected profile’s managed staging path or expected acquisition path, treat it as profile-matched even if the label/slug comparison is imperfect.

Preferred rule:

```text
Path match beats label mismatch.
```

Use normalized path comparison, not raw string comparison.

Reason:

```text
The acquisition target is the staging path. If the report/status path matches the selected profile’s managed staging path, the operator should see it as this profile’s acquisition.
```

Still document the label mismatch in code comments or closeout if found.

---

## 3. Should “latest global acquisition” wording be suppressed when profile match is found?

Yes.

If a profile match is found by any accepted matching method:

```text
source_id
path
slug
normalized label
```

then suppress the “latest global acquisition” wording.

Show profile-specific wording instead:

```text
Latest acquisition for this profile completed.
```

or:

```text
Acquisition completed for this iCloud Source Profile.
```

Only show “latest global acquisition” when the system is truly falling back to a global status that is not matched to the selected profile.

---

## Final Lock-in

For this quick fix, implement deterministic acquisition matching with this priority:

```text
1. source_id if available
2. managed/expected staging path
3. sanitized source slug
4. normalized source label
5. latest global fallback
```

Behavior:

```text
- If matched, the iCloud Workflow Summary should treat it as this profile’s acquisition.
- If matched, do not show “different source profile” or “latest global acquisition” wording.
- If unmatched, global fallback wording is acceptable, but should be clear.
```

Keep this narrow. Do not implement a broader backend redesign unless the frontend cannot reliably distinguish profile-specific versus global acquisition status with available fields.
