## Core design decision

12.62.27 inventory is a **selection ledger**, not a complete download manifest.

Therefore:

```text
Do not construct helper download requests from inventory rows alone.
Do not assume inventory rows contain enough per-resource data for download.
Do re-resolve inventory rows against a fresh helper listing.
Do build preview selected-item/resource counts from fresh helper listing metadata.
```

## Inventory row selection rules

Select only inventory rows where:

```text
source_profile_id matches requested source
eligibility_state = "eligible_metadata_only"
known_state = "pending_known_state_check" or another not-yet-known state
backfill_completed is not true, if such field already exists
acquired is not true, if existing 12.62.27 schema has that field
remote_identity_basis is populated
remote_identity is populated
```

Skip and count rows that are:

```text
unsupported
ambiguous
pending_classification
already known
already completed/acquired
missing required identity fields
```

Use deterministic ordering:

```text
created_remote_at ASC NULLS LAST
added_remote_at ASC NULLS LAST
first_observed_at ASC
id ASC
```

Do **not** use `observed_remote_position` as a cursor.

It is only a scan-local ordinal.

## Fresh helper listing re-resolution

Use the existing helper/list-candidates machinery.

The preview service should:

```text
1. Select bounded eligible inventory rows.
2. Run a fresh metadata listing using existing helper listing path.
3. Build an in-memory lookup by helper item identity.
4. Match inventory.remote_identity to fresh listing item.item_id.
5. For matched rows, build preview logical-item/resource data from the fresh listing metadata.
6. For missing rows, count them as stale/retryable for this run.
```

Remote identity semantics:

```text
remote_identity = helper item_id
remote_identity_basis = "helper_item_id_observed_stable"
```

Do not claim CloudKit/iCloud permanent stability.

Rows missing from fresh helper listing should be skipped with:

```text
skip_reason = "skipped_stale_listing"
```

or equivalent, but should remain retryable and should **not** be marked completed.

## Live Photo behavior

Live Photo inventory rows count as **one logical asset**.

When fresh helper listing exposes paired/multiple resources, preview should report:

```text
preview_selected_logical_count = 1
preview_selected_resource_count >= 2
```

Do not allow partial Live Photo manifest preview to be treated as acquisition-ready.

If resource information is incomplete, count it as unsafe/ambiguous for execution preview, but do not block safe unrelated rows.

## Known-state strategy

Do not invent fuzzy duplicate logic.

For 12.62.28.2:

```text
- do not mark rows as newly acquired
- do not mark rows as backfill_completed
- do not infer known_state from media type
- do not use filename-only matching
- do not bypass Source Intake
```

If existing strong known-state logic can safely identify already-known rows, preview may count them as skipped known. Otherwise leave uncertain rows as pending and let future Source Intake decide.
