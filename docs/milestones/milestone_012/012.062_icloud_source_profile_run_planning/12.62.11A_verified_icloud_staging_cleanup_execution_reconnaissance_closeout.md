## 2. Current Operational Behavior

### Backend readiness and start behavior

There is no dedicated backend "cleanup readiness" response. Current readiness is assembled from two layers:

1. `get_icloud_source_readiness()` in `backend/app/services/admin/icloud_readiness_service.py` validates the general iCloud Source Profile and reports profile/path/registration/auth/operation state.
2. `icloudCleanupDryRunViewModel` in `frontend/src/components/IngestionView.tsx` combines that general readiness with cached acquisition, Source Intake, and cleanup status to decide whether the dry-run button should be enabled.

The cleanup API is implemented in `backend/app/api/admin.py`:

- `GET /api/admin/icloud-staging-cleanup/status?source_id=<id>` returns the latest cleanup run for that source.
- `POST /api/admin/icloud-staging-cleanup/run` accepts `{ source_id, dry_run }` and starts a background run.

The POST route first calls `get_ingestion_operation_guardrail_snapshot()`. It blocks when any acquisition, Source Intake, or cleanup run is active. It then calls `start_cleanup_run()`.

`start_cleanup_run()` in `backend/app/services/admin/icloud_staging_cleanup_execution_service.py`:

- blocks another active cleanup globally
- independently blocks active Source Intake for the selected source
- requires an existing `IngestionSource`
- requires `source_type == "cloud_export"`
- resolves `source.source_root_path`
- requires that resolved root to be below `storage/exports/icloud`
- persists an `IcloudStagingCleanupRun` row and launches a daemon thread

### Candidate evaluation and classification

`_run_cleanup_background()` scans all files recursively below the resolved source root. It builds:

- a provenance map keyed by selected `source_id` plus normalized `source_relative_path`
- an Asset map keyed by provenance `asset_sha256`
- negative evidence from all readable Source Intake JSON reports matching the selected source ID

A file is currently eligible when:

1. the file was found under the selected source root
2. its resolved path remains under that root
3. at least one Source Intake report exists for the selected source
4. matching provenance exists for selected source ID and relative path
5. provenance has an asset SHA-256
6. the linked Asset row exists
7. `asset.vault_path` is non-empty and exists on disk
8. the relative path does not occur in collected failure/deferred evidence

Current skip/protection reasons are:

- `source_not_under_icloud_exports_root`
- `status_evidence_missing`
- `failed_or_deferred_evidence`
- `no_provenance`
- `asset_missing`
- `vault_missing`
- `conflicting_status_evidence`
- `file_missing`

The service does not expose separate per-file `protected`, `failed`, or `unknown` result classes. All non-eligible per-file outcomes are represented as skipped reasons. Run-level exceptions set the run to `failed`.

### Dry run and live execution

Dry run and deletion already share the same backend loop. With `dry_run=true`, eligible files are reported but not deleted. With `dry_run=false`, the current service calls `Path.unlink()` immediately after the eligibility checks, then removes empty descendant directories on a best-effort basis while retaining the source root.

The existing API does not require:

- a completed dry-run run ID
- a recent dry run
- a candidate-set match with a prior dry run
- an explicit confirmation field or confirmation phrase

Therefore any caller able to invoke the Admin endpoint can currently submit `dry_run=false` directly.

### Frontend behavior

`frontend/src/components/AdminView.tsx` is the historical Milestone 12.44.1 surface. It exposes both `Preview Cleanup` and `Execute Cleanup`. `Execute Cleanup` calls the existing endpoint with `dry_run:false` directly. There is no confirmation modal and no binding to a completed dry run.

`frontend/src/components/IngestionView.tsx` is the newer guided surface. It:

- loads cleanup status scoped to the selected Source Profile
- shows cleanup readiness and a dry-run confirmation modal
- calls only `runIcloudStagingCleanupDryRun(source_id)`
- explicitly tells the operator that the Ingestion tab will not delete files
- shows eligible, skipped/protected, deleted, byte, report, and skip-reason results

The guided Ingestion surface does not expose live deletion. Its active-status set omits `pending`, although the backend treats `pending` as active. This can briefly impair polling/state accuracy immediately after a run starts and should be corrected in 12.62.11B.

## 4. Existing API / Data Model Findings

### Current API contract

The current request schema is:

```text
source_id: int
dry_run: bool = true
```

The same endpoint launches preview or deletion. There is no server-enforced confirmation or preview token/run reference.

Status may be scoped by `source_id`, and `get_cleanup_status()` correctly queries the latest cleanup row for that source when supplied. The guided UI also discards a non-idle status whose `source_id` does not match the open profile.

### Current cleanup-run model

`IcloudStagingCleanupRun` persists:

- source ID/label/root snapshot
- `dry_run`
- status/timestamps/elapsed time
- eligible/deleted/skipped counts and bytes
- skip-reason counts/samples
- report path and run error
- creator and creation time

It does not persist:

- parent dry-run ID for an execution
- dry-run expiry/freshness
- explicit-confirmation evidence
- candidate manifest or manifest fingerprint
- per-file verification/deletion outcomes
- source profile provider/acquisition method/managed path snapshot

Adding a safe preview-to-execution binding will require a schema change or a new persistence structure. The project currently uses an idempotent schema-ensure service for this table; 12.62.11B should explicitly choose whether to extend that pattern or introduce a formal migration.

### Current report format

Reports are written to `storage/logs/icloud_cleanup_reports/icloud_cleanup_<timestamp>_run<id>.json` and include:

- report type and timestamps
- run/source identity and source root
- `dry_run`
- eligible/deleted/skipped counts and bytes
- skip-reason counts and samples
- full relative-path lists for eligible/deleted/skipped files
- a 100-item preview section

Missing for verified execution are:

- the authorizing dry-run run ID
- dry-run creation/expiry information
- candidate manifest fingerprint
- explicit-confirmation state
- positive verification details per candidate
- first-class delete failures and error details
- final outcome such as completed, partially completed, or failed

Reports are operational artifacts, not the system of record. Execution authorization should not depend solely on reading a mutable JSON report.

### Source Intake report evidence limitations

Cleanup scans all readable Source Intake reports for the source, but:

- `has_report_evidence` means only that any matching report was found
- deferred paths come from `deferred_unready_sample`, which is limited to 20 items, not the complete deferred set
- negative evidence is aggregated across history without resolving whether a later successful provenance record supersedes an older failure/defer event

Provenance + Asset + Vault evidence is therefore the stronger durable proof. Report evidence is useful conservatively, but it is not a complete per-file success/failure ledger.

## 5. Existing Safety Boundaries

The current implementation already provides several useful foundations:

- source targeting is by numeric source ID
- the selected source root must resolve below the project iCloud exports root
- each scanned file is resolved and checked against the selected source root
- provenance lookup includes selected source ID and exact normalized relative path
- both slash variants are queried for Windows/POSIX historical compatibility
- linked Asset and on-disk Vault path existence are required
- known failure/deferred conflicts are skipped
- dry run and live execution use a background run record and durable JSON report
- empty-directory removal is descendant-only and retains the source root
- the API-level guardrail blocks active acquisition, Source Intake, or cleanup
- no cleanup code calls an iCloud/cloud deletion API
- no DB, provenance, Source Profile, source registry, Drop Zone, or Vault deletion is present in the cleanup service
- startup marks interrupted cleanup runs failed
- the guided Ingestion UI is currently dry-run-only and source-scopes displayed status

These path, provenance, status, and reporting helpers are suitable building blocks for 12.62.11B.

## 6. Safety Gaps / Risks Identified

### Must resolve before guided execution

1. **Direct live execution is already reachable.** The API accepts `dry_run=false`, and the legacy Admin UI invokes it without a confirmation modal or completed dry-run requirement.

2. **No dry-run-to-execution binding exists.** An execution does not reference, validate, or match a prior dry run. A stale preview provides no server-side authorization or candidate constraint.

3. **Source validation is not iCloud-profile-complete.** The cleanup service checks only `source_type == cloud_export` and source-root containment. It does not require:
   - `cloud_provider == icloud`
   - active profile status
   - `acquisition_method == icloudpd` where applicable
   - managed staging path presence
   - equality among source root, managed staging path, and canonical expected iCloud path
   - general readiness/source-registration alignment

4. **Staged content is not reidentified.** Eligibility is based on relative path and historical provenance. The staged file is not hashed and compared with `provenance.asset_sha256`. A different file replacing an ingested file at the same relative path could be deleted.

5. **Vault evidence is existence-only.** The service does not verify that the Vault path is inside the configured Vault root, is a regular file, or hashes to the Asset SHA-256. A missing/corrupt/replaced archival copy could still satisfy the current existence check.

6. **Candidate changes are not detected.** Files added, removed, renamed, or modified between dry run and execution are not compared with the previewed set.

7. **Cross-operation start has a check/start race.** The API reads guardrail state before creating a run row. Concurrent requests can both pass the query before either commits. `start_cleanup_run()` rechecks active cleanup and same-source intake but does not independently recheck acquisition or intake for another source. There is no DB uniqueness/advisory lock or shared atomic coordinator covering all three operation types.

8. **Per-file errors are overgeneralized.** Any unexpected per-file exception becomes `status_evidence_missing`. There is no explicit `delete_failed` or `verification_failed` result, and useful exception context is lost.

9. **Partial execution is not represented clearly.** A run may delete some files, skip later failures, and still be marked `completed`. Conversely, a report-writing or outer exception after deletions can mark the run failed without a complete durable per-file report.

10. **Report creation occurs after deletion.** If the process crashes or report writing fails after one or more unlinks, the DB retains only eventual counts if finalization succeeds; it does not durably record each completed deletion as it happens.

11. **The current eligible list can overstate successful live handling.** A file is appended to `eligible` before `unlink`. Non-`FileNotFoundError` deletion exceptions do not remove it from eligible and are reported under a generic skip reason.

12. **Guided UI polling omits `pending`.** Backend `RUNNING_STATUSES` includes `pending`; `IngestionView` active statuses do not.

13. **Cleanup readiness is coupled to acquisition readiness.** The guided dry-run button reuses Source Intake/acquisition readiness, including authentication-related state that is not inherently needed for local cleanup. A dedicated cleanup readiness contract would be clearer and safer.

### Lower-priority observations

- Empty directory cleanup intentionally swallows errors and may leave folders behind. This is acceptable if reported as best-effort behavior.
- Cleanup has no stop endpoint even though `stop_requested` appears in shared status sets. Adding cancellation is not required for 12.62.11B and could complicate partial-delete semantics.

## 8. Recommended Implementation Plan for 12.62.11B

### Must-have safety requirements

1. **Create a dedicated cleanup readiness/preview contract.** Backend truth should validate that the selected source is an active iCloud Source Profile and that source root, managed staging path, and canonical expected staging path all resolve to the same approved directory. Local cleanup should not require a healthy Apple authentication session.

2. **Split planning from execution.** Extract a pure candidate-evaluation function used identically by dry run and execution. It should return structured eligible/protected/verification-failed results without deleting.

3. **Strengthen positive verification.** For every eligible candidate, require:
   - existing regular staged file
   - resolved containment below selected canonical managed staging root
   - exact provenance for selected source ID and relative path
   - staged-file SHA-256 equal to provenance/Asset SHA-256
   - linked Asset row
   - existing regular Vault file below the configured Vault root
   - Vault-file SHA-256 equal to Asset SHA-256
   - no unresolved protection/conflict evidence

4. **Persist a preview binding in DB state.** A completed dry run should store at least:
   - source/profile ID and canonical root snapshot
   - generation and expiry time
   - eligible count/bytes
   - deterministic fingerprint of the sorted candidate manifest
   - sufficient version information to invalidate old planner semantics

   The fingerprint should include relative path, staged content hash, Asset hash, size, and relevant Vault identity. Full human-readable candidates can remain in the report, but the DB fingerprint—not the report alone—should authorize execution.

5. **Use a separate execution request/endpoint or a strictly discriminated request.** Recommended request:

   ```text
   source_id
   dry_run_run_id
   explicit_confirmation
   ```

   The backend must reject execution unless the referenced run is completed, is a dry run, belongs to the same source/root, is within an owner-approved freshness interval, has eligible candidates, and has the expected confirmation value. Frontend confirmation is necessary but not sufficient.

6. **Recompute immediately before deleting and require exact match.** Re-evaluate all candidates, recompute the manifest fingerprint, and compare it with the authorized dry run. If anything changed, delete nothing and require a new dry run. This is safer than deleting the intersection because it keeps the operator preview exact.

7. **Recheck each file immediately before unlink.** Revalidate containment, regular-file status, size/stat identity, and content hash before deletion. If a candidate changes after planning, skip/fail that file rather than guessing.

8. **Make operation start atomic.** Use a shared process/DB-backed operation lock or PostgreSQL advisory-lock pattern so acquisition, Source Intake, and cleanup cannot pass guardrails concurrently. Recheck all conflicts within the protected start section. A frontend disabled state is not a concurrency control.

9. **Improve durable auditing and partial failure handling.** Create the execution report/audit artifact before the first deletion, then record/finalize per-file outcomes safely. Classify at least:
   - `deleted`
   - `protected` / `skipped`
   - `verification_failed`
   - `delete_failed`
   - `file_missing`

   A run with some deletions and some failures should report an explicit partial/completed-with-errors outcome rather than plain success. Preserve successfully deleted results if later files fail.

10. **Close the legacy bypass.** `AdminView` must not retain a direct `dry_run=false` button that bypasses the new contract. It may be removed, disabled with guidance to use Ingestion, or routed through the same preview/confirmation flow.

11. **Add guided execution UI without broader redesign.** In `IngestionView`:
   - show Execute only for a fresh, completed dry run for the selected profile
   - display source, canonical local staging path, eligible count/bytes, skipped/protected count, dry-run time/expiry, and report
   - state clearly that only local staging copies are removed; iCloud, Vault, DB, and provenance are untouched
   - require an explicit checkbox plus typed confirmation or equally unambiguous confirmation control
   - include `pending` in active polling states
   - refresh readiness/status/summary after execution

12. **Keep cleanup manual.** Do not chain execution automatically after Source Intake and do not add cloud deletion, scheduling, reacquisition, or broader Ingestion UX changes.

### Product-owner lock-ins needed for 12.62.11B

Recommended defaults:

- dry-run freshness: 10 minutes
- candidate mismatch: fail closed and require a new dry run
- confirmation: checkbox plus exact typed phrase such as `DELETE LOCAL STAGING COPIES`
- partial outcome: `completed_with_errors` when at least one deletion succeeds and at least one candidate fails
- legacy Admin execution: remove or disable, directing execution to the selected Source Profile in Ingestion
- empty child directories: best effort; retain source root

### Nice-to-have refinements to defer

- consolidation of all acquisition/intake/cleanup tiles
- binary readiness redesign across the whole Ingestion tab
- report browser UI
- cleanup stop/cancel support
- generalized operational history
- automatic post-intake job orchestration

- confirmation that cleaned files are not unnecessarily redownloaded in non-repeat mode
- caught-up/incomplete/unknown completeness reporting

## 9. Recommended Tests / Validation

### Candidate planner/service tests

- rejects missing source and non-iCloud/non-active profiles
- rejects source root outside approved iCloud exports root
- rejects mismatch among source root, managed staging path, and canonical expected path
- accepts nested icloudpd date folders
- rejects file/symlink escape outside source root
- requires provenance for selected source ID plus normalized relative path
- handles Windows and POSIX path separators
- requires linked Asset
- requires staged hash to match provenance/Asset SHA-256
- requires Vault path below configured Vault root
- requires Vault regular file and matching SHA-256
- conservatively classifies failure/deferred/conflicting evidence
- proves dry run never unlinks or removes directories
- produces deterministic manifest fingerprint independent of scan order

### Execution authorization tests

- rejects direct/legacy `dry_run=false` without confirmation contract
- rejects missing, failed, live, stale, wrong-source, wrong-root, or already-consumed dry-run ID
- rejects incorrect or missing explicit confirmation
- rejects candidate fingerprint changes caused by add/remove/rename/replace
- deletes nothing when preview validation fails
- rechecks content before each unlink
- allows only one execution consumption of a dry-run authorization

### Failure/reporting tests

- records successful deletion
- preserves and reports a protected file
- classifies permission/locking error as `delete_failed`
- reports partial completion accurately
- retains source root and removes only empty descendants
- safely handles file disappearance during execution
- refuses to begin deletion when initial report/audit creation fails
- preserves durable deletion results if final report update fails
- repeat execution/cleanup is idempotent

### Guardrail/API tests

- active acquisition blocks cleanup
- active Source Intake for same or other source blocks cleanup
- active cleanup blocks acquisition, intake, and another cleanup
- concurrent starts cannot both create active operations
- status remains scoped to selected source
- API response includes authorizing dry-run ID/freshness/fingerprint status and structured result counts

### Frontend tests/checks

- local/external profiles never show iCloud cleanup execution
- execution remains hidden/disabled before a valid dry run
- stale or mismatched dry run forces another preview
- modal shows the exact selected source/path/counts/bytes
- confirmation control gates the request
- `pending` is polled as active
- partial/failure outcomes are distinguishable from success
- frontend build/type check passes

No real iCloud account is required for automated tests. Use temporary staging/Vault directories and temporary test files. Real deletion validation should occur only later against a deliberately bounded disposable staging fixture, followed by the explicit 12.62.12 operator loop.
