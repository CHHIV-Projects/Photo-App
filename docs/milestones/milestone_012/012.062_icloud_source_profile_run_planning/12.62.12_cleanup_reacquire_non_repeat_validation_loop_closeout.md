### Durable known-state evidence

Preflight candidates are identified by normalized source-relative resource path. Known-state is scoped by:

```text
Provenance.ingestion_source_id
+ Provenance.source_relative_path
```

This source-specific provenance survives local staging cleanup.

Classification is:

- `staged_known`: the resource exists in local staging; this alone is not durably known
- `ingested_known`: exact source-scoped provenance exists
- `vault_verified_known`: provenance links to an Asset whose Vault path exists
- `unknown`: no exact source-scoped provenance match

Each resource path is evaluated independently. A known HEIC resource does not infer that its `_HEVC.MOV`, AAE sidecar, or another sibling resource is known.

### Download decision

The current design is conservative and window-level:

- if every preflight candidate is durably known and no candidate identity is unknown, the download subprocess is skipped
- if any candidate is not known, acquisition proceeds for the complete requested recent window and reports `partial_window_only`

This distinction explained the manual E2E results.

The current and desired behaviors must not be conflated:

```text
Current fixed-window mode:
inspect the latest N iCloud candidates
→ skip the complete download only when all N candidates are known
→ otherwise download the complete recent window
→ stop after that window

Desired new-count mode:
scan iCloud candidates beyond the first window as necessary
→ skip known candidates individually
→ continue until up to N new/unknown items are downloaded
→ stop at N new items or a clear safe end/stopping condition
```

`recent_count` currently controls the number of recent iCloud assets considered, not the number of new assets to download. The normal user would still have to guess or increase the window to search farther back, which is not the intended v1 behavior.

## 8. Known Limitations

### Mixed known/new recent windows — blocker for desired new-count semantics

`list_first_non_repeat` is not a per-resource selective downloader. When a recent window contains both known and unknown candidates, it reports the identities correctly but runs acquisition for the complete recent window. Cleaned known resources can therefore be downloaded again alongside genuinely new candidates.

The first manual run demonstrated this behavior:

```text
13 known + 16 not known to this source profile
→ partial_window_only
→ complete 29-resource window downloaded
```

The successful final run validates the all-known cleanup/reacquire path, not selective behavior for a mixed ongoing window. Because normal ongoing acquisition is expected to contain a mixture of known and genuinely new items, this is a blocker for presenting the mode as true `Download new items only` behavior.

### Validation observation classification

| Observation | Classification | Meaning |
|---|---|---|
| Empty descendant subfolders after cleanup | Harmless cleanup artifact | All eligible files were deleted and the selected source root is intentionally retained; empty-directory removal is best effort on Windows. |
| Standard-mode repeat download after staging cleanup | Expected but inefficient | Standard mode depends on the requested recent window and local file presence; it does not use durable known-state to prevent reacquisition. |
| Mixed known/new list-first window redownloaded known resources | Blocker for desired new-count semantics | Current mode downloads the complete fixed window when any candidate is unknown. |
| Recent Count returned to `25` after view reload/remount | UX clarity issue | The modal initializes an upcoming-run input; it does not display the previous persisted run value. |
| All-known window produced zero downloads | Valid narrow validation | Proves cleanup-surviving profile-scoped known-state and the all-known short-circuit, but not true new-count acquisition. |

### Identity basis

Identity is exact source-relative resource path scoped to the ingestion source. Cloud-native iCloud asset IDs are not first-class provenance keys. Renames, path changes, account/profile remapping, or cloud identity ambiguity remain outside this milestone.

### Fixed-window completeness

`likely_caught_up` means the bounded recent window was entirely known. It does not prove the complete iCloud library is caught up. Large-library checkpoint/until-found behavior remains deferred.

### Reporting

- `skipped_existing_count` remains an `icloudpd` local-file count and is zero when the complete download subprocess is skipped. The authoritative non-repeat fields are `already_known_count` and `download_skipped_due_to_all_known` in the JSON report.
- Candidate detail is bounded to 50 report samples. Counts remain available for larger windows, but complete identity-by-identity UI review is not designed for large initial libraries.
- The guided UI shows report paths and aggregate acquisition counts but does not surface every known-state report field directly.

### Empty directories

Cleanup removes empty descendant directories best effort and intentionally retains the selected source root. The manual run left eight empty descendant directories on Windows while deleting all 29 eligible files. This did not affect non-repeat behavior or safety.

### UI status/profile behavior

No stale-status or different-profile-status defect interfered with validation. The workflow summary used the selected profile's evidence, while the global acquisition status area retained its existing different-profile warning behavior.

The acquisition confirmation modal's Recent Count field initializes to `25` when the Ingestion view is mounted and is not populated from the previous acquisition run. A page refresh or remount can therefore show `25` even when the prior persisted run used another value, such as `10`. The modal is an input for the upcoming run, not historical evidence. Persisted acquisition reports and terminal run status are authoritative for the mode and recent count actually submitted. This is a legitimate UX clarity limitation for the later Ingestion workflow simplification milestone.

## 9. Recommended Next Milestone

Create a dedicated Extra High-reasoning design/implementation milestone for true iCloud new-count acquisition before presenting a normal user-facing `Download new items only` action.

Required semantics:

```text
Requested new count = N
→ scan/list candidates beyond the first recent window as necessary
→ identify known resources from durable profile-scoped evidence
→ skip known resources individually
→ download up to N genuinely new/unknown items
→ stop at N new items or a clear bounded/end/checkpoint condition
```

The follow-up must define safe internal scan bounds, pagination/checkpoint behavior, asset-versus-resource counting for Live Photos and sidecars, deterministic stop reasons, mixed-window selective acquisition feasibility, and reporting for considered/known/new/downloaded counts. It must not rely only on cleaned local staging files or require the normal user to guess progressively larger recent windows.

Cloud workflow consolidation / Ingestion UX simplification should follow or explicitly account for this technical milestone. It should not label the current fixed-window all-known short-circuit as `Download new items only`.
