# Milestone 12.62.13 — iCloud New-Count Acquisition Semantics

## Goal

Implement or, if code/tooling reality blocks implementation, precisely define the smallest viable path for true iCloud “download up to N new items” acquisition.

The desired user-facing behavior is:

```text
Download up to N new resources/items for the selected iCloud Source Profile.
Skip resources already known for that Source Profile.
Continue looking farther back as needed until:
- N new resources/items are selected/downloaded,
- a configured scan/stop condition is reached,
- or there are no more candidates available.
```

This milestone addresses the blocker discovered in 12.62.12: current `list_first_non_repeat` behavior is fixed-window based and does not selectively download only unknown resources from a mixed known/new window.

## Background

12.62.12 validated that known-state identity is durable and profile-scoped:

```text
Provenance.ingestion_source_id
+ Provenance.source_relative_path
```

That evidence survives guarded local staging cleanup.

However, 12.62.12 also showed the current acquisition behavior is not true new-count acquisition.

Current behavior:

```text
Inspect latest N candidates.
If all are known, download nothing.
If any are unknown, download the entire N-candidate window.
Stop.
```

Desired behavior:

```text
Download up to N new/unknown resources for the selected Source Profile.
Known resources should not be redownloaded.
The system should continue scanning farther back as needed to fill the requested new count or reach a safe stopping condition.
```

The user’s intended future model is:

```text
Batch: 50
Limit: 1000
Mode: Download only new

Acquire 50 new
Source Intake those 50
Cleanup those 50
Acquire next 50 new
Source Intake next 50
Cleanup next 50
Continue until 1000 new items are acquired or no more new items are available.
```

This milestone should focus on the acquisition semantics needed for that future workflow. Do not implement the full repeated acquire/intake/cleanup orchestration yet.

## Safety-Critical Rules

Preserve these rules:

- Acquisition may only download files into the selected Source Profile’s managed iCloud staging path.

- Acquisition must never write directly to Vault.

- Acquisition must never write directly to DB asset records or provenance.

- Source Intake remains the only path into Drop Zone, Vault, DB asset records, and provenance.

- Cleanup remains guarded by the 12.62.11B contract.

- Cleanup must never delete iCloud cloud-library data.

- Cleanup must never call any iCloud/cloud delete API.

- No iCloud cloud data may be deleted.

- Vault files must not be deleted or modified.

- DB/provenance/source records must not be deleted.

- The app must not store Apple passwords, 2FA codes, session cookies, auth tokens, or secrets.

- Do not log secrets or expose credentials in reports.

- Do not suppress genuinely new resources because another profile or sibling resource is known.

## Requirements

### 1. Define acquisition count semantics clearly

Distinguish these concepts:

```text
candidate_scan_window:
  how many recent iCloud candidates are inspected

target_new_count:
  how many unknown/new resources the user wants to download

overall_limit:
  maximum new resources to acquire across a future multi-batch workflow

batch_size:
  number of new resources per future acquire/intake/cleanup cycle
```

For this milestone, implement or prepare for:

```text
target_new_count
```

Do not confuse it with fixed recent candidate window.

### 2. Preserve fixed-window behavior accurately

The existing fixed-window mode may remain available for diagnostics or advanced use, but it must not be labeled as true “Download new items only.”

Acceptable labels:

```text
List first / non-repeat
List first / non-repeat (fixed recent window)
Standard recent download
```

Do not use `Download new items only` unless the behavior truly attempts to download up to the requested number of new items.

### 3. Implement true new-count behavior if feasible

If feasible with existing tooling, add a new behavior that:

- lists candidates

- evaluates known-state per selected Source Profile

- skips known resources

- continues scanning farther back as needed

- selects up to the requested number of unknown/new resources

- downloads only those selected new resources

- reports selected, known/skipped, downloaded, and stopping reason

Stopping reasons should include, where applicable:

```text
target_new_count_reached
no_more_candidates
scan_limit_reached
tooling_limit_reached
auth_or_acquisition_error
```

### 4. Do not fake selective behavior

Do not claim success if the implementation still downloads the whole mixed window.

This does not satisfy the milestone:

```text
13 known + 16 unknown
→ download all 29
```

This does satisfy the milestone:

```text
13 known + 16 unknown
→ download only the 16 unknown, then continue scanning if target_new_count is not filled
```

Or, if the tooling cannot do that, report the blocker.

### 5. Reporting requirements

Reports should clearly distinguish:

- mode used

- target new count

- candidate scan count

- known/skipped count

- selected-new count

- downloaded count

- stopping reason

- whether the result is complete, partial, blocked, or tooling-limited

Reports should use counts and bounded samples, not giant UI lists.

### 6. Tests

Add or update tests for:

- mixed known/new candidate window does not redownload known resources in true new-count mode

- known-state is scoped by selected Source Profile

- wrong-profile known evidence does not suppress download

- Live Photo still and motion resources are independently evaluated

- unknown sidecar is not inferred known from sibling known resource

- requested target count continues past known candidates where fixture supports it

- all-known set downloads zero and reports caught up or target not fillable

- new mode does not alter existing standard mode

- existing fixed-window mode remains accurately labeled/reported

- reports distinguish fixed-window behavior from true new-count behavior

If selective download cannot be automated because of `icloudpd` limits, create tests proving the known-state planner can select the correct unknown resources, then document the tooling blocker.
