# Milestone 12.62.23 — Enable All Supported iCloud Assets in Single-Flow Ingestion

## Recommended Reasoning Level

Extra High.

This milestone touches iCloud exact-selection, logical asset/resource counting, Live Photo pairing, videos, Source Intake handoff, cleanup reconciliation, and the internal/admin single-flow UI/API added in 12.62.22.

The goal is not to redesign media architecture.

The goal is to remove the artificial ordinary-stills-only execution limitation where existing system behavior already supports broader asset ingestion.

## Goal

Enable the internal/admin single-flow iCloud run to ingest **all currently supported iCloud assets**, rather than treating media type as a normal ingestion-time choice.

Product direction:

```text
iCloud ingestion should get assets.
Photo Organizer should interpret asset type after ingestion.
```

In real use, the operator should not need to choose between stills, videos, or Live Photos. The normal iCloud ingestion concept should become:

```text
Get new iCloud assets
```

The system should preserve and report what those assets are:

```text
ordinary still
video
Live Photo pair
unsupported/blocked item
ambiguous item
```

But media type should not be the final user-facing ingestion decision.

## Terminology

Use these concepts clearly:

```text
logical asset:
  One user-visible iCloud item/photo/video/Live Photo.

physical resource:
  One downloaded file/resource belonging to a logical asset.

ordinary still:
  A logical asset with one still-image resource.

standalone video:
  A logical asset with one video resource.

Live Photo:
  A logical asset with paired resources:
    still image resource
    motion companion video resource
```

`batch_size` and `total_limit` should count logical assets, not physical resources.

Reports must make physical resource counts clear.

Example:

```text
batch_size: 3 logical assets

Selected:
  1 ordinary still = 1 resource
  1 standalone video = 1 resource
  1 Live Photo = 2 resources

logical_assets_selected: 3
resources_selected: 4
```

## Required Behavior

### 1. Ordinary stills

Existing ordinary-stills behavior must continue to work.

Do not regress 12.62.22 behavior.

### 2. Standalone videos

If the existing Source Intake path already supports standalone video ingestion, then the single-flow iCloud wrapper should allow standalone video assets to be selected, acquired, intaken, cleaned, and reported.

If standalone video selection/acquisition is not currently safe, stop and report the exact missing piece.

Do not pretend unsupported video behavior is successful.

### 3. Live Photos

If the existing system already supports Live Photo still+MOV pairing, the single-flow iCloud wrapper should preserve that pairing.

A Live Photo should be treated as one logical asset with paired physical resources.

Required safety behavior:

```text
Do not acquire only half of a Live Photo pair when the pair is expected.
Do not Source Intake only one half and clean the other blindly.
Do not count one Live Photo as two logical assets.
Do not clean up a companion resource unless it belongs to the current selected/acquired/intaken logical asset.
```

If Live Photo pairing cannot be proven from current adapter/source evidence, block with a clear stop reason such as:

```text
LIVE_PHOTO_PAIRING_NOT_PROVEN
```

or equivalent.

### 4. All supported assets

Enable the normal all-assets path only as the union of asset types proven safe in this milestone.

Preferred behavior:

```text
requested_media_scope: all_supported_media
effective_mode: all_supported_assets
```

or equivalent.

If only ordinary stills and videos are proven, but Live Photos are not, then `all_supported_media` should either:

```text
stop and report Live Photos not yet proven
```

or:

```text
execute only if the selector can exclude unsupported/ambiguous Live Photos while clearly reporting blocked counts
```

Do not silently pretend all supported assets includes items that were actually skipped without reporting.

Lock-in answers:

1. `all_supported_media` with unsupported/ambiguous candidates present

Yes. Proceed when the selected target set contains safe/proven logical assets, even if unsupported or ambiguous candidates are also present in the scanned candidate set.

Do not stop merely because unsupported/ambiguous candidates exist somewhere in the scan window.

Required behavior:

```text
selected set: safe/proven assets only
unsupported/ambiguous candidates: excluded and counted/reported
no silent downgrade
no unsupported selected resources
```

Stop only if the requested run cannot select a safe target set without violating scope semantics or reconciliation safety.

2. Live Photo pairing proof incomplete

Use a balanced rule.

If a Live Photo candidate itself has incomplete/ambiguous pairing, exclude/block that candidate and report:

```text
LIVE_PHOTO_PAIRING_NOT_PROVEN
```

But do not hard-stop the entire `all_supported_media` run if other safe assets can fill the requested logical batch and the report clearly counts the blocked/ambiguous candidate.

Hard-stop only if:

```text
the selected set would include an unproven/incomplete Live Photo pair
or all_supported_media cannot be honored without silently skipping unsupported classes
or logical/resource reconciliation is ambiguous for the selected set
```

In plain English: proceed with safe selected assets; block unsafe candidates; do not select half-proven Live Photos.

3. Batch fill semantics

Yes. Define batch fill strictly by logical asset count.

```text
batch_size = logical assets
total_limit = logical assets
resources_selected may be greater than logical_assets_selected
```

Example:

```text
batch_size: 3

Selected:
1 ordinary still = 1 logical asset, 1 resource
1 video = 1 logical asset, 1 resource
1 Live Photo = 1 logical asset, 2 resources

logical_assets_selected: 3
resources_selected: 4
```

Update the execution-safe criterion accordingly. Do not require selected resource count to equal batch target.
