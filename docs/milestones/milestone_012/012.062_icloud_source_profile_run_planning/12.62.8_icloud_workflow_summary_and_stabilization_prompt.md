# Summary Content

The summary should be organized into three compact sections:

```text
1. Acquisition
2. Source Intake
3. Overall Result / Next Step
```

## 1. Acquisition Section

Show available fields from the latest relevant iCloud acquisition status/report.

Use only fields that exist or can be safely derived.

Preferred fields:

```text
- acquisition status
- started / finished time, if available
- requested recent count
- file inventory count
- downloaded / staged count, if available
- skipped count, if available
- failed count, if available
- acquisition mode
- report path/reference, if available
```

If only partial data is available, show partial data clearly.

Example wording:

```text

```

If no recent acquisition exists:

```text

```

Do not block Source Intake solely because no recent acquisition exists, provided readiness/path checks are valid.

---

## 2. Source Intake Section

Show available fields from the latest relevant Source Intake status/report for the same source profile.

```text
- selected
- staged to Drop Zone

```

# Summary Logic

Use conservative logic.

## Ready for Source Intake

Show when:

```text
- profile is iCloud
- profile is active
- readiness has no blocking path/source-registration issues
- no operation conflict exists
- acquisition is not currently running
```

It may still show a warning if:

```text
- no recent acquisition exists
- staged inventory count is unavailable
- acquisition was stopped but files may exist
```

## Ready for Cleanup Later

Show only when:

```text
- Source Intake has terminal status
- Source Intake did not have obvious fatal failure
- summary data is available enough for operator review
```

But wording must say:

```text
Cleanup will be added in a later milestone.
```

Do not add cleanup action in this milestone.

## Attention Needed

Show when:

```text
- readiness is not_ready
- blocking reasons exist
- path mismatch exists
- source registration mismatch exists
- approved root blocked
- auth required/session expired
- active operation conflict exists
```

Display reason details if already available.

# Answers to Coder Questions — Milestone 12.62.8

## 1. “Latest relevant” acquisition/intake record selection

Use the most recent record by timestamp for that source, even if failed/stopped.

Do not prefer older successful terminal records over newer failed/stopped records.

Reason:

```text
The operator summary should reflect the current operational reality, not the last clean success.
If the most recent acquisition or intake failed/stopped, that is what the user needs to see.
```

You may also show a small note if an older successful run exists, but do not make it the primary summary unless the newer run is unrelated to the selected source.

---

## 2. Source Intake active run vs latest report

Yes. Prioritize a currently active Source Intake run for this source over the latest completed report.

Precedence:

```text
1. Active Source Intake run for this source
2. Most recent terminal Source Intake run/report for this source
3. No Source Intake run found
```

Reason:

```text
If Source Intake is running now, the operator needs live status more than historical report data.
```

If an active run exists for a different source, show it only as an operation conflict/blocker, not as this profile’s Source Intake summary.

---

## 3. Overall Result precedence: blockers vs active-running states

Use this precedence:

```text
1. Active operation for this same iCloud profile
2. Hard blockers / attention-needed state
3. Active operation conflict for another source
4. Ready for Source Intake / ready for acquisition
5. Review results
6. No recent activity / ready to acquire
```

Interpretation:

```text
If this profile is actively acquiring or running Source Intake:
  show the active-running state first.

If this profile is not active but has readiness blockers:
  show Attention Needed.

If another operation is active:
  show Operation Conflict.

If no blockers/conflicts:
  show the next logical action.
```

Reason:

```text
A same-profile active run is the most important real-time state.
Readiness blockers matter most when no run is already in progress.
```

Add a short code comment documenting this precedence table.

---

## 4. Button wording: “Prepare Source Intake” vs “Run Source Intake for Staged iCloud Files”

Rename the visible action to:

```text
Run Source Intake for Staged iCloud Files
```

This is clearer and matches the workflow.

However, preserve the safety behavior:

```text
Clicking the button opens the confirmation dialog.
It does not immediately start Source Intake.
```

Optional supporting text under the button:

```text
You will confirm Total Limit and Batch Size before the run starts.
```

Avoid wording that implies automatic execution without confirmation.

---

## 5. Report references: text paths or clickable links?

Use simple text paths/references for now.

Clickable links are allowed only if there is already an existing safe UI pattern for opening local reports from the app.

Default:

```text
Show report path/reference as text.
Do not invent a new file-opening route in this milestone.
```

Reason:

```text
The goal is operator clarity without adding a new report browser or file-serving behavior.
```

---

## 6. No recent acquisition, but recent Source Intake exists

Yes. If no recent acquisition exists but recent Source Intake exists for this iCloud profile, Overall Result should prioritize:

```text
Review intake results
```

not:

```text
Ready to acquire
```

Recommended wording:

```text
No recent acquisition found. Recent Source Intake results are available for this profile. Review intake results or acquire again if you need newer iCloud files.
```

Reason:

```text
The existence of Source Intake results is more informative than simply saying the user is ready to acquire.
```

---

## 7. Legacy iCloud profiles

Yes. Include legacy iCloud profiles in the summary using the same fallback detection introduced in 12.62.7.

Use the same iCloud-profile detection helper/logic consistently across:

```text
- readiness display
- Acquire from iCloud
- Source Intake handoff
- combined workflow summary
```

Do not create a second, divergent detection rule.

Reason:

```text
Older cloud_export records should not disappear from the iCloud workflow just because cloud_provider was not explicitly populated.
```

---

## 8. Stale-state handling

Yes. Prefer auto-refresh on terminal transitions first.

Use manual Refresh Summary only as fallback.

Preferred behavior:

```text
When acquisition reaches a terminal state:
  refresh readiness/status/summary using existing patterns.

When Source Intake reaches a terminal state:
  refresh readiness/status/summary using existing patterns.

Do not add a new persistent polling loop after terminal states.
```

If automatic refresh is unreliable or too invasive, add a lightweight button:

```text
Refresh Summary
```

Do not overbuild this.
