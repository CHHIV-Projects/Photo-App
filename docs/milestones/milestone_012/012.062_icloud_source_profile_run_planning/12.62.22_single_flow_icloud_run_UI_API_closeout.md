## 2. Backend Changes

### 2.1 Schemas

Added new payloads in admin schemas:

- `InternalIcloudRunRequest`
- `InternalIcloudRunStatus`
- `InternalIcloudRunResponse`
- `InternalIcloudRunStatusResponse`

Behavioral notes:

- bounded limits are validated at schema layer (`batch_size`, `total_limit`, `candidate_search_cap`)
- status payload carries explicit safety/report fields and execution-state counters

### 2.2 Service Wrapper

Created internal wrapper service:

- path: `backend/app/services/admin/internal_icloud_run_service.py`
- key functions:
  - `start_internal_single_flow_run(...)`
  - `get_internal_single_flow_run_status(...)`

Key guardrail/safety behavior:

- source profile existence required
- cross-operation guardrail snapshot respected before launch
- dry-run plan must be execution-safe before starting background execution
- only `ordinary_stills` is executable in this milestone
- non-ordinary scope returns stopped status with `MEDIA_SCOPE_NOT_SUPPORTED_FOR_EXECUTION`
- `auto_cleanup_if_safe=true` continues guarded cleanup automatically when orchestration pauses for cleanup review
- `auto_cleanup_if_safe=false` leaves explicit manual cleanup continuation action

### 2.3 Admin API Routes

Added routes under existing admin router:

- `POST /api/admin/internal/icloud-runs` (start)
- `GET /api/admin/internal/icloud-runs/{run_id}` (poll)

Response semantics:

- accepted start -> `200` with `status="started"`
- safe non-accepted start (e.g., unsupported media scope) -> `409` with `status="stopped"` and detailed `current`
- status missing run -> `404` with `INTERNAL_ICLOUD_RUN_NOT_FOUND`

## 3. Frontend Changes

### 3.1 Types + API Client

Added internal run types to shared UI API definitions and client functions:

- `startInternalIcloudRun(req)`
- `getInternalIcloudRunStatus(runId)`

Updated API client behavior for `startInternalIcloudRun(req)`:

- safe HTTP `409` responses from `POST /api/admin/internal/icloud-runs` are now parsed and returned as valid `InternalIcloudRunResponse` payloads
- this preserves backend semantics while avoiding generic request-failure handling for expected safe-stop paths

### 3.2 Admin UI Card

Added a dedicated component and integrated it in Admin view:

- new component: `frontend/src/components/IcloudInternalRunCard.tsx`
- styles: `frontend/src/components/IcloudInternalRunCard.module.css`
- mounted from `AdminView`

UI behavior:

- select iCloud source profile
- configure batch size / total limit / candidate search cap / media scope / auto-cleanup
- start run and manually refresh
- auto-poll while run is `running` or `paused_for_cleanup`
- show run id, phase, stop/failure reason, safety flags, and report path
- for safe-stop `409` responses, render backend payload fields instead of generic request failure text (including `status`, `stop_reason`, scope fields, `next_safe_action`/`failure_reason`, and `cloud_deletion_occurred`)
