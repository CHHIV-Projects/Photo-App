## Architectural boundary

Preserve this boundary:

```text
iCloud acquisition/backfill
    ↓ staged files + manifest
Source Intake / Ingestion
    ↓ assets + provenance + metadata + duplicate/canonicalization logic
Photo Library / Vault
```

### Acquisition may do this

```text
- read eligible inventory rows
- perform bounded iCloud download/acquisition using existing exact-selection machinery where possible
- stage downloaded files only in the existing controlled iCloud staging path
- hand off to existing Source Intake
- update inventory/backfill state after successful Source Intake
- record run/status/report data
```

### Acquisition must not do this

```text
- write directly to Vault
- bypass Source Intake
- redesign duplicate detection
- redesign canonicalization
- change Source Intake semantics
- delete remote iCloud assets
- perform unbounded downloads
- perform scheduler/automation work
- add broad Admin UI redesign
```

Source Intake remains the only owner of:

```text
hashing
exact duplicate detection
Asset creation/reuse
provenance recording
metadata extraction
canonicalization
Vault placement
```

## Decisions

### 1. Completion naming

Use safer naming:

```text
backfill_completed
backfill_completed_at
backfill_resolution_state
```

Do not make `acquired=true` mean “new Vault asset was created.”

A completed inventory row may resolve as:

```text
newly_imported
already_known
exact_duplicate_resolved
skipped_unsupported
skipped_stale_retryable
failed_retryable
failed_terminal
```

If an `acquired` field already exists from 12.62.27, do not remove it in this milestone. Prefer additive fields and clarify semantics in code/tests. Avoid destructive schema changes.

### 2. 12.62.28.2 scope

Implement only:

```text
- inventory row selector
- fresh helper listing re-resolution
- selected logical-item/resource preview construction from fresh listing metadata
- dry-run/preview backend service
- dry-run/preview Admin API endpoint
- status count extensions if needed
- focused tests
- closeout
```

Do not implement:

```text
- actual helper download
- staging writes
- Source Intake handoff
- Vault writes
- cleanup
- Admin UI
- scheduler
```

### 3. API shape

For 12.62.28.2, use a clearly dry-run/preview endpoint, for example:

```text
POST /api/admin/icloud-backfill/acquire-preview
```

Request fields:

```text
source_id
acquire_limit default 500, max 1000
max_listing_candidates max 100000
```

Response should include:

```text
selected_inventory_count
matched_listing_count
preview_selected_logical_count
preview_selected_resource_count
skipped_stale_count
skipped_known_count
skipped_unsupported_count
skipped_ambiguous_count
stop_reason
next_safe_action
```

Use existing admin API style.

Do not add an execute acquisition endpoint until 12.62.28.3.

### 4. Source Intake decision

Defer automatic Source Intake to 12.62.28.3.

For 12.62.28.2, no Source Intake call should occur. Tests should assert it is not called.

### 5. Stale rows

Rows missing from the fresh helper listing should remain retryable, not terminal.

Use a state or reason like:

```text
skipped_stale_listing
```

but do not mark `backfill_completed=true`.

They can be retried after a later inventory scan.

### 6. Ordering

Use deterministic inventory selection order:

```text
created_remote_at ASC NULLS LAST
added_remote_at ASC NULLS LAST
first_observed_at ASC
id ASC
```

Do not use `observed_remote_position` as a cursor.

### 7. Live Photos

Live Photo inventory rows remain one logical asset.

Dry-run preview should show:

```text
logical_count = 1
resource_count >= 2 when helper listing exposes paired resources
```

Do not allow partial Live Photo manifest preview to be treated as fully acquisition-ready.
