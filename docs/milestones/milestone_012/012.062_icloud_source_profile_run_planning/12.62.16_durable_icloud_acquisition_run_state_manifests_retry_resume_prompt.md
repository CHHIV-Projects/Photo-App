# Milestone 12.62.16 — Durable iCloud Acquisition Run State, Manifests, Retry, and Resume

## Recommended Reasoning Level

Extra High.

This milestone crosses real iCloud validation, exact selected-resource acquisition, batch manifests, retry/resume, timeout handling, crash recovery, partial staging, Source Intake handoff boundaries, and future durable multi-batch orchestration.

## Goal

Validate the 12.62.15 exact-selection helper against a bounded real iCloud account run, then implement the durable acquisition run-state foundation required for secure batch/limit iCloud ingestion.

The durable target is:

```text
Plan exact new logical iCloud items
→ freeze a secret-free selection manifest
→ download selected resources through .partial staging
→ verify item/resource completion
→ publish only complete items
→ record per-item/per-resource state
→ support retry/resume/stop/failure reporting
→ produce the acquisition batch inventory that Source Intake will later consume
```

This milestone does **not** implement the full acquire → Source Intake → cleanup loop yet.

## Product Direction / Lock-ins

The user has chosen the secure and durable iCloud path.

Durable iCloud ingestion must eventually support:

```text
Batch: 50
Limit: 1000
Mode: Download only new

Acquire 50 new logical iCloud items
→ Source Intake those 50
→ Cleanup those 50
→ checkpoint
→ repeat until total limit, no more new items, user stop, or recoverable/unrecoverable failure
```

Key product lock-ins:

- `N` counts logical iCloud items, not raw resource files.

- Reports must show both logical item counts and physical resource/file counts.

- Live Photos should be treated as one logical item with multiple resources where safely identifiable.

- Batch and total limit belong at acquisition, not Source Intake.

- Source Intake should eventually process the acquisition batch inventory.

- Cleanup should eventually act on files verified by Source Intake for that batch.

- Authentication may eventually be guided from the Photo Organizer UI, but Photo Organizer must not own Apple credentials.

Authentication direction:

- Photo Organizer may eventually expose UI actions such as:
  
  - Check iCloud Session
  
  - Authenticate iCloud
  
  - Re-authenticate
  
  - Open iCloud Authentication Helper

- Actual credential/session handling must remain in an isolated helper, browser flow, or external process.

- Photo Organizer must not collect, store, log, or persist Apple passwords, 2FA codes, cookies, tokens, or session secrets.

- Photo Organizer may persist only secret-free auth state.

## Publication / Crash Safety Requirements

12.62.15 identified a narrow crash window: an OS/process crash after one resource rename but before the final resource rename could leave a partially published logical item in normal staging.

This milestone should improve or explicitly model that risk.

Preferred direction:

- use a batch/item manifest gate so Source Intake can identify which files belong to a completed acquisition batch

- ensure Source Intake does not process incomplete logical items

- reconcile normal-staging files against a batch manifest before declaring the batch ready for Source Intake

- preserve `.partial` suffix for incomplete files

- block or quarantine suspicious partial publication states

If full crash-safe publication cannot be completed in this milestone, document exactly what remains and how Source Intake will be protected before production use.

## Retry / Resume Requirements

For a single acquisition batch, define and implement where feasible:

1. Retry after helper timeout.

2. Retry after auth/session failure once re-authenticated.

3. Retry after network/iCloud timeout.

4. Retry after partial item failure.

5. Resume/reconcile after lost helper response.

6. Detect already published complete files from the frozen manifest.

7. Detect incomplete or conflicting files.

8. Avoid redownloading known resources where safe.

9. Avoid publishing duplicate or ambiguous resources.

10. Keep next safe action clear.

Do not claim durable resume if the implementation only restarts blindly.

## Restart-safe identity

Approved: you may persist a per-run salted SHA-256 digest of each remote logical-item ID for durable run/restart safety.

Rules:

* raw remote IDs remain only in memory or the transient JSON pipe
* raw remote IDs must not appear in command arguments, logs, reports, filenames, DB records, provenance, or UI
* the salt should be per run
* persisted digest is for run/resume matching only
* this does not make remote iCloud IDs part of provenance
* on resume, the helper must re-list, hash transient IDs, require one unique digest match, and verify expected path, role, size, and checksum before treating it as the same item

Without this digest approach, true post-restart resume would not be reliable, so this is approved for the durable-run foundation.
