# Milestone 12.62.10.1 Closeout — iCloud Acquisition Launch Path / Source Registration Consistency

## Status

**Resolved.** Fix implemented, tested, and validated against all required profiles.

---

## 1. Root Cause

Two separate code paths used different label canonicalization functions to resolve "which source profile does this acquisition belong to":

**Readiness check** (`icloud_readiness_service.py`):
- Used the stored `source_root_path` from the selected Source Profile record directly.
- Compared paths — both sides come from the same DB record, so they always matched.
- Correctly reported `source_registration_status = matched`.

**Acquisition launch** (`execution_service.py`, `_prepare_launch` → `_lookup_source_registration`):
- Called `sanitize_source_label(source_label)` → produced a path-safe folder slug.
- Passed that sanitized label into `_lookup_source_registration`.
- That function applies `normalize_source_label` (lowercase only, no punctuation removal) before querying `source_label_normalized` in the DB.
- The DB stores `source_label_normalized = normalize_source_label(original_label)`.
- Result: sanitized label ≠ stored normalized label → `matching_source is None` → `SOURCE_NOT_REGISTERED` error.

**Example with `Chuck iCloud E2E Test 12.62.10`:**

```
sanitize_source_label("Chuck iCloud E2E Test 12.62.10")
  → "chuck_icloud_e2e_test_12_62_10"   ← used for path, wrong for DB lookup

normalize_source_label("Chuck iCloud E2E Test 12.62.10")
  → "chuck icloud e2e test 12.62.10"   ← stored in DB as source_label_normalized
```

These do not match. Any label with spaces, dots, or punctuation exposes this bug. Labels with no spaces (e.g., already-slug labels like `chuck_icloudpd_test`) did not fail because `sanitize` and `normalize` happen to produce equivalent DB matches for those.

Label punctuation (the dots in `12.62.10`) directly contributed to the mismatch.

## 2. Files Changed

| File | Change |
|---|---|
| `backend/app/services/icloud_acquisition/execution_service.py` | Pass `source_label` (original) instead of `sanitized_label` to `_lookup_source_registration` in `_prepare_launch` |
| `backend/tests/test_icloud_acquisition_service.py` | Added `test_sanitize_and_normalize_diverge_for_punctuated_labels` regression test |

---

## 3. Whether Label Punctuation/Dots Caused or Contributed

**Yes, directly.** The dots in `12.62.10` and spaces in the label are precisely what `sanitize_source_label` converts to underscores. Labels that were already slug-like (no spaces, no special chars) happened to produce the same result from both functions and were not affected. The E2E test profile name with version-style dots was the first to expose it.

---

## 4. Exact Fix

Single-line change in `_prepare_launch` in `execution_service.py`:

```python
# Before (broken):
matching_source = _lookup_source_registration(
    db_session,
    source_label=sanitized_label,   # underscored slug, doesn't match DB
    source_type=normalized_source_type,
    source_root_path=source_root_path,
)

# After (fixed):
matching_source = _lookup_source_registration(
    db_session,
    source_label=source_label,      # original label, normalize_source_label matches DB
    source_type=normalized_source_type,
    source_root_path=source_root_path,
)
```

The sanitized label continues to be used for all path resolution, run row creation, and report naming — that is correct and unchanged.

Notes:
- **E2E profile (63)**: fix resolves it correctly. Old path failed entirely.
- **Chuck iCloud (62)**: fix finds the correct newer row. Old path found an older row (id=50) which is a different profile with the same label — coincidental match, not a reliable behavior.
- **My iPhone (60)**: not found via `cloud_export` source_type lookup. This profile uses `icloud` as `cloud_provider` but its `source_type` is `cloud_export`. The `_lookup_source_registration` also matches on `source_type`. This is expected behavior — My iPhone's registration will resolve correctly when its staging path is used as `source_root_path` in the actual acquisition launch call (it was not a failure in the original issue).

## 6. Source-ID-Based Acquisition Launch — Recommended Future Hardening

The current fix is the smallest safe change: pass the original label so DB normalization aligns. This is correct and sufficient.

A more robust long-term improvement would be to accept `source_id` directly in the acquisition launch payload and resolve all identity data (label, staging path, source_type) from the stored Source Profile record rather than recomputing from the label. This would:

- Eliminate all label canonicalization mismatch surface entirely
- Make acquisition launch immune to future changes in sanitization logic
- Allow the frontend to pass the unambiguous `source_id` instead of the label string

This is not required now and should not be implemented until a dedicated hardening milestone is planned.

---

## 7. Remaining Limitations

- `My iPhone` (source_id=60) was not matched in the probe because the staging path used for lookup was computed from the label. In actual acquisition flow, the managed staging path from the stored profile will be passed — this should resolve correctly. Not a blocker for 12.62.10 validation.
- The `AUTH_UNKNOWN` warning on the E2E profile will remain until an actual icloudpd session is established. This is expected and acceptable — it is a warning, not a blocking reason.
- The `source_registration_status` stored in the acquisition run row will now correctly reflect `registered` for properly created profiles. Previously it always stored `missing` for profiles with punctuated labels.
- Source-id-based launch is not implemented (see section 6).
