# Milestone 12.62.17 — iCloud Acquisition Batch Source Intake Handoff

## 1. Scope Completed

Implemented the internal durable iCloud acquisition batch -> Source Intake handoff.

Completed scope:

- Added an internal batch Source Intake service for durable exact-selection iCloud acquisition batches.
- Added an internal backend script for operator/manual validation.
- Added explicit selected-record support to the ingestion pipeline so Source Intake can process only the acquisition batch inventory.
- Added nullable Source Intake linkage/status fields to durable iCloud acquisition batch/item/resource rows.
- Added nullable Source Intake mode/batch linkage fields to `source_intake_runs`.
- Added schema-sync support for existing databases.
- Added batch-level reporting for iCloud acquisition batch Source Intake.
- Added conservative retry/reconciliation behavior.
- Added tests for success, duplicate/existing asset provenance linkage, wrong-profile rejection, missing file blocking, batch-not-ready blocking, partial-file exclusion, SHA mismatch blocking, and safe retry/reconciliation.

No normal UI, normal Admin API route, one-click orchestration, automatic Source Intake, or automatic cleanup was added.

## 2. Operational Behavior

The internal flow is now:

```text
durable exact-selection acquisition batch
-> verify batch/source/root/resource readiness
-> build explicit FileScanRecord inventory from ready acquisition resources
-> Source Intake stages only those files to Drop Zone
-> normal Source Intake Vault/Asset/provenance/duplicate behavior runs
-> acquisition batch/item/resource rows receive Source Intake outcome evidence
-> batch is marked ready for guarded cleanup dry run when all resources are verified
```

The internal operator command is:

```powershell
.\.venv\Scripts\python.exe .\backend\scripts\run_icloud_batch_source_intake.py --batch-id <batch_id> --source-id <source_id>
```

The script prints JSON and returns success only when the batch reaches `intake_completed`.

The normal UI/API does not expose this path.

## 3. Source Intake Handoff Design

Source Intake can now consume a durable acquisition batch through:

```text
app.services.icloud_acquisition.batch_source_intake_service.run_batch_source_intake(...)
```

The handoff validates:

- acquisition batch exists
- acquisition run exists
- acquisition run mode is `internal_exact_selection`
- requested Source Profile matches the acquisition run Source Profile
- Source Profile is active
- Source Profile is an iCloud `cloud_export`
- managed staging root exists
- acquisition run staging path matches the Source Profile managed staging root
- batch is marked `batch_ready_for_source_intake`
- Drop Zone is empty
- no acquisition/source-intake/cleanup operation is active
- each selected resource path is relative and inside the managed staging root
- no selected resource is a `.partial` path
- each selected resource exists as a file
- file size still matches durable metadata
- local SHA-256 still matches durable metadata where present
- existing same-profile provenance either matches the durable SHA-256 or blocks as conflicting evidence

Source Intake uses acquisition batch inventory, not independent limit/batch input:

- `source_intake_limit` is `None`
- `ingest_batch_size` is set internally to the verified resource count
- the pipeline receives `explicit_source_records`
- the pipeline does not scan the full iCloud staging folder in this path

Existing Source Intake scan/limit behavior remains unchanged for local, external, existing guided Source Intake, standard iCloud, and list-first/non-repeat workflows.

Resource outcomes are linked back to durable acquisition rows:

- new unique asset persisted -> `resource_intake_processed`
- existing/Vault duplicate with selected-profile provenance recorded -> `resource_intake_duplicate_linked`
- already safely processed on retry -> `resource_intake_skipped_known`
- missing before intake -> `resource_missing_before_intake`
- SHA mismatch before intake -> `resource_sha_mismatch_before_intake`
- `.partial` selected before intake -> `resource_intake_deferred`
- unsafe/failed processing -> `resource_intake_failed`

Exact duplicates and Vault duplicates still flow through Source Intake. The existing persistence layer records or confirms provenance for the selected Source Profile when appropriate.

## 4. Cleanup Readiness Behavior

Cleanup readiness is batch-aware.

When all batch resources are safely processed, duplicate-linked, or reconciled as already known for the selected Source Profile, the acquisition batch is marked:

```text
ready_for_cleanup_dry_run = true
cleanup_readiness_reason = source_intake_verified
next_safe_action = Run guarded iCloud cleanup dry run
```

Actual cleanup execution remains the existing source-wide guarded cleanup.

This milestone does not implement batch-specific cleanup execution. The existing guarded cleanup remains conservative and verifies provenance, staged file hash, Vault file hash, and source identity before deleting local staging files.

## 7. Safety Boundaries Preserved

Preserved:

- Acquisition still writes only to managed iCloud staging.
- Acquisition still does not write to Drop Zone, Vault, Asset rows, or provenance.
- Source Intake remains the only path into Drop Zone, Vault, Asset rows, and provenance.
- Cleanup remains guarded and local-staging-only.
- No iCloud cloud deletion was added.
- No Vault deletion was added.
- No DB/provenance/source deletion was added.
- No Source Profile deletion was added.
- No Apple credentials, cookies, tokens, sessions, 2FA codes, raw remote IDs, or raw download URLs are stored.
- Existing `standard` and `list_first_non_repeat` iCloud acquisition behavior was not changed.
- Existing local/external Source Intake behavior was not changed.
- Existing iCloud fixed-window Source Intake behavior was not changed.
- Existing guarded cleanup behavior was not weakened.
