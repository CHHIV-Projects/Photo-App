# Milestone 12.62.24 — Bounded All-Supported iCloud Assets Live Validation

## Recommended Reasoning Level

High.

This is primarily a bounded live validation milestone with permission for small targeted fixes only if validation exposes a real integration bug.

Do not broaden into architecture redesign.

## Goal

Validate that the internal/admin single-flow iCloud run can process **all supported iCloud assets** through the new asset-oriented flow.

The product direction is:

```text
iCloud ingestion gets assets.
Photo Organizer interprets asset type after ingestion.
```

The goal is to prove that the single-flow wrapper can handle a bounded mix of:

```text
ordinary still asset
standalone video asset
Live Photo asset with paired still + motion resources
```

using:

```text
Asset Scope: All supported assets
```

## Background

12.62.22 added the internal/admin single-flow iCloud run UI/API.

12.62.23 then:

```text
removed the blanket all_supported_media execution block
mapped all_supported_media to ordinary_still_only=False
changed batch fill semantics from resource count to logical asset count
added mixed-asset report fields
updated UI wording from Media Scope to Asset Scope
confirmed no silent downgrade from all_supported_media to ordinary_stills
```

However, 12.62.23 did not complete a live all-supported-assets execution. The bounded live all-supported run stopped safely with:

```text
stop_reason: DRY_RUN_NOT_SAFE
reason: insufficient safe supported inventory within bounds
```

Standalone video live validation and Live Photo live validation were not performed/selected in 12.62.23.

12.62.24 should close that gap.

## Scope

Scope includes:

1. Prepare or confirm a small known iCloud test inventory.

2. Run bounded dry-run validation through the internal/admin single-flow path.

3. Run bounded execution if dry run is safe.

4. Confirm logical asset count versus physical resource count.

5. Confirm ordinary still handling.

6. Confirm standalone video handling if selected.

7. Confirm Live Photo still+motion companion pairing if selected.

8. Confirm Source Intake, ingestion, cleanup, and final verification.

9. Confirm no cloud deletion.

10. Confirm final staging, Drop Zone, and `.partial` states are clean.

11. Add only small targeted fixes if the validation exposes a clear integration bug.

12. Document results honestly.

## Out of Scope

Do not redesign media architecture.

Do not redesign Source Intake.

Do not redesign Live Photo pairing.

Do not add cloud deletion.

Do not expose this as normal user-facing UI.

Do not remove dry-run-before-execution.

Do not bypass Source Intake.

Do not weaken cleanup safety.

Do not run a large import.

Do not implement a new media-type product workflow.

Do not silently downgrade `All supported assets` to ordinary stills.

## Product Direction Lock-In

The final iCloud ingestion UX should not ask the user to choose media type.

The normal mode should become:

```text
Get new iCloud assets
```

For now, the internal/admin UI may still show Asset Scope for validation/debugging, but the product direction is:

```text
All supported assets = normal/default
```

Type-specific handling belongs inside Photo Organizer after ingestion.

## Acceptable Safe Stop Results

If the run stops safely, that is acceptable only if it reports a clear reason.

Examples:

```text
DRY_RUN_NOT_SAFE
LIVE_PHOTO_PAIRING_NOT_PROVEN
LIVE_PHOTO_PAIR_INCOMPLETE
VIDEO_ACQUISITION_NOT_PROVEN
LOGICAL_RESOURCE_RECONCILIATION_FAILED
UNSUPPORTED_RESOURCE_IN_SELECTED_SET
AMBIGUOUS_MEDIA_CLASSIFICATION
CLEANUP_TARGET_RECONCILIATION_FAILED
```

The closeout must explain:

```text
what was available
what was selected
what was blocked
why execution did not proceed
whether cloud deletion occurred
whether staging remained clean
```

Do not treat a safe stop as a failed milestone if it reveals a real missing piece honestly.

But if the stop is caused by stale wording, stale ordinary-stills-only assumptions, or artificial wrapper blocking, fix that in this milestone.

## Targeted Fix Permission
