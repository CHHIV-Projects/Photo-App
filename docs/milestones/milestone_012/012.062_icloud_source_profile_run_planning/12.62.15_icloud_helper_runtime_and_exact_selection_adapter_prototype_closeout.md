### Reproduction and verification

From the project root on Windows:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File ".\scripts\runtime\bootstrap_icloud_exact_helper.ps1"
powershell.exe -NoProfile -ExecutionPolicy Bypass -File ".\scripts\runtime\bootstrap_icloud_exact_helper.ps1" -VerifyOnly
```

Verification reads installed distribution metadata and `direct_url.json`; it asserts both version `1.32.3` and the exact Git commit. This is necessary because the upstream direct-Git build's `icloudpd --version` banner contains placeholder build constants and is not reliable provenance evidence.

### Authentication boundary

User-operated authentication is outside the Photo Organizer process:

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File ".\scripts\runtime\authenticate_icloud_exact_helper.ps1" -Username "<Apple ID>"
```

Authentication uses only the helper process, the OS keyring, and:

```text
%LOCALAPPDATA%\PhotoOrganizer\icloud_exact_helper\auth
```

The directory can be overridden with `PHOTO_ORGANIZER_ICLOUD_EXACT_AUTH_DIR`. The Photo Organizer backend receives only a secret-free auth state such as `authenticated`, `authentication_required`, `session_expired`, `reauthentication_required`, `authentication_failed`, or `helper_unavailable`.

Apple passwords, 2FA codes, keyring values, cookies, session tokens, and raw download URLs do not cross into the backend protocol.

## 3. Capability Findings

| Capability | Prototype result |
|---|---|
| Logical item ID | Available from pinned `PhotoAsset.id`; kept transient |
| Newest-to-oldest listing | Available through the selected photo library iterator |
| Item/resource relationship | Explicit from one `PhotoAsset` and its version map |
| Live Photo grouping | Primary original and motion original are one logical item |
| Exact selected download | Per-version download is executable; fixtures prove only requested unknown resources are opened |
| Known sibling avoidance | Proven; a known primary is not redownloaded when only motion is unknown |
| Unknown sibling preservation | Proven; a known sibling does not suppress the unknown resource or logical item |
| Profile-scoped known-state | Proven, including wrong-profile evidence not suppressing acquisition |
| Logical/resource counts | Reported separately for listing, selection, completion, and failure |
| Verification | Expected provider checksum and size are required and checked |
| Bounded pagination | Available within one bounded invocation through the provider's mutable offset iterator |
| Durable checkpoint | Not available; the numeric offset is not a cross-run continuation token |
| Remote sidecars | Block as `unsupported_remote_sidecar` |
| RAW/alternative pairs | Block as `unsupported_raw_or_alternative` |
| Adjusted/custom-rendered resources | Block as `unsupported_adjusted_resource` |

The list phase reads one candidate beyond the backend scan limit to distinguish `no_more_candidates` from `scan_limit_reached`. The download phase re-enumerates only the bounded candidate range, matches selected transient logical IDs, and revalidates each selected resource manifest before downloading.

Unsupported or colliding identity blocks rather than being silently ignored. A destination-path collision, duplicate logical ID, duplicate resource ID, absent primary identity, unsupported companion relationship, or missing verification metadata prevents ambiguous acquisition.

Remote IDs exist only in helper memory and the transient JSON pipe. They are not command arguments, filenames, planner summaries, reports, DB fields, Asset rows, provenance rows, or UI data.

## 4. Operational Behavior

### Internal prototype flow

```text
validate active iCloud Source Profile and canonical staging root
→ fail on prior .partial workspace files
→ block if normal staging contains unknown files awaiting Source Intake
→ obtain secret-free helper auth state
→ list a bounded logical candidate window
→ apply selected-profile durable known-state per resource
→ plan up to N unknown logical items
→ build an in-memory exact resource selection
→ revalidate profile, staging state, partial state, and auth
→ re-enumerate and revalidate selected remote manifests
→ download each selected resource to a filename ending in .partial
→ verify size and checksum
→ publish the logical item only after all selected resources for it pass
→ return secret-free item and resource counts/failures
```

Unknown files already in normal staging produce:

```text
staged_unknown_pending_intake
Run Source Intake first for staged unknown resources.
```

Files from an interrupted prior helper run beneath `.partial` conservatively block a new acquisition until resolved. Normal item failures clean the item's partial workspace. Every incomplete download filename ends in `.partial`, so current recursive Source Intake extension filtering does not treat it as a supported complete media file.

The adapter enforces a subprocess timeout, discards helper stderr, bounds request/response JSON, rejects secret- or URL-shaped helper output, and maps provider/auth/network/local failures to safe codes without returning upstream exception text.

### Production behavior

Production behavior is unchanged:

```text
standard
→ download the requested fixed recent window

list_first_non_repeat
→ inspect the requested fixed recent window
→ skip the download only when the complete window is durably known
→ otherwise download the complete window
```

No new acquisition mode, API route, UI option, background job, report type, or automatic Source Intake/cleanup handoff was registered. The prototype cannot be selected by a normal user.

## 10. Known Limitations

### Real provider behavior remains unvalidated

The source pin and callable contracts are verified, and provider mapping/exact download are fixture-tested. A real account could still expose an unmodeled CloudKit field shape, filename collision, auth transition, checksum variant, library behavior, or item type. The bounded list-only validation must precede any real download.

### Internal upstream API maintenance risk

`PhotoAsset`, `PhotoAlbum`, record fields, version enums, checksums, and authentication exceptions are upstream internals rather than a supported exact-selection API. Any source-pin upgrade requires compatibility fixtures and a new bounded real-account validation.

### Unsupported resource relationships block

Remote sidecars, RAW/alternative pairs, adjusted/custom-rendered resources, missing primary originals, and unavailable verification metadata are not acquired. They deliberately block the affected plan instead of risking incomplete logical items.

### Pagination is not a durable checkpoint

The provider's mutable numeric offset supports one bounded list/download invocation. It cannot safely resume across process restarts or library mutation. Remote-ID deduplication and a durable manifest are still required for retry/resume.

### List and download are separate enumerations

The helper re-enumerates the bounded window for download and matches the selected transient IDs and manifest. If the library changes between phases, execution fails with selected-item-not-found or manifest-changed rather than substituting a different asset. Durable run manifests must make this state visible and resumable.

### Narrow publication crash window

Ordinary move failures roll back already moved resources. An operating-system or process crash after the first resource rename but before the last resource rename can leave a partially published logical item in normal staging. `.partial` suffixes protect files that have not yet been renamed, but current Source Intake has no run-manifest gate for the already renamed sibling. Crash-safe manifest-gated publication is required in the durable-run milestone.

### Lost-response and retry idempotency

If all resources publish but the helper response is lost, a retry sees existing destination files and fails conservatively. It does not yet reconcile completed files against a durable batch manifest.

### Partial workspace recovery is conservative

Interrupted `.partial` files block subsequent prototype acquisition. There is no automated resume, quarantine, or verified cleanup workflow yet.

### Deployment portability

The helper Python adapter resolves both Windows `Scripts` and Unix `bin` layouts, but bootstrap and authentication scripts are currently PowerShell/Windows-oriented. A mini-server/Linux deployment needs an equivalent pinned environment installer, external keyring/session integration, filesystem permission review, and compatibility validation.

### Timeout policy is coarse

Provider HTTP calls use a 30-second timeout and the backend enforces a configurable helper-process timeout, defaulting to 7200 seconds. Durable orchestration still needs batch-level heartbeats, cancellation, retry classification, and shorter operation-specific timeout policy.
