# Milestone 12.62.22 — Single-Flow iCloud Run UI/API

## Goal

Implement a single-flow internal/admin iCloud run experience that lets the operator define one bounded run and have the system execute the proven internal loop automatically when safe.

The desired operator flow is:

1. Select/confirm Source Profile.

2. Set `batch_size`.

3. Set `total_limit`.

4. Select Media Scope.

5. Start run once.

6. System internally handles preflight, dry run, execution, ingestion, cleanup eligibility, safe cleanup/reconciliation, recovery handling if needed, final verification, and reporting.

This milestone should simplify operation without weakening the safety behavior validated in 12.62.21.

## Context

Milestone 12.62.21 successfully completed bounded live validation for Source Profile 66, including:

- real iCloud dry run

- real iCloud execution

- cleanup continuation

- live already-consumed preview recovery path

- clean final staging, Drop Zone, and `.partial` workspace

- no cloud deletion

- no normal UI/API exposure

- full backend suite passing

The next step is to wrap the proven internal-loop behavior in a simpler internal/admin flow.

This is not a redesign of Live Photo or video architecture. The current architecture is assumed capable of handling supported media types, including logical/resource counting and Live Photo pairing. This milestone should expose Media Scope as an operator control and report mixed-media counts clearly so later user testing can confirm behavior.


## Required Media Scope Options

Add or expose a clear Media Scope setting with these options, or equivalent names if existing enum names differ:

```text
ordinary_stills
stills_with_live_photo_pairs
videos_only
all_supported_media
```

Behavior should be explicit in reports.

Examples:

```text
ordinary_stills:
  normal still resources only; exclude MOV/video/Live Photo motion companions

stills_with_live_photo_pairs:
  still images plus paired Live Photo motion companions where supported

videos_only:
  supported video resources only

all_supported_media:
  all currently supported iCloud media resources, including ordinary stills, Live Photo pairs, and videos
```

Do not treat Live Photos/videos as structurally unsupported unless current code proves that limitation. If a limitation exists, stop and report it clearly before UI work proceeds.

## Desired Normal-Case Flow

The operator should not need separate manual dry-run, execution, and cleanup steps in the normal case.

When the operator starts a bounded run, the system should internally perform:

```text
1. preflight checks
2. bounded dry run
3. safe-to-execute validation
4. bounded execution
5. source intake handoff
6. ingestion
7. cleanup eligibility check
8. automatic cleanup/reconciliation if safe
9. recovery handling if applicable
10. final verification
11. final report
```

Manual/operator intervention should only be required if the system cannot prove that continuing is safe.

## Safety Requirements

Preserve the established safety semantics from 12.62.21.

The simplified flow must not skip:

```text
dry-run-before-execution
bounded selection validation
source profile validation
operation lock/conflict checks
staging/Drop Zone/.partial preflight checks
cleanup authorization linkage
cleanup target reconciliation
already-consumed/stale preview recovery handling where safe
final cleanup verification
final report generation
```

## Auto-Cleanup Rules

Auto-cleanup may proceed only when all required safety checks pass.

At minimum, auto-cleanup should require:

```text
cleanup dry run belongs to the same orchestration run
cleanup target count matches expected staged/intake result
cleanup target is local/staging cleanup only
no unknown staging files exist
Drop Zone state is clean or expected
.partial workspace state is clean or expected
no active conflicting operation lock exists
cleanup is explicitly tied to the current run
final cleanup verification will run after cleanup
```

If any condition fails, stop before cleanup and report a clear exception state.

## Required Stop / Alert Conditions

The single-flow run should stop safely and report a clear reason for any of the following:

```text
Source Profile missing, inactive, or mismatched
authentication/helper access failure
backend/import/runtime preflight failure
dry run not safe to execute
selected logical count exceeds total_limit
resource count does not reconcile with logical count
Media Scope cannot be honored
Live Photo pair incomplete or ambiguous, if selected media scope includes pairs
unexpected MOV/video handling failure, if selected media scope includes video
unknown files in staging
Drop Zone conflict
.partial workspace conflict
operation lock conflict
cleanup preview stale in an unsafe or ambiguous way
cleanup target does not match expected run
cleanup verification failure
any delete/acquisition/intake/ingestion failure requiring review
```

Already-consumed preview recovery should not be treated as terminal failure when the system can prove clean staging reconciliation and no second deletion is launched.

## 1. Scope: wrapper orchestration only

Yes — treat this milestone as a **thin single-flow wrapper over the existing internal loop service**, not a parallel flow and not a rewrite of acquisition/intake/cleanup semantics.

Do not change underlying acquisition, source-intake, ingestion, cleanup, or recovery semantics except where needed for:

* media-scope request plumbing
* UI/API run definition
* pollable status/reporting
* count/report fields
* stop/alert behavior

If implementation reveals an actual structural limitation in the existing media path, stop and report it clearly rather than quietly redesigning.

## 2. Media scope support

Use strict behavior. Do **not** silently downgrade or reinterpret the requested media scope.

Implement the requested scope values as the API/UI contract:

```text
ordinary_stills
stills_with_live_photo_pairs
videos_only
all_supported_media
```

If the current exact-selection/acquisition path does not yet fully support a requested scope end-to-end, use staged explicit behavior:

```text
status: stopped
stop_reason: MEDIA_SCOPE_NOT_SUPPORTED_FOR_EXECUTION
requested_media_scope: <requested>
effective_media_scope: null or not_executed
```

It is acceptable to expose a scope before it is executable only if the UI/report clearly says it is not yet supported for execution. Do not let the system proceed by falling back to `ordinary_stills` unless the operator explicitly selected `ordinary_stills`.

## 3. Admin/internal route location

Use the existing admin routing infrastructure if that is already the guarded/internal area, but give the route explicit internal naming so it cannot be confused with normal app UI/API.

Preferred shape, adapted to current conventions:

```text
/admin/internal/icloud-runs
/admin/internal/icloud-runs/{run_id}
```

or equivalent within the existing admin router.

Do not expose this through normal user-facing photo review, source setup, or production ingestion UI.

## 4. `auto_cleanup_if_safe=false`

When `auto_cleanup_if_safe=false`, the flow should pause/stop at the cleanup review boundary and preserve the old guarded behavior.

Use a clear status such as:

```text
cleanup_review_required_manual
```

or:

```text
paused_before_cleanup
```

In this mode:

* do not perform cleanup automatically
* persist/report the cleanup dry-run/preview ID if available
* allow continuation through the existing guarded continuation path
* preserve the confirmation/recovery semantics already validated

When `auto_cleanup_if_safe=true`, cleanup may proceed automatically only if all safe-cleanup/reconciliation checks pass.

## 5. UI execution model

Use the background/poll model.

The UI/API should submit one run definition and return quickly with a run ID/status reference. The orchestration can run in the background or through the existing background/report pattern.

Preferred behavior:

```text
POST starts run and returns run_id
GET polls run status/report
UI refreshes progress/status
```

Avoid a long blocking HTTP request for the full orchestration.

## 6. “Media Scope cannot be honored” behavior

Use both preflight and post-dry-run blocking depending on when the mismatch is knowable.

Preflight-block before dry run when the system already knows the requested scope cannot be executed, for example:

```text
requested scope unsupported by current selector
required adapter capability missing
configuration missing
```

Allow dry run first when support is expected but actual candidate evidence may reveal a mismatch, for example:

```text
Live Photo pairing ambiguity
video resource mismatch
logical/resource reconciliation issue
scope filter not honored by selected candidates
```

In that case, block before execution.

Do not execute if the requested scope cannot be proven honored.
