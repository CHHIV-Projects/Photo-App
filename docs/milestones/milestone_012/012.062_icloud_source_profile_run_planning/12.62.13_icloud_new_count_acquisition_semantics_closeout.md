## 2. Operational Behavior

### Capability audit

The project currently uses `icloudpd 1.32.3`. The matching official repository tag `v1.32.3` at commit `879c561240d993d748ddb4546f935090502b16d3` was inspected, including:

- `docs/reference.md`
- `src/icloudpd/cli.py`
- `src/icloudpd/base.py`
- `src/pyicloud_ipd/services/photos.py`

The supported CLI behaviors do not provide the contract required for safe new-count execution:

- `--recent N` enumerates exactly the N most recently added remote assets.
- `--until-found N` stops after N consecutive resources already present in the local download directory; it does not use Photo Organizer's durable, profile-scoped known-state and does not fill historical gaps reliably.
- `--only-print-filenames` prints local paths that would be downloaded, but does not expose stable cloud asset IDs, logical-item boundaries, a continuation token, or a remote-candidate offset.
- The CLI has no exact asset-ID, filename, or resource allowlist for downloading an arbitrary planner-selected set.
- Album/date filters cannot safely express an arbitrary mixed known/new selection.

The internal Python library contains paginated photo-album objects, cloud asset IDs, and per-asset download operations. Calling those internals directly would be a new raw iCloud adapter with new authentication, compatibility, failure, and maintenance responsibilities. That expansion was not approved for this milestone.

Consequently, the current adapter cannot both:

1. scan farther back until it finds up to N unknown logical items, and
2. download only the exact selected unknown resources without redownloading known resources.

Downloading an oversized recent window and deleting unwanted files afterward was rejected because it would still reacquire known resources, create an ambiguous staged batch, and misrepresent execution as selective acquisition.

### Approved count semantics

The internal planner defines:

```text
target_new_item_count
    Number of logical iCloud items requested by the user.

candidate_scan_limit
    Maximum logical candidates the backend may consider during one plan.

candidate_resource_count
    Total physical resource paths represented by considered logical items.

selected_new_item_count
    Unknown logical items selected, up to target_new_item_count.

selected_new_resource_count
    Unknown physical resources required for the selected logical items.
```

The target counts logical items rather than raw files. A Live Photo still and its recognized motion companion count as one logical item while retaining two separately reported resources. Required unknown resources of a selected logical item are selected together in one plan.

The planner never treats a known sibling as proof that another resource is known. For example, a Vault-verified HEIC does not suppress an unknown `_HEVC.MOV` resource.

### Logical-item grouping

The current CLI output lacks cloud asset IDs and explicit logical-item/resource relationships. The planner therefore uses deliberately narrow filename grouping only to define and test the desired contract:

- a still plus a same-directory, same-basename `.MOV` may form one Live Photo item
- a still plus a same-directory `_HEVC.MOV` companion may form one Live Photo item
- `.AAE` and other sidecars remain independent resources/items
- multiple matching still candidates make the motion relationship ambiguous and block the plan
- an unavailable normalized identity blocks the plan rather than suppressing a candidate

This grouping is not sufficient for production execution against arbitrary real libraries. Stable remote logical-item and resource identities remain an adapter prerequisite.

### Known-state and selection behavior

The planner consumes candidates already evaluated against one Source Profile by the existing `known_state_service`. It does not create another known-state store.

For a safe future adapter, the intended flow is:

```text
newest-to-oldest logical candidates
→ evaluate every resource against profile-scoped durable known-state
→ skip fully known logical items
→ select unknown logical items until the target is met
→ select only required unknown resources for those logical items
→ stop at the target, source exhaustion, or an explicit bounded condition
```

If an unknown resource is already present in managed staging but is not ingested, planning blocks with:

```text
STAGED_UNKNOWN_PENDING_INTAKE
Run Source Intake first for staged unknown resources.
```

This prevents acquisition from adding an oversized or ambiguous mixed batch. A future runtime integration must inventory managed staging independently; `icloudpd --only-print-filenames` can omit resources that already exist locally.

### Planner outcomes

The planner reports explicit classifications and stopping reasons:

| Classification | Stopping reason | Meaning |
|---|---|---|
| `complete` | `target_new_count_reached` | The plan selected the requested logical new-item count. |
| `partial` | `no_more_candidates` | The candidate source was exhausted before the target was met. |
| `partial` | `scan_limit_reached` | The backend-controlled logical candidate cap stopped the plan. |
| `tooling_limited` | `tooling_limit_reached` | The provider returned too little evidence without proving exhaustion or reaching the backend cap. |
| `blocked` | `staged_unknown_pending_intake` | Unknown resources are already staged; Source Intake must run first. |
| `blocked` | `identity_unavailable` | A candidate lacks safe normalized identity. |
| `blocked` | `logical_item_identity_ambiguous` | Resource-to-item grouping is not deterministic. |

The generated summary is explicitly marked:

```json
{
  "mode": "new_count_planner_only",
  "planner_only": true
}
```

It is bounded to a configurable sample count and is not written as an acquisition execution report.

### Production/UI behavior

No new acquisition mode was registered. No API request field, run route, modal option, or workflow action invokes the planner.

The existing UI remains labeled:

```text
List first / non-repeat
```

Its existing helper text describes fixed recent-window, window-level behavior. It is accurate for the current implementation and is not labeled `Download new items only`.

Current production behavior remains:

```text
inspect the fixed recent window
→ skip the download subprocess only if the complete window is durably known
→ otherwise download the complete recent window
→ stop
```

Therefore known resources are still redownloaded in a mixed known/new production window, and production does not continue scanning farther back to fill a target of N new logical items.

## 8. Known Limitations

### True new-count acquisition remains blocked

The production application still cannot promise:

```text
download up to N unknown logical iCloud items
while skipping known resources individually
and scanning farther back as necessary
```

The current `list_first_non_repeat` mode still downloads the complete recent window whenever that window is mixed. Known resources are therefore not avoided in mixed known/new production downloads.

### Planner identity is not a production cloud identity

The planner consumes normalized source-relative paths because that is the identity available to the current Photo Organizer known-state service. Its narrow filename-based Live Photo grouping is useful for specifying and testing semantics, but it cannot replace stable remote asset IDs and explicit resource relationships from an acquisition adapter.

Filename collisions, renamed resources, alternate Live Photo naming, edited variants, burst assets, and other iCloud resource structures may not be safely groupable from CLI path output alone. Ambiguity must continue to block.

### Candidate pagination/checkpoint is not implemented

The planner accepts an ordered candidate sequence and reports whether a backend scan cap, source end, or provider/tooling limitation stopped selection. The current adapter cannot supply a stable paginated sequence or checkpoint through the CLI, so these planner outcomes are not yet connected to real acquisition.

The backend default and maximum scan limit are both currently `500` logical candidates for pure-planner validation. A future adapter milestone must tune operational bounds using real response sizes, latency, retries, and large-library behavior before exposing Advanced settings.

### Atomic logical-item download is not implemented

The planner selects all required unknown resources represented for a chosen logical item. It cannot make the current CLI download those exact resources atomically. A future adapter must define retry and rollback/reporting behavior for partial resource failure within one logical item.

### Staging gate is not wired to execution

The planner blocks when its supplied known-state evaluation marks an unknown resource as staged. A future runtime must perform a complete managed-staging inventory before candidate selection rather than relying on `--only-print-filenames`, because the tool may omit already-local paths.

### Batch/limit workflow remains deferred

The following are not yet safe to build on top of the current adapter:

- normal UI target-new-item control
- Advanced candidate scan limit
- overall import limit or multi-batch orchestration
- resume/checkpoint controls
- planner/execution comparison reports
- Source Intake handoff for exact selected batches

Those features require executable exact-selection semantics first.

## 9. Recommended Next Milestone

Create an Extra High-reasoning adapter decision/implementation milestone with explicit approval for one of these paths:

1. Prefer an upstream-supported `icloudpd` interface that exposes stable logical asset IDs, paginated/continuable listing, item-resource relationships, and an exact asset/resource download allowlist.
2. If no supported interface exists, design a dedicated Photo Organizer iCloud adapter around the library's asset-ID and pagination capabilities, with an explicit compatibility and authentication boundary. Direct use of `pyicloud_ipd` internals must be treated as a new adapter, not a small wrapper change.

The next milestone should first prove, against disposable fixtures and then one bounded user-operated real profile, that the adapter can:

```text
list stable logical candidates newest-to-oldest
→ retain explicit item/resource relationships
→ evaluate source-profile scoped durable known-state
→ block when unknown resources are already staged
→ continue until N unknown logical items, source end, or scan limit
→ download exactly the planner-selected unknown resources
→ report logical item counts and physical resource counts separately
→ detect and report partial logical-item download failure
```

Only after exact execution passes should the planner be integrated with API/report contracts and exposed as a normal `Download new items only` action. The user-facing control should request target new logical-item count; the backend candidate scan limit should remain internal or Advanced. Future batch/overall-limit orchestration should follow that adapter validation rather than precede it.
