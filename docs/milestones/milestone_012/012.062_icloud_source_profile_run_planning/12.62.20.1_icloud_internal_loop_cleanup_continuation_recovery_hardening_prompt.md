# Milestone 12.62.20.1 — iCloud Internal Loop Cleanup-Continuation Recovery Hardening

## Recommended Reasoning Level

Extra High.

This is a narrow safety patch. Do not broaden scope.

## Goal

Fix the internal iCloud loop cleanup-continuation recovery issue discovered during 12.62.20 live validation.

12.62.20 successfully implemented candidate-search terminology hardening, `--candidate-search-cap`, batch size 2 ordinary-still validation, MOV exclusion under `ordinary_still_only`, and larger bounded live validation. However, the internal orchestration run ended failed after batch 2 cleanup continuation with:

```text
DRY_RUN_ALREADY_CONSUMED
```

Manual recovery was safe and successful:

```text
fresh cleanup dry-run 41
→ cleanup execution 42
→ deleted 2 local staged files
→ 0 delete failures
→ 0 verification failures
→ managed iCloud staging file count 0
```

The goal of 12.62.20.1 is to make the orchestrator handle this stale/expired/already-consumed cleanup-preview situation safely and clearly.

This is a narrow patch for cleanup-continuation/recovery behavior in the internal iCloud loop.

1. Harden cleanup continuation when the stored cleanup dry run is stale, expired, or already consumed.

## Required Behavior

### 1. Valid cleanup dry run

If the cleanup dry run is valid and executable:

```text
execute guarded cleanup
verify staging clean
mark batch complete
checkpoint orchestration
continue to next batch or complete total limit
```

### 2. Expired cleanup dry run

If the cleanup dry run is expired:

```text
do not mark unrelated completed batches failed
do not attempt unsafe cleanup
block safely
report stop_reason such as cleanup_preview_expired
print next_safe_action:
  Run a fresh guarded cleanup dry run for this source, then continue cleanup with the new dry-run ID.
```

### 3. Already-consumed cleanup dry run + staging clean

If the cleanup dry run is already consumed and staging is clean:

```text
reconcile idempotently
do not delete again
mark or leave batch complete if evidence supports it
checkpoint the orchestration
continue or complete as appropriate
```

### 4. Already-consumed cleanup dry run + staging not clean

If the cleanup dry run is already consumed and staging is not clean:

```text
do not mark orchestration failed solely because the dry run is stale/consumed
do not delete with stale evidence
block safely
report next safe action:
  Run a fresh guarded cleanup dry run for this source, then continue cleanup with the new dry-run ID.
```

### 5. Fresh dry-run recovery

If the operator provides a fresh cleanup dry-run ID after the previous one expired or was consumed, the orchestrator may continue only if:

```text
same source profile
same orchestration run
same managed staging root
no .partial files
Drop Zone clean
Vault/provenance verification passes through guarded cleanup
explicit confirmation phrase is present
```

The required confirmation phrase remains:

```text
DELETE LOCAL STAGING COPIES
```

### 6. Script output

The script should not merely say `failed`.

When safe recovery is possible, print:

```text
status
stop_reason
failure_reason if any
next_safe_action
cleanup_dry_run_id if applicable
orchestration_run_id
ready-to-copy recovery command if applicable
report_path
```

Answers to lock decisions:

1. For `DRY_RUN_EXPIRED` and `DRY_RUN_ALREADY_CONSUMED` with dirty staging:

Yes — treat this as a safe blocked/recovery-required state, not a terminal failed state.

Preferred behavior:

```text
run status: blocked or cleanup_recovery_required
batch status: blocked or cleanup_recovery_required
stop_reason: cleanup_preview_expired or cleanup_preview_already_consumed
original_cleanup_error_code: DRY_RUN_EXPIRED or DRY_RUN_ALREADY_CONSUMED
failure_reason: null unless there is a true unsafe/inconsistent failure
failed_batches: do not increment
next_safe_action: Run a fresh guarded cleanup dry run for this source, then continue cleanup with the new dry-run ID.
```

Do not mark the run failed solely because a cleanup preview is stale, expired, or already consumed.

2. For `DRY_RUN_ALREADY_CONSUMED` with clean staging:

Use evidence-based reconciliation.

If staging is clean and evidence supports that the batch is already safely handled, reconcile idempotently:

```text
do not delete again
do not fail the run
mark or leave the batch completed
checkpoint the orchestration
continue or complete as appropriate
```

Prior `cleanup_execution_run_id` evidence is ideal, but do not require it as the only possible proof. It is acceptable to reconcile if all of the following are true:

```text
batch Source Intake completed
batch resources have safe Source Intake/provenance evidence
batch resource files are no longer present in managed staging
Drop Zone is clean
.partial workspace is clear
final cleanup/staging verification confirms no eligible staged files remain
same source profile and staging root
```

If staging appears clean but evidence is incomplete or ambiguous, block with `cleanup_state_ambiguous` and give a clear next safe action. Do not pretend success.

For counters, avoid claiming a new deletion happened. Use existing cleanup execution counts when available. If reconciling from clean state without linked execution evidence, report it as reconciled/idempotent cleanup rather than a new delete.

3. For fresh recovery dry runs:

Yes — `continue-cleanup` should be able to accept a fresh `cleanup_dry_run_id` that was not previously linked to the batch, but only in a recovery-eligible state.

Required checks:

```text
same orchestration run
same source profile
same managed staging root
current batch is paused/blocked for cleanup recovery
fresh dry run is valid, unexpired, and executable
Drop Zone clean
no .partial files
guarded cleanup verification still passes
explicit confirmation phrase is present
```

If valid, attach or replace `batch.cleanup_dry_run_id` with the fresh preview ID and continue. Preserve the prior dry-run ID in details/report history if feasible.

4. Stop reason naming:

Use friendlier normalized orchestration stop reasons, while preserving the original cleanup service code in details.

Preferred:

```text
stop_reason: cleanup_preview_expired
original_cleanup_error_code: DRY_RUN_EXPIRED
```

and:

```text
stop_reason: cleanup_preview_already_consumed
original_cleanup_error_code: DRY_RUN_ALREADY_CONSUMED
```

Do not lose the raw code because it is useful for debugging, but do not make the operator interpret raw backend cleanup codes as the main next action.

5. Recovery output:

Yes — blocked recovery-required outcomes should include ready-to-copy recovery guidance, not only prose.

At minimum, output:

```text
next_safe_action
orchestration_run_id
current batch index
source_profile_id
stale_cleanup_dry_run_id
recovery command guidance
continue_cleanup_command template using a placeholder fresh cleanup_dry_run_id
report_path
```

If the app can generate the exact fresh guarded cleanup dry-run command, include it. If not, provide the exact existing way to create a fresh cleanup preview and then the continue command template.

6. CLI exit behavior:

A blocked recovery-required state may return non-zero because the operation did not complete. That is acceptable.

But the JSON/status should clearly distinguish:

```text
blocked / recovery required
```

from:

```text
failed / unsafe failure
```

So a non-zero exit code is acceptable for automation, but the run itself should not be marked failed unless there is a true unsafe or unrecoverable failure.

```

```
