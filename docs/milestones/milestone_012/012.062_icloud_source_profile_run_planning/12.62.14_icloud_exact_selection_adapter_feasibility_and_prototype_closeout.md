## 1. Scope Completed


## 2. Capability Findings

### Current installed distribution

The project-local runtime is configured under:

```text
.tools/icloudpd/
```

Installed version metadata reports `icloudpd 1.32.3`.

The actual installation record contains:

```text
top-level packages: icloud, icloudpd
bundled payloads:  icloud/icloud, icloudpd/icloudpd
console launchers: Scripts/icloud.exe, Scripts/icloudpd.exe
```

The installed `icloudpd/__main__.py` only launches the bundled `icloudpd` executable through `subprocess.call()`.

The installation contains no importable:

```text
pyicloud_ipd
pyicloud_ipd/services/photos.py
icloudpd/base.py
icloudpd/cli.py
```

The backend virtual environment also raises `ModuleNotFoundError` for `icloudpd` and `pyicloud_ipd`.

Therefore, a source-controlled helper executed with the current helper environment cannot import `PhotoAlbum`, `PhotoAsset`, `PyiCloudService`, or the resource download functions audited in upstream source. The current runtime offers only the CLI/bundled executable contract already found insufficient in 12.62.13.

The helper environment's `Scripts/python.exe` also could not be launched from the Codex shell because its Microsoft Store base interpreter reported a terminated logon session. This is a separate validation/deployment concern; even if the launcher worked, the required internal modules are absent from its installed site-packages.

### Matching official source

The matching official repository tag `v1.32.3`, commit `879c561240d993d748ddb4546f935090502b16d3`, exposes technically promising internal APIs:

- `PhotoAsset.id` returns the master record name as a stable-in-response logical item ID.
- `PhotoAsset.versions` maps one logical asset to primary and Live Photo versions.
- primary originals expose size, content type, checksum, and download URL.
- `LivePhotoVersionSize.ORIGINAL` exposes a Live Photo motion resource where present.
- `PhotoAsset.download(session, url, start)` can request one specific version URL.
- `PhotoAlbum` iterates in the provider order used by existing `--recent` behavior and advances through a stateful numeric offset.

These capabilities are internal implementation details, not a supported exact-selection CLI/interface.

### Required capability matrix

| Required capability | Current CLI / installed runtime | Matching upstream internals | Milestone result |
|---|---|---|---|
| List logical items newest-to-oldest | Fixed CLI enumeration only; no structured result | `PhotoAlbum` yields `PhotoAsset` objects | Not available to project adapter |
| Stable remote logical item ID | Not exposed by CLI | `PhotoAsset.id` / master record name | Not available to project adapter |
| Explicit item/resource relationship | CLI prints destination paths only | One `PhotoAsset` owns primary/Live Photo versions | Not available to project adapter |
| Pagination/continuation | No CLI offset/token contract | Stateful numeric offset and bounded pages | Internal only; not a durable cross-run checkpoint |
| Exact selected resource download | No ID/resource allowlist | Per-version URL and asset download call | Internal only; not executable from installed runtime |
| Item-level verified publish | Not provided | Would require new custom staging code | Not implemented |
| Per-item/per-resource failure report | CLI aggregate/log behavior only | Could be built around internal calls | Not implemented or validated |
| Existing auth-boundary reuse | Bundled CLI owns keyring/cookies/session | Raw service construction would require a new helper dependency/auth design | Not proven safe end-to-end |

### Live Photos and sidecars

The matching upstream source can identify a primary original and a Live Photo motion version under one `PhotoAsset`. That is enough to model one logical Live Photo with multiple downloadable resources.

Remote sidecar fields appear in the provider query's desired keys, but they are not part of the normal `PhotoAsset` version lookup used for primary/Live Photo downloads. Generated XMP output is local derivative output rather than a remote cloud resource. Unsupported sidecar relationships must therefore block in a future adapter until their identity and download semantics are explicit.

### Pagination limitations

The internal `PhotoAlbum` implementation uses a mutable numeric offset and a page size. This is potentially adequate inside one bounded run, but it is not an opaque Apple continuation token or durable cross-run checkpoint. Library changes between list and download could shift offsets; a future adapter would need remote-ID deduplication and manifest revalidation.

### Authentication and maintenance risk

Using official-source internals would create a materially different dependency boundary:

- Photo Organizer would need a separately installed, importable raw/internal library environment.
- The helper would need to reuse or recreate keyring/cookie/session behavior without transferring secrets to the backend.
- Internal CloudKit record structures and private API behavior may change without a stable adapter contract.
- Dependency upgrades would require compatibility tests against logical IDs, resource maps, checksums, pagination, and authentication behavior.
- The source-level download helper does not itself provide logical-item atomic publication into Photo Organizer staging.

Installing that dependency, vendoring upstream source, extracting bundled executable contents, or changing the authentication owner would exceed this milestone's approved “smallest prototype” boundary.

## 3. Operational Behavior

### Production behavior

Production behavior is unchanged:

```text
standard
→ download the requested fixed recent window

list_first_non_repeat
→ inspect the requested fixed recent window
→ skip the download only when the complete window is durably known
→ otherwise download the complete window
```

Mixed known/new production windows can still redownload known resources. Production does not continue scanning farther back to fill a target of N unknown logical items.

No exact-selection helper, acquisition mode, API route, report type, or UI option was registered.

### Retained planner enhancement

The pure 12.62.13 planner now supports two input paths:

1. Existing flat path candidates, grouped conservatively by filename.
2. Future adapter-provided logical groups with explicit item and resource identifiers.

The explicit contract uses:

```text
ExplicitLogicalItemCandidate
ExplicitLogicalResourceCandidate
plan_explicit_new_count_selection()
```

When an adapter supplies authoritative groups, the planner does not reconstruct Live Photo relationships from filenames. It preserves the adapter item/resource identifiers only in memory, applies the existing Source Profile known-state supplied for each resource, and selects only unknown resources from selected logical items.

The explicit path blocks when it detects:

- a missing logical item ID
- duplicate logical item IDs
- an empty logical item
- a missing or duplicate resource ID
- duplicate destination paths within an item
- destination path collisions across logical items
- unavailable resource identity
- adapter-declared item/resource ambiguity, including unsupported sidecar relationships
- staged unknown resources pending Source Intake

Remote IDs are not included in the bounded planner summary and are not persisted.

### Unknown staged files

The planner continues to block an adapter-provided candidate when its known-state says the resource is staged but not durably ingested:

```text
staged_unknown_pending_intake
Run Source Intake first for staged unknown resources.
```

Because no executable adapter was integrated, the broader runtime staging inventory and immediate pre-download recheck remain future adapter requirements.

## 9. Known Limitations

### Exact selected download remains unavailable

The current application cannot download a planner-selected arbitrary set of unknown iCloud resources. Known resources can still be redownloaded in mixed fixed windows.

### Source capability is not runtime capability

The matching repository source contains useful internal classes, but the installed runtime does not expose them as imports. Source-code feasibility must not be described as an executable project capability.

### Internal APIs are unsupported adapter dependencies

Even with an importable installation, `PhotoAsset`, `PhotoAlbum`, CloudKit record shapes, version keys, and authentication behavior are internal. A production adapter would require version pinning, compatibility probes, safe upgrade policy, and failure isolation.

### Sidecars and alternate resources remain unresolved

Primary originals and Live Photo motion originals are identifiable in matching source. Remote sidecars, RAW/alternative pairs, adjusted versions, and other provider variants require explicit product and identity rules. Unsupported relationships must block.

### Item-level atomicity is not supplied by the library

Per-version download calls do not provide multi-resource transactional publication. A future helper must download selected unknown resources beneath a `.partial` area, verify expected size/checksum, publish only after the item's selected resources pass, and report/roll back partial publication failures.

### Offset pagination is bounded-run only

The source's numeric offset model is not a durable checkpoint. A future helper must bind listing and execution by remote ID/resource manifest and fail safely if the library changes between phases.

### Authentication boundary is unproven for raw internals

The supported bundled CLI currently owns external/project-local keyring, cookie, and session behavior. A new importable helper must prove it can use that boundary without sending or logging secret material and without making the backend a credential owner.

### Future batch/limit workflow remains blocked

The following cannot safely be built yet:

- normal UI `Download new items only`
- exact batch-size acquisition
- full-run total limit
- repeated acquire/intake/cleanup orchestration
- Advanced candidate scan controls
- durable cross-run checkpoint/resume

## 10. Recommended Next Milestone

Create an Extra High-reasoning dependency and adapter architecture decision milestone before further exact-selection implementation.

Evaluate two explicit paths:

### Preferred: upstream-supported interface

Request or contribute an `icloudpd` interface that provides:

```text
bounded JSON logical-item listing
stable item IDs
explicit resource IDs/roles
continuation/offset metadata
exact item/resource allowlist download
per-item/per-resource result JSON
no credential/session output
```

This keeps `icloudpd` as the supported acquisition owner and minimizes Photo Organizer coupling to private internals.

### Alternative: separately approved raw/internal helper runtime

If upstream support is unavailable, explicitly approve and design a distinct isolated helper environment containing an importable, pinned official-source library rather than the current bundled-only distribution.

That milestone must decide and validate:

- exact package/source and commit pin
- installation, licensing, integrity, and upgrade policy
- Windows and future mini-server deployment
- keyring/cookie/session reuse without secret transfer
- non-interactive expired-session behavior
- structured secret-free stdin/stdout protocol
- logical item and resource schema
- primary, Live Photo, RAW/alternative, adjusted, and sidecar policy
- bounded offset enumeration and remote-ID deduplication
- list-to-download manifest binding
- exact selected-resource download
- size/checksum verification
- `.partial` item-level staging and rollback
- per-item/per-resource failure reporting
- one bounded user-operated real-account validation

Cloud-native IDs should remain transient during that prototype. Persisting them to provenance must remain a separate schema/provenance approval.

Only after the isolated helper passes fixture and user-operated real-account validation should Photo Organizer integrate it with acquisition reports or expose true new-count acquisition. The future batch/total-limit workflow remains downstream of that proof.
