# Milestone 12.62.28.3 - Historical Backfill Acquisition Execution and Source Intake Handoff

## 1. Summary

Implemented the backend execution path for historical iCloud backfill acquisition from inventory.

Real acquisition execution was added, but only when `dry_run=false` is explicitly requested. The new default remains safe: `POST /api/admin/icloud-backfill/acquire` defaults to `dry_run=true` and returns the existing preview behavior without downloading, staging, Source Intake, or inventory completion updates.

When execution is explicitly requested, the service selects eligible unresolved inventory rows, re-resolves them against a fresh helper listing, builds selected item/resource metadata from that fresh listing, executes through the durable exact-selection acquisition path, optionally hands the durable batch to existing batch Source Intake, and marks inventory rows complete only when Source Intake proves every selected resource for the logical item resolved.

## 4. Implementation Details

Added `run_icloud_backfill_acquisition()` in `backend/app/services/icloud_backfill_acquisition_execution_service.py`.

Dry-run behavior:

- delegates to 12.62.28.2 preview,
- performs no download,
- performs no staging,
- performs no Source Intake,
- does not update completion state.

Execution behavior when `dry_run=false`:

- validates source/profile and limits,
- selects eligible unresolved inventory rows,
- skips already completed rows,
- re-resolves against fresh helper listing metadata,
- skips stale/missing helper rows as retryable,
- skips unsafe or partial manifests as retryable,
- builds exact selected item/resource payloads from fresh helper metadata,
- uses durable exact-selection acquisition structures for run/batch/item/resource evidence,
- uses the existing helper selected-download path for actual staging,
- optionally calls `run_batch_source_intake()` after durable staging is ready.

Added a narrow durable entrypoint:

- `run_durable_exact_selection_preparation()`

This accepts an already-prepared exact-selection manifest and reuses existing durable run/batch creation, helper execution, reconciliation, and Source Intake readiness logic.

The selected item cap remains 1000. The helper download request can use the larger historical listing window to locate selected historical item IDs, but it still downloads only the explicitly selected items.

## 6. Source Intake Handoff Behavior

When `dry_run=false` and `auto_run_source_intake=true`:

1. Durable acquisition runs first.
2. Source Intake is called only if the durable batch is ready for Source Intake.
3. Inventory completion is applied only after Source Intake result reconciliation.

For each selected logical item, `backfill_completed=true` is set only when every selected resource has a successful Source Intake resource state:

- `resource_intake_processed`
- `resource_intake_duplicate_linked`
- `resource_intake_skipped_known`

Resolution states:

- `newly_imported`
- `exact_duplicate_resolved`
- `already_known`
- `source_intake_partial_failed`
- `failed_retryable`

Download/staging success alone does not mark inventory completion.

Partial Live Photo completion is not accepted. A Live Photo inventory row completes only when both the still and motion resources resolve through Source Intake.

## 9. Deviations from Prompt

The existing batch Source Intake handoff test exposed a circular import risk around Admin package imports. To preserve the required Source Intake handoff regression, `batch_source_intake_service.py` now lazily imports Admin guardrail/schema helpers inside the handoff function. Behavior is unchanged.

The selected-download helper request now allows the historical listing window for locating explicitly selected item IDs, while keeping selected logical items capped at 1000. This keeps the prompt-approved distinction between:

- metadata listing/re-resolution cap: 100000,
- selected acquisition item cap: 1000.

## 10. Known Limitations

- The implementation maps Source Intake results back to inventory rows by durable batch item order created from the selected inventory order.
- If future execution becomes asynchronous or multi-process across restarts, a stronger persistent inventory-to-batch-item mapping may be useful.
- Stale and unsafe rows are marked retryable, but there is not yet a UI for reviewing them.
- Cleanup is not automated here; existing cleanup dry-run/review controls still apply after Source Intake.
- No Admin UI was added; this remains API-only.
