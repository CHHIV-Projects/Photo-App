# Milestone 12.62.19 — Internal iCloud Multi-Batch Loop Orchestration

## Recommended Reasoning Level

Extra High.

This milestone coordinates multiple safety-sensitive processes: iCloud exact-selection acquisition, durable batch state, Source Intake handoff, Vault/Asset/provenance creation, cleanup dry run, guarded cleanup execution, staging verification, retry/reconciliation, stop conditions, and checkpointing.

## Goal

Implement an internal, non-UI orchestration loop for the durable iCloud exact-selection workflow.

The target workflow is:

```text
preflight
→ plan batch
→ acquire exact-selection batch
→ verify acquired staging files
→ Source Intake that batch
→ verify Source Intake evidence
→ cleanup dry run
→ operator-approved guarded cleanup execution
→ verify staging clean
→ checkpoint
→ repeat until total limit, no more new items, user stop, or failure
```

This is the batch/limit routine we have been building toward.

Important correction:

```text
The loop is not Acquire → Cleanup directly.
The loop is Acquire → Source Intake → Cleanup.
```

Source Intake remains the only path into Drop Zone, Vault, Asset rows, and provenance.

### Phase 3 — Verify Acquired Staging Files

This is where the current manual “verify staging” step belongs.

The orchestrator must perform this automatically as an internal gate.

Verify:


```text

```

Do not proceed to Source Intake if staging verification fails.

### Phase 5 — Verify Source Intake Evidence

After Source Intake, orchestrator verifies:

```text

```

If Source Intake evidence is ambiguous, stop.

Do not proceed to cleanup if Source Intake evidence is incomplete.

### Phase 7 — Operator Approval Before Cleanup Execution

Do not automatically execute cleanup unless the operator has explicitly approved it.

For 12.62.19, acceptable approaches are:

```text

Option B:

--cleanup-dry-run-id <id>
--confirm "DELETE LOCAL STAGING COPIES"

Option C:
orchestrator has --auto-execute-cleanup only if explicitly included with the exact confirmation phrase, but this should be used only for bounded internal validation
```

Preferred default:

```text

```

## Retry / Resume / Failure Policy

For this milestone, be conservative.

Retry/resume may be limited, but it must not blindly restart.

Before retrying any phase:

```text
reconcile filesystem
reconcile acquisition batch/resource rows
reconcile Source Intake evidence
reconcile cleanup evidence
verify staging state
verify Drop Zone state
```

Allowed safe behavior:

```text
already acquired and batch_ready_for_source_intake → resume at Source Intake
Source Intake completed and batch_ready_for_cleanup_dry_run → resume at cleanup dry run
cleanup dry run exists and preview still valid → allow cleanup execution with confirmation
cleanup executed and staging clean → checkpoint batch complete
```

Blocked behavior:

```text
missing files
SHA mismatch
partial files
Drop Zone dirty
ambiguous provenance
wrong source profile
cleanup preview expired
mixed processed/unprocessed evidence that cannot be proven safe
```

Every blocked state must provide a clear next safe action.
