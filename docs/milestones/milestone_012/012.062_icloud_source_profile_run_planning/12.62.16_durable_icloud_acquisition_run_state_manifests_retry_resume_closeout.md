## 3. Findings and Corrections During Validation

### 3.1 Authentication refresh correction

Initial helper authentication could fail with Apple `-20101` / invalid credential behavior when stale keyring credentials were present.

Correction:

- Added `-RefreshCredential` support to `scripts/runtime/authenticate_icloud_exact_helper.ps1`.
- Refresh now removes the helper keyring credential through `pyicloud_ipd.utils.delete_password_in_keyring`.
- Avoided the legacy `icloudpd --delete-from-keyring` path because it could trigger an undesirable old two-step flow.
- Authentication script now uses info-level logging and does not enable debug logging by default.

User validation:

- Refresh path allowed direct 2FA flow.
- Helper auth completed successfully.

### 3.2 Apple checksum metadata decoding correction

Initial list-only validation failed with `helper_invalid_response`.

Cause:

- Apple `fileChecksum` values can be base64 values with a one-byte marker plus a 20-byte value.
- The prototype initially accepted only raw digest lengths.

Correction:

- Added shared checksum metadata decoding in `exact_selection_protocol.py`.
- Accepted Apple marker-prefixed checksum metadata as syntactically valid.
- Continued to reject unknown checksum metadata shapes.

### 3.3 Adjustment metadata-only correction

Initial bounded list-only validation found one unknown logical item but blocked with:

```text
logical_item_identity_ambiguous
```

Further classification narrowed this to:

```text
unsupported_adjustment_metadata_only
```

Interpretation:

- The item had adjustment-related metadata fields.
- The item did not expose an actual adjusted downloadable resource.
- Treating metadata-only as a hard blocker was too conservative for ordinary original selection.

Correction:

- Actual adjusted downloadable resources still block as `unsupported_adjusted_resource`.
- Adjustment metadata-only no longer blocks Phase0 selection.
- The adapter also strips metadata-only as non-blocking if an older/internal helper response emits it.
- Sidecars, RAW/alternative resources, missing identity, missing verification metadata, collisions, and actual adjusted resources remain hard blockers.

Architect review note:

- This correction assumes metadata-only does not alter exact original-resource semantics when no adjusted resource is exposed.
- Architect should confirm this is acceptable for durable acquisition.

### 3.4 Execution failure diagnostics correction

The first one-still execution attempt failed with:

```text
published_manifest_verification_failed
```

That summary hid the real helper item failure.

Correction:

- Added safe item-level failure extraction.
- Phase0 can now report safe bounded stop reasons such as:
  - `selection_manifest_changed`
  - `selected_item_not_found`
  - `download_failed`
  - `size_mismatch`
  - `checksum_mismatch`
  - `destination_exists`

No raw remote IDs, paths, URLs, checksums, tokens, cookies, sessions, or credentials are exposed.

### 3.5 Apple `fileChecksum` content-verification correction

After diagnostics improved, the execution failure narrowed to:

```text
checksum_mismatch
```

Reconnaissance against the installed pinned `icloudpd` / `pyicloud_ipd` source showed:

- `pyicloud_ipd` exposes Apple `fileChecksum` from CloudKit metadata.
- `icloudpd` uses that checksum value to derive a stable temporary `.part` filename.
- The upstream downloader does not appear to validate downloaded file bytes against that value as a content digest.

Conclusion:

- Apple marker-prefixed `fileChecksum` should be treated as opaque provider metadata, not as a local content digest.

Correction:

- Apple marker-prefixed checksum metadata is accepted for manifest consistency and protocol validation.
- Exact byte count is still required.
- Raw MD5/SHA1/SHA256 checksums, if supplied by a provider or fixture, are still content-verified.
- Phase0 post-publish verification reports:

```text
size_only_passed
```

when the selected resource used Apple opaque checksum metadata.

Architect review note:

- Durable manifests should probably store Apple `fileChecksum` as provider metadata, not as a content hash.
- Durable acquisition should compute and store a local content digest after download/publish if content-level verification or dedupe evidence is required.

The child tables track:

### Secret-free manifest strategy

The frozen manifest includes:

- run id
- batch index
- source profile id
- target/scan counts
- planner stop reason
- item ordinal
- per-run salted remote logical-item digest
- grouping
- resource ordinal
- resource role
- expected local relative path
- expected size
- provider checksum kind
- selected-for-download flag

The frozen manifest does not include:

- raw remote iCloud item IDs
- raw remote iCloud resource IDs beyond approved resource roles
- raw download URLs
- Apple credentials
- 2FA codes
- cookies
- session tokens
- keyring values

The persisted remote identity is:

```text
sha256(run_identity_salt + ":" + transient_remote_item_id)
```

The salt is per run. This digest is for run/retry matching only and does not become provenance identity.

### Retry / resume strategy

Retry is not a blind restart.

Before retrying, the durable service:

1. reconciles the filesystem against the frozen manifest
2. blocks if partial publication or conflict is detected
3. revalidates Source Profile/staging state
4. blocks on staged unknown files awaiting Source Intake
5. blocks on `.partial` workspace files
6. checks helper auth state
7. re-lists the bounded iCloud candidate window
8. re-hashes transient remote IDs with the persisted run salt
9. requires exactly one matching candidate per selected logical item
10. verifies expected relative path, role, size, and provider checksum metadata before retrying

If all expected files are already published after a lost helper response, reconciliation marks the batch ready without redownloading.

If some but not all resources for a logical item are published, the batch blocks as:

```text
partial_publication_detected
```

and is not marked ready for Source Intake.

### Source Intake handoff contract

The internal handoff manifest is available through:

```text
get_source_intake_handoff_manifest(...)
```

It returns:

```text
batch_id
run_id
source_profile_id
batch_ready_for_source_intake
ready_resources[]
failed_or_deferred_count
report_path
```

Each ready resource includes:

```text
resource row id
logical item row id
relative path
resource role
byte count
local SHA-256
```

No automatic Source Intake or cleanup is run in this milestone.

### Global iCloud acquisition lease

The internal exact mode is:

```text
internal_exact_selection
```

It is not accepted by the existing public acquisition launch normalization and is not exposed as a UI/API mode.

However, active internal exact runs are included in the global active-run query so standard/list-first acquisition cannot start concurrently and mutate the same iCloud staging area.

Public status/stop paths filter internal exact runs so the new internal mode is not accidentally displayed in the current UI/API status surface.

### Cancellable helper polling and heartbeat

The default exact helper subprocess path now uses a polling loop instead of a single blocking `subprocess.run` call.

The polling path supports:

- heartbeat callback
- stop-request callback
- timeout classification as `helper_timeout`
- stop classification as `user_stopped`
- stderr discard
- stdout capture through a temporary file to avoid pipe deadlock

The durable exact service attaches heartbeat and stop callbacks to helper operations while preserving test-time runner injection.
