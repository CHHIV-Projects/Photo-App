# Milestone 12.62.10.1 — iCloud Acquisition Launch Path / Source Registration Consistency

## Issue Found During 12.62.10 Validation

A newly created iCloud Source Profile reaches an apparently valid readiness state, but acquisition launch fails before `icloudpd` starts.

Test profile:

```text
Chuck iCloud E2E Test 12.62.10
```

Readiness/details show:

```text
Lifecycle: Active
Source type: cloud_export
Staging status: Exists
Managed staging path: approved root OK
Expected acquisition path: path alignment OK
Source registration: Matched
Operation conflicts: None
Blocking reasons: None
Acquire from iCloud: Ready
```

But when clicking **Acquire from iCloud**, the confirmation modal shows:

```text
No matching source registration exists for the requested iCloud acquisition path.
```

This persists after Refresh Readiness, closing/reopening the drawer, and retrying.

## Interpretation

This is not an Apple password / `icloudpd` authentication issue. The app is failing before acquisition reaches `icloudpd`.

The likely problem is that readiness/source-registration lookup and acquisition launch/source-registration lookup are using different identity/path normalization logic.

Possible causes:

```text
- source label sanitization differs between Source Profile creation/readiness and acquisition launch
- acquisition launch recomputes expected path differently from stored managed_staging_path
- source_label with periods/dots in “12.62.10” exposes path canonicalization mismatch
- readiness uses selected Source Profile data, while acquisition launch uses recomputed source_label/path
- Ingestion acquisition modal sends a label/path that does not match the registered source
```

## Goal

Make iCloud acquisition launch use the same source identity/path resolution as iCloud readiness for the selected Source Profile.

For the same selected iCloud Source Profile:

```text
If readiness says Source Registration = Matched and Path alignment = OK,
then Acquire from iCloud must not fail with “No matching source registration exists”
for that same path/source.
```

## Preferred Fix Direction

Prefer the smallest safe fix.

The Ingestion acquisition launch should use the selected Source Profile’s canonical stored identity/path where possible, not a separately recomputed path that can diverge.

If the existing acquisition API only accepts `source_label`, ensure that:

```text
- frontend sends the exact stored source_label from the selected profile
- backend canonicalizes it using the exact same helper as Source Profile creation/readiness
- acquisition launch and readiness agree on expected path
```

If a better source-id-based launch path is feasible with low risk, propose it before implementing. Do not redesign the acquisition system unless necessary.

# Answers to Coder — 12.62.10.1 iCloud Acquisition Launch Path / Source Registration Consistency

## 1. Error origin

Based on the screenshot, the error appeared inside the **Confirm iCloud Acquisition** modal before a successful acquisition launch occurred.

The modal showed:

```text
No matching source registration exists for the requested iCloud acquisition path.
```

with the final **Acquire from iCloud** button still visible.

So treat this as a pre-launch / prepare-launch validation failure surfaced in the modal. However, the important fix is still the same: the acquisition preparation/launch path must use the same source registration identity logic as the selected Source Profile.

Please make sure both:

```text
- confirmation/pre-launch validation
- final launch execution
```

do not use mismatched sanitized-label lookup logic.

## 3. Fix scope: targeted label fix vs source-id-based launch

Use the **smallest safe targeted fix** now:

```text
Pass the original source_label into _lookup_source_registration, not the sanitized label.
```

Preferred fix:

```python
# Broken:
matching_source = _lookup_source_registration(
    db_session,
    source_label=sanitized_label,
    ...
)

# Fix:
matching_source = _lookup_source_registration(
    db_session,
    source_label=source_label,
    ...
)
```

Rationale:

```text
- The DB stores source_label_normalized from the original label.
- _lookup_source_registration uses normalize_source_label.
- Passing sanitized_label introduces underscores and breaks matching.
- The sanitized label should still be used for path/run/report naming where appropriate.
- This is the smallest fix and should unblock 12.62.10 validation.
```

Do **not** implement the source-id-based launch redesign in this milestone unless the targeted fix fails. The source-id approach is likely a good future hardening improvement, but it changes more API/flow surface and is not needed to fix the confirmed bug.

Please document source-id-based acquisition launch as a possible future improvement in the closeout.
