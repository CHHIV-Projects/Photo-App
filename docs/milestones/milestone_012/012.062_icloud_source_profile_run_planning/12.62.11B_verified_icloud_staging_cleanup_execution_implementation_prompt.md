# Milestone 12.62.11B — Verified iCloud Staging Cleanup Execution Implementation

## Goal

Implement a safe, explicitly confirmed cleanup execution path for verified local iCloud staging files.

This milestone converts the existing guided cleanup dry-run workflow into a guarded preview-to-execution flow.

The milestone must also close the existing legacy Admin execution bypass so real cleanup execution cannot be triggered without the new safety contract.

This milestone is one step toward the long-term iCloud ingestion goal:

```text
one coordinated iCloud intake action:
Acquire from iCloud
→ avoid repeats
→ ingest through Source Intake
→ verify imported files
→ clean up local staging
→ report result / next safe action
```

Do not implement full one-click orchestration in this milestone. This milestone should make cleanup execution safe enough to become a future step inside that coordinated workflow.

## Background

Milestone 12.62.11A confirmed that iCloud cleanup dry run exists and that historical live cleanup execution is still present.

Current important findings:

- `POST /api/admin/icloud-staging-cleanup/run` currently accepts `{ source_id, dry_run }`.

- `dry_run=false` can currently trigger live local file deletion.

- The older `AdminView.tsx` surface exposes `Execute Cleanup`.

- The newer guided `IngestionView.tsx` currently exposes dry run only.

- There is no server-enforced completed-dry-run requirement.

- There is no dry-run freshness requirement.

- There is no candidate-manifest binding.

- There is no explicit confirmation field.

- Staged file content is not currently rehashed before deletion.

- Vault evidence is currently existence-only.

- Candidate changes between dry run and execution are not detected.

- Guided UI polling omits `pending`.

This milestone should use the existing cleanup evaluator, path, provenance, reporting, and run-record foundations where appropriate, but it must not preserve the unrestricted `dry_run=false` execution contract.

## Safety-Critical Rules

This milestone introduces destructive local staging-file deletion. These rules are mandatory:

- Cleanup must never delete iCloud cloud-library data.

- Cleanup must never call any iCloud/cloud delete API.

- Cleanup must never delete Vault files.

- Cleanup must never delete Drop Zone files.

- Cleanup must never delete DB records.

- Cleanup must never delete provenance records.

- Cleanup must never delete Source Profile records.

- Cleanup must never delete source registry / ingestion source records.

- Cleanup must only delete verified local files inside the selected iCloud Source Profile’s canonical managed staging path.

- Cleanup execution must require a completed, fresh dry run.

- Cleanup execution must require explicit server-side confirmation.

- Cleanup execution must recompute the candidate manifest and fail closed if it differs from the approved dry run.

- Cleanup execution must revalidate each candidate immediately before deletion.

- Cleanup execution must produce a durable report.

- Cleanup failure must be safe by default.

- Legacy Admin execution must not bypass the new safety contract.

- Frontend confirmation is necessary but not sufficient; backend must enforce all execution rules.

- The app must not store Apple passwords, 2FA codes, session cookies, auth tokens, or secrets.

## Requirements

### 1. Future workflow alignment

Design this milestone so it supports the eventual iCloud goal:

```text
Acquire
→ non-repeat selection
→ Source Intake
→ verified cleanup
→ result summary
```

This milestone should not implement that full chain, but cleanup execution should be designed so it can later become one safe step in that coordinated workflow.

The implementation should preserve separate backend responsibilities:

```text
Acquisition acquires.
Source Intake ingests.
Vault preserves.
DB/provenance explain.
Cleanup only acts on verified local staging.
```

### 2. Performance / scale requirements

Design cleanup execution for both:

- initial iCloud ingestion involving thousands of staged assets

- ongoing incremental ingestion involving tens of assets

Avoid UI or backend assumptions that cleanup candidate lists are small.

Requirements:

- Backend cleanup planning and execution should handle thousands of files without forcing the browser to render thousands of rows.

- JSON reports may contain full detail where appropriate, but UI responses should prefer summary counts, reason counts, and bounded samples/previews.

- Manifest fingerprinting must be deterministic for large candidate sets.

- Hashing staged and Vault files is required for safety, but should run as backend/background work with clear progress/status rather than blocking the browser.

- Execution must remain bounded to the selected Source Profile’s managed staging path.

- Partial failure reporting must remain understandable at large scale.

- Frontend display should summarize large candidate sets rather than rendering thousands of rows.

- Avoid excessive memory use where practical; prefer streaming or iterative hashing/planning patterns when feasible in the existing service structure.

- Keep reports useful for troubleshooting large runs without overwhelming the normal guided UI.

Do not optimize away required SHA-256 verification for speed. For cleanup, archival safety outweighs performance.

### 3. Dedicated cleanup readiness / execution eligibility

Create or extend backend cleanup readiness so cleanup execution is evaluated independently from iCloud authentication readiness.

Local cleanup should not require a healthy Apple/iCloud session.

Cleanup readiness must validate that the selected source is an appropriate iCloud Source Profile and that:

- the source exists

- the profile/source is active or otherwise eligible

- the provider is iCloud where the model supports provider metadata

- the acquisition method is `icloudpd` where applicable

- source root resolves under the approved iCloud exports root

- managed staging path exists where applicable

- source root, managed staging path, and canonical expected iCloud staging path resolve to the same approved directory

- there is no active conflicting acquisition, Source Intake, or cleanup operation

### 4. Split cleanup planning from execution

Refactor the cleanup logic so candidate evaluation is side-effect-free.

The planner should be used by both dry run and execution.

The planner must not delete files.

It should return structured results, including at least:

```text
eligible
protected / skipped
verification_failed
file_missing
```

Use clearer classifications than the current all-skipped model where practical.

### 5. Strengthen positive verification

A candidate may be eligible for deletion only when all required positive evidence exists:

- local staged file exists

- staged path resolves inside selected canonical managed staging root

- staged file is a regular file

- exact provenance exists for selected source ID and normalized relative path

- provenance has asset SHA-256

- linked Asset row exists

- staged file SHA-256 equals provenance / Asset SHA-256

- Asset has Vault path

- Vault path resolves inside configured Vault root

- Vault path is a regular file

- Vault file SHA-256 equals Asset SHA-256

- candidate is not protected by failed/deferred/conflict evidence

Do not rely only on Source Intake report samples as proof of success. Treat provenance + Asset + Vault evidence as stronger durable proof.

### 6. Persist dry-run preview binding

A completed dry run must persist enough information to authorize or reject later execution.

Persist or otherwise store in DB state at least:

- dry-run run ID

- source ID

- canonical source root snapshot

- dry-run generated timestamp

- dry-run expiry timestamp or freshness basis

- eligible count

- eligible bytes

- deterministic manifest fingerprint

- planner/version marker sufficient to invalidate old preview semantics

The manifest fingerprint should be deterministic and based on sorted candidate information, including:

- relative path

- staged content hash

- Asset/provenance SHA-256

- file size or stable stat information

- relevant Vault identity/hash information

Human-readable candidate lists may remain in JSON reports, but execution authorization must not depend solely on reading a mutable report file.

### 7. Add guarded execution contract

Replace or supplement the old `{ source_id, dry_run: false }` pattern with a guarded execution request.

Recommended request shape:

```text
source_id
dry_run_run_id
explicit_confirmation
```

The backend must reject execution unless:

- referenced run exists

- referenced run belongs to the same source

- referenced run is a dry run

- referenced run completed successfully

- referenced run is fresh within 10 minutes

- referenced run has at least one eligible candidate

- referenced run has a manifest fingerprint

- explicit confirmation exactly matches `DELETE LOCAL STAGING COPIES`

- no conflicting operation is active

- recomputed manifest exactly matches the dry-run manifest

A stale, missing, mismatched, wrong-source, failed, live, or already-consumed dry-run authorization must delete nothing.

### 8. Recompute before deletion and fail closed on mismatch

Immediately before execution:

1. Recompute the cleanup candidate plan.

2. Recompute the candidate manifest fingerprint.

3. Compare it to the approved dry-run fingerprint.

4. If anything differs, delete nothing.

5. Return a clear status requiring a new dry run.

Do not delete only the unchanged intersection. The operator preview must match the execution set exactly.

### 9. Revalidate each file before unlink

Immediately before deleting each file, revalidate:

- resolved containment

- regular-file status

- staged content hash

- expected relative path

- selected source identity

- Vault evidence

If a file changes, disappears, or fails verification after planning, do not delete that file. Classify the outcome accurately.

### 10. Atomic operation start / race protection

Improve operation start safety so acquisition, Source Intake, and cleanup cannot pass guardrails concurrently.

Use a DB-backed or advisory-lock-style mechanism if feasible in the existing architecture.

At minimum:

- do not rely only on frontend disabled state

- acquire a shared backend lock or protected start section

- recheck all acquisition/intake/cleanup conflicts inside that protected section

- ensure concurrent cleanup starts cannot both proceed

- ensure cleanup execution cannot start while acquisition or Source Intake is active

- ensure acquisition or Source Intake cannot start while cleanup execution is active

If a full shared cross-operation lock requires broader refactoring than is safe in this milestone, implement the strongest safe targeted protection for cleanup execution and document the remaining broader lock work as a follow-up.

### 11. Durable reporting and partial failure handling

Create the execution report/audit artifact before the first deletion.

Record or preserve per-file outcomes as safely as practical.

Classify at least:

```text
deleted
protected / skipped
verification_failed
delete_failed
file_missing
```

A run with at least one successful deletion and at least one candidate failure should be marked or summarized as:

```text
completed_with_errors
```

A run that deletes nothing because authorization, readiness, or candidate matching failed should clearly report that no deletion occurred.

A final report-writing failure after deletions must not erase the fact that deletions occurred. Preserve DB counts and any available durable evidence.

### 12. Legacy Admin bypass closure

`AdminView.tsx` must not retain an unrestricted `Execute Cleanup` control.

Recommended behavior:

- remove or disable the historical Admin execute button

- retain preview/status visibility if useful

- direct the operator to the guided Ingestion Source Profile flow for cleanup execution

The backend must also reject any old direct `dry_run=false` request that does not satisfy the new server-side execution contract.

### 13. Guided Ingestion execution UI

Add minimal execution controls to `IngestionView.tsx`.

Do not redesign the full Ingestion tab.

Show execution only for iCloud profiles and only when a fresh completed dry run exists for the selected profile.

The UI should display:

- selected Source Profile

- canonical local staging path

- dry-run run ID or timestamp

- dry-run freshness/expiry

- eligible count and bytes

- skipped/protected count

- dry-run report link/path where currently supported

- clear statement that only local staging copies will be deleted

- clear statement that iCloud, Vault, DB, provenance, Source Profile, and source registry records are untouched

Require:

- explicit checkbox

- typed phrase `DELETE LOCAL STAGING COPIES`

- server-side confirmation field

After execution, refresh:

- cleanup status

- workflow summary/status areas currently shown

- readiness/status state

Also fix active polling so `pending` is treated as active wherever cleanup status polling depends on active status.

### 14. Manual only

Keep cleanup manual.

Do not automatically execute cleanup after Source Intake.

Do not automatically reacquire after cleanup.

Do not schedule cleanup.

Do not turn this milestone into one-click ingestion.
