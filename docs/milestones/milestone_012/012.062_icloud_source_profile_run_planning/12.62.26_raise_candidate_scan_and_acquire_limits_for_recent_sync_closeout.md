# Milestone 12.62.26 — Raise Candidate Scan and Acquire Limits for Recent Sync Closeout

## Objective Summary

Finalized the **Recent Sync** limit behavior by raising hardcoded 500 limits to 1000 across the protocol, API schema, and frontend.

This enables Recent Sync to:
- scan up to 1000 recent iCloud candidates (default)
- acquire up to 500 logical assets by default
- allow up to 1000 logical assets as an admin override

No historical backfill implementation in this milestone.

---

## Changes Implemented

### 1. Backend Protocol Constants

**File:** `backend/app/services/icloud_acquisition/exact_selection_protocol.py`

| Constant | Was | Now | Purpose |
|---|---|---|---|
| `MAX_CANDIDATE_SCAN_LIMIT` | 500 | 1000 | Max items in a single listing scan |
| `MAX_SELECTED_ITEM_COUNT` | 500 | 1000 | Max logical items for download |
| `MAX_HELPER_JSON_BYTES` | 5_000_000 | (unchanged) | Payload size limit remains safe |

These constants are used by the helper protocol to validate and bound requests from the exact-selection adapter.

### 2. API Request Schema

**File:** `backend/app/schemas/admin.py`

Class `InternalIcloudRunRequest` updated:

| Field | Was | Now | Notes |
|---|---|---|---|
| `candidate_search_cap` | default=50, le=500 | default=1000, le=1000 | Recent Sync scans newest 1000 candidates by default |
| `total_limit` | default=10, le=500 | default=500, le=1000 | Recent Sync acquires 500 logical assets by default; admin can override to 1000 |
| `batch_size` | (unchanged) | (unchanged) | Internal orchestrator batch granularity, not a limit |

The schema now properly enforces bounds: requests above 1000 for either field are rejected with validation error.

Visible user-facing wording updated to clarify "scan newest candidates" and "acquire up to N logical assets."

## Assumptions & Notes

1. **Listing order stability:** Recent Sync relies on observed newest-first candidate ordering from pyicloud_ipd. This is not a CloudKit guarantee but is observed to be stable in practice. If order changes in future pyicloud_ipd versions, recent sync may miss new assets. Mitigation: metadata in reports allows regressions to be detected.

2. **Payload size:** Worst-case 1000-item payload (mixed stills/videos/Live Photo pairs with realistic metadata) is 0.39 MB, well under 5 MB. No payload size increase needed.

3. **Batch size unchanged:** Internal orchestrator batch_size (default=5) remains a tunable granularity parameter, distinct from the scan/acquire limits now raised to 1000.

4. **Future automation:** This milestone establishes the correct limit bounds for Recent Sync. Future automation (12.62.27+) will add Historical Backfill with separate state tracking; Recent Sync itself remains fully manual/operator-launched.
